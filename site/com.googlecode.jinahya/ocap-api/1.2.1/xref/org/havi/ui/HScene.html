13  /**
14     An {@link org.havi.ui.HScene HScene} is a container representing
15     the displayable area on-screen within which the application can
16     display itself and thus interact with the user. 
17  
18     <p>
19     {@link org.havi.ui.HScene HScene} may be
20     regarded as a simple connection to the window management policy
21     within the device, acting as a &quot;screen resource reservation
22     mechanism&quot; denoting the area within which an application may
23     present components.
24  
25     <h3>Rendering Behavior</h3>
26  
27     By default, {@link org.havi.ui.HScene HScene} does not paint itself
28     on-screen, only its added &quot;child&quot; components and hence
29     its only immediate graphical effect is to &quot;clip&quot; its
30     child components. However, it is possible to request that the
31     entire {@link org.havi.ui.HScene HScene} be painted in the current
32     background color before any drawing takes place, and/or that
33     a background image be drawn before the children are rendered.
34     <p>
35  
36     An application which sets both of
37     setBackgroundMode(NO_BACKGROUND_FILL) and setImageMode(IMAGE_NONE)
38     shall be responsible for ensuring that all pixels in the HScene are
39     filled with a value which is either opaque or transparent only to
40     video. Such an application cannot make any assumptions about the
41     previous contents of the graphics objects into which it is
42     drawing. For implementations which double-buffer the display of
43     graphics, these existing contents are implementation dependent.   
44     
45     <h3>Class Behavior</h3>
46  
47     For all interoperable applications, the {@link org.havi.ui.HScene
48     HScene} is considered the top-level component of the
49     application. No parent component to an {@link org.havi.ui.HScene
50     HScene} should be accessible to applications. Interoperable
51     applications should not use the getParent method in {@link
52     org.havi.ui.HScene HScene}, since results are implementation
53     dependent and valid implementations may generate a run-time error.
54  
55     <p>
56     Although {@link org.havi.ui.HScene HScene} is a subclass of
57     <code>java.awt.Container</code>, implementations are allowed to
58     insert extra classes in the inheritance tree between
59     {@link org.havi.ui.HScene HScene} and <code>Container</code>. It is
60     allowed that this may result in {@link org.havi.ui.HScene HScene}
61     inheriting additional methods beyond those specified here. This
62     allows platforms with only one native <code>java.awt.Frame</code>
63     to use {@link org.havi.ui.HScene HScene} as specified, whereas platforms
64     with support for multiple <code>java.awt.Frame</code> or
65     <code>java.awt.Window</code> classes can use an
66     {@link org.havi.ui.HScene HScene} class derived from the appropriate class.
67  
68     <p> 
69  
70     {@link org.havi.ui.HScene HScenes} follow the design pattern of the
71     <code>java.awt.Window</code> class. They are not a scarce resource
72     on the platform. On platforms which only support one {@link
73     org.havi.ui.HScene HScene} being visible at one time the current
74     {@link org.havi.ui.HScene HScene} both loses the input focus and is
75     hidden (e.g. iconified) when another application successfully
76     requests the input focus. Two
77     <code>java.awt.event.WindowEvent</code> events, with ids
78     <code>WINDOW_DEACTIVATED</code> and <code>WINDOW_ICONIFIED</code>,
79     shall be generated and sent to the {@link org.havi.ui.HScene
80     HScene} which has lost the focus and the {@link
81     org.havi.ui.HScene#isShowing isShowing} method for that HScene
82     shall return false.
83  
84     <p>
85     In terms of delegation, the {@link org.havi.ui.HScene HScene} shall
86     behave like a <code>Window</code> with a native peer
87     implementation, in that it will not appear to delegate any
88     functionality to any parent object. Components which do not specify
89     default characteristics inherit default values transitively from
90     their parent objects. Therefore, the implementation of {@link
91     org.havi.ui.HScene HScene} must have valid defaults defined for all
92     characteristics, e.g. <code>Font</code>, foreground
93     <code>Color</code>, background <code>Color</code>,
94     <code>ColorModel</code>, <code>Cursor</code> and
95     <code>Locale</code>.
96  
97     <h3>Additional Z-order support</h3>
98  
99     {@link org.havi.ui.HScene HScene} extends the
100    <code>java.awt.Container</code> class by providing additional
101    Z-ordering capabilities, which are required since components in the
102    HAVi user-interface are explicitly allowed to overlap each
103    other. The Z-ordering capabilities are defined by the {@link
104    org.havi.ui.HComponentOrdering HComponentOrdering} interface. 
105    
106    <p> 
107    Note that these Z-ordering capabilities (<code>addBefore, addAfter,
108    pop, popInFrontOf, popToFront, push, pushBehind and
109    pushToBack</code>) must be implemented by (implicitly) reordering
110    the child Components within the {@link org.havi.ui.HScene HScene},
111    so that the standard AWT convention that the Z-order is defined as
112    the order in which Components are added to a given Container is
113    maintained. See the description for {@link
114    org.havi.ui.HComponentOrdering HComponentOrdering} for more
115    details.
116 
117    <h3>Shortcut Keys</h3>
118 
119    It is an implementation option for HAVi systems to implement
120    shortcut keys by forwarding <code>java.awt.KeyEvent</code> events
121    to the parent {@link org.havi.ui.HScene HScene} of an
122    application. Under these circumstances it is the responsibility of
123    the application designer to ensure that the relevant
124    <code>java.awt.KeyEvent</code> events used for &quot;shortcut
125    keys&quot; are not consumed by any custom components.
126    
127    <p>
128    Implementations of the standard HAVi UI components which process
129    <code>java.awt.event.KeyEvent</code> events shall not consume any
130    KeyEvent, thus allowing their use as a shortcut key on
131    implementation which rely on KeyEvents being available in this way.
132 
133    <h3>Event Handling</h3>
134 
135    The mechanism by which input events are passed to the {@link
136    org.havi.ui.HScene HScene} and its component hierarchy is not
137    specified.
138 
139    <p> 
140    Note that whether the {@link org.havi.ui.HScene HScene} is visible
141    or not (as determined by the {@link org.havi.ui.HScene#isVisible
142    isVisible} method) does not guarantee that it has the input focus
143    and is receiving events.
144    
145    <p> 
146    When the application initially gains the input focus, this is
147    indicated by the system sending a
148    <code>java.awt.event.WindowEvent</code> of type
149    <code>WINDOW_ACTIVATED</code> to the {@link org.havi.ui.HScene
150    HScene}. The {@link org.havi.ui.HScene HScene} should request that
151    a child component gain the focus, if one is available. However, the
152    mechanism by which this occurs is intentionally not specified here.
153 
154    <p>
155    When the entire application loses the user's focus, the system
156    shall notify the {@link org.havi.ui.HScene HScene} that it is no
157    longer receiving events by sending a java.awt.event.WindowEvent of
158    type WINDOW_DEACTIVATED to the {@link org.havi.ui.HScene HScene}.
159 
160 
161    <h3>Acquiring and Displaying HScenes</h3>
162 
163    <p>
164    There is no public constructor for {@link org.havi.ui.HScene
165    HScene}, it is constructed by an {@link org.havi.ui.HSceneFactory
166    HSceneFactory}. Only one {@link org.havi.ui.HScene HScene} per {@link
167    org.havi.ui.HGraphicsDevice HGraphicsDevice} can be acquired at any
168    one time for each application.
169 
170    <p> 
171    The application may request that it be made visible by calling the
172    the {@link org.havi.ui.HScene#show show} method. This method should
173    ensure that the {@link org.havi.ui.HScene HScene} is completely
174    visible to the user, e.g. by expanding an icon, or changing the
175    stacking order between competing overlapping applications.
176 
177    <p> 
178    Making the {@link org.havi.ui.HScene HScene} visible shall not
179    automatically cause it to receive or even request input
180    focus. Input focus can be requested by the application by calling
181    the {@link org.havi.ui.HScene#requestFocus requestFocus}
182    method at any time. 
183
