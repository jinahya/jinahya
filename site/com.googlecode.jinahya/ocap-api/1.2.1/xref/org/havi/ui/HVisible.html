11  /**
12    The {@link org.havi.ui.HVisible} class is the base class
13    for all non-interactive components. 
14  
15    <p>
16    If a layout manager is associated with the Container into which a
17    {@link org.havi.ui.HVisible} component is placed, the size
18    and location of the component will be controlled by the layout
19    manager.
20  
21    <p>
22    {@link org.havi.ui.HVisible} provides the following
23    features for the use of subclasses:
24  
25    <ul>
26    <li> support for interaction states.
27    <li> a mechanism to associate the component with a pluggable {@link
28    org.havi.ui.HLook} class to which all drawing is delegated.
29    <li> support for state-related content which is drawn by the
30    associated {@link org.havi.ui.HLook}.
31    <li> support for scalable and alignable content.
32    <li> support for private look data.
33    <li> support for preferred sizes when used with a layout manager.
34    <li> control over the background painting behavior.
35    <li> a pluggable text layout management mechanism.
36    </ul>
37  
38    Some of these features are discussed in more detail below.
39  
40    <h3>Interaction State</h3>
41  
42    Interaction states for {@link org.havi.ui.HVisible}
43    components are defined in the {@link org.havi.ui.HState}
44    interface. The only interaction states valid for a plain
45    (un-subclassed) {@link org.havi.ui.HVisible} are:
46    <p><ul>
47    <li>{@link org.havi.ui.HState#NORMAL_STATE} -
48    indicating that the component is in its normal interaction state.
49    <li>{@link org.havi.ui.HState#DISABLED_STATE} -
50    indicating that the component is disabled. While this has no effect
51    on user interaction with a plain {@link org.havi.ui.HVisible
52    HVisible} (since there is no user interaction) it will affect the
53    visual appearance.
54    </ul><p>
55    Subclasses of {@link org.havi.ui.HVisible} may use other
56    states. However, {@link org.havi.ui.HVisible#setInteractionState}
57    is the only means by which state can be changed
58    and will throw an IllegalArgumentException if the state is not valid
59    for a particular component type. Subclasses of {@link
60    org.havi.ui.HVisible} must not manipulate state in other
61    ways.
62    <p>
63    The {@link org.havi.ui.HVisible#getInteractionState
64    getInteractionState} method is provided to allow any class to read
65    the current state of a component.
66  
67    <h3>State-based content</h3>
68  
69    Specific content may be set for any interaction state defined in {@link
70    org.havi.ui.HState}, irrespective of whether the subclass of
71    {@link org.havi.ui.HVisible} can ever be in that state.
72    <p>
73  
74    Note that content is set on the various <code>STATE</code> constants
75    defined in {@link org.havi.ui.HState}, and <em>not</em> on
76    the <code>STATE_BIT</code> constants. A
77    <code>java.lang.IllegalArgumentException</code> will be thrown by
78    any method which takes a state as a parameter if a
79    <code>STATE_BIT</code> rather than a <code>STATE</code> is passed.
80  
81    <p> 
82

120   <p>
121   It is recommended that HLook implementations be capable of drawing
122   components in styles that allow the interaction state to be visually
123   distinguished even when no content is set, or the
124   content is the same for several states. 
125   
126   <p>
127   Content for any component may be changed &quot;on the fly&quot;
128   using the setContent methods. However, some components (e.g. {@link
129   org.havi.ui.HStaticAnimation} and {@link
130   org.havi.ui.HAnimation}) may be asynchronously
131   referencing their content (i.e. through a separate
132   implementation-specific animation thread).  Therefore the following
133   restrictions apply to the {@link
134   org.havi.ui.HVisible#setAnimateContent} method
135   when the instance of {@link org.havi.ui.HVisible} on which
136   it is invoked implements the {@link org.havi.ui.HAnimateEffect}
137   interface:
138 
139   <p><ul>                 
140   <li>The method must be synchronized with any implementation-specific
141   animation thread such that content cannot be changed while a
142   different thread is using it. 
143   <li>If the animation was running the method should stop the
144   animation in a synchronized manner before changing content.  
145   <li>The method should reset the animation to a starting position
146   defined by the current play mode. The repeat count of the animation
147   should be reset to 0.  
148   <li>If the animation was running the method should start the
149   animation.  
150   </ul>
151   
152   Alternative platform-specific animation schemes which do not use
153   {@link org.havi.ui.HAnimateEffect} are outside the
154   scope of this document. However, a similar strategy should be
155   employed to avoid synchronization problems.

157   <h3>Content Scaling and Alignment</h3>
158   
159   Where state-based content is used, it is an implementation option to
160   support scaling of the content to fit the HVisible. Some looks also
161   support alignment of content. See the class description of {@link
162   org.havi.ui.HLook} for details of which looks support scaling
163   and alignment. See the fields description of this class for
164   constants which can be used to specify scaling and alignment modes.

168

363   <h3>Private HLook data</h3>
364 
365   Private data for the use of an associated {@link org.havi.ui.HLook}
366   to optimize the redraw of the component may be set on any
367   {@link org.havi.ui.HVisible}. Examples of such data are:
368   
369   <ul>
370   <li>cached bitmap representations of complex drawing operations
371   <li>scrolling metrics and positions
372   <li>cached layout data
373   </ul>
374 
375   Data is set using the {@link org.havi.ui.HVisible#setLookData}
376   method, and retrieved with the {@link
377   org.havi.ui.HVisible#getLookData} method.
378 
379   <p> 
380   To keep the on-screen representation of an {@link
381   org.havi.ui.HVisible} synchronized with its state, any
382   class may call {@link org.havi.ui.HVisible#setLookData}
383   with a parameter of <code>null</code> to invalidate any cached
384 
385   data. Instances of {@link org.havi.ui.HLook} must be able to
386   regenerate any data they store on {@link org.havi.ui.HVisible}
387   classes at any time. This mechanism is intended for
388   implementation optimization only and shall not be used for storing
389   content.
390 
391   <p> 
392   Use of this mechanism is an implementation option. If this mechanism
393   is not used by an implementation, {@link
394   org.havi.ui.HVisible#getLookData} shall always return
395   <code>null</code> and {@link org.havi.ui.HVisible#setLookData}
396   shall do nothing.  Interoperable systems shall not
397   assume that this mechanism is implemented.

399   <h3>Preferred Sizes</h3>
400 
401   The {@link org.havi.ui.HVisible#setDefaultSize} and
402   {@link org.havi.ui.HVisible#getDefaultSize} methods
403   provide support for a user-defined default preferred size to be
404   passed to a layout manager. This value is returned to the layout
405   manager through the {@link org.havi.ui.HLook#getPreferredSize}
406   method and if set overrides any
407   look-specific value based on content calculated by the {@link
408   org.havi.ui.HLook}.
409 
410   <p>
411   Note that the interpretation of the size passed to {@link
412   org.havi.ui.HVisible#setDefaultSize} is <em>not</em>
413   the overall size of the component, but the area available to the
414   look to render the component into excluding any look-specific
415   borders. Therefore when a layout manager is in use the
416   <em>actual</em> size of the component is likely to be larger than
417   this default size. See the descriptions of {@link
418   org.havi.ui.HLook#getMinimumSize}, {@link
419   org.havi.ui.HLook#getPreferredSize} and {@link
420   org.havi.ui.HLook#getMaximumSize} in {@link
421   org.havi.ui.HLook} for details of the exact algorithm used.
422 
423   <p> 
424   Note that constructors of {@link org.havi.ui.HVisible} and
425   subclasses which accept <code>width</code> and <code>height</code>
426   parameters shall set the default size to <code>[width,
427   height]</code> as if these parameters were passed to {@link
428   org.havi.ui.HVisible#setDefaultSize}. If a layout
429   manager is in use the actual size of the component will probably be
430   larger than this size after layout due to the {@link
431   org.havi.ui.HLook} adding borders. However, if no layout
432   manager is used the <code>width</code> and <code>height</code>
433   parameters are simply used to set the <em>actual</em> size of the
434   component.

436   <h3>Background Painting</h3>
437  
438   Normally the associated {@link org.havi.ui.HLook} does not
439   paint the background of the {@link org.havi.ui.HVisible},
440   allowing for non-rectangular components and text overlaying bitmaps.
441   However, {@link org.havi.ui.HVisible} provides for components
442   which require their background to be painted through the
443   setBackgroundMode method. Note that if the mode is set to {@link
444   org.havi.ui.HVisible#BACKGROUND_FILL} the return
445   value of the {@link org.havi.ui.HComponent#isOpaque} method
446   <em>may</em> be true, depending on whether the current background
447   color of the {@link org.havi.ui.HVisible} is opaque. If
448   the background mode is set to {@link
449   org.havi.ui.HVisible#NO_BACKGROUND_FILL} the
450   {@link org.havi.ui.HComponent#isOpaque} method 
451   <em>must</em> return false.

453   <h3>Event Handling</h3>
454 
455   While implementations of {@link org.havi.ui.HVisible} may
456   enable certain java.awt.AWTEvents, applications should assume that
457   an {@link org.havi.ui.HVisible} class does not generate or
458   respond to any java.awt.AWTEvents. If this behavior is required the
459   standard AWT mechanisms (i.e. processEvent and similar functions)
460   may be used to handle events on {@link org.havi.ui.HVisible}
461   or subclasses. However, it is strongly recommended that
462   component implementors use the HAVi events defined in the
463   <code>org.havi.ui.event</code> package where possible.
464 
465   <p> 
466   For example, a component wishing to respond to user action should
467   normally subclass {@link org.havi.ui.HComponent},
468   implement the {@link org.havi.ui.HActionInputPreferred}
469   interface and handle {@link
470   org.havi.ui.event.HActionEvent} events in the {@link
471   org.havi.ui.HActionInputPreferred#processHActionEvent}
472   method, instead of overriding
473   <code>java.awt.Component#processEvent</code> or similar.
