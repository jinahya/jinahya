7    * without the <code>KEY_RELEASED</code> type event for the same key code or vice versa then 
8    * exclusive reservations shall be made for both event types but only the one 
9    * requested shall be received by the listener. Where a repository includes
10   * a <code>KEY_TYPED</code> event without the corresponding <code>KEY_PRESSED</code> and 
11   * <code>KEY_RELEASED</code> 
12   * events (excluding <code>KEY_PRESSED</code> or <code>KEY_RELEASED</code> events for modifiers),
13   * when an exclusive reservation is requested, it shall also be made for
14   * those corresponding <code>KEY_PRESSED</code> and <code>KEY_RELEASED</code> events but only the
15   * requested event shall be received by the listener.<p>
16   * Repositories do not keep a count of the number of times a particular user 
17   * event is added or removed. Repeatedly adding an event to a repository has no 
18   * effect. Removing an event removes it regardless of the number of times it has 
19   * been added. For example, org.dvb.event.UserEventRepository.addUserEvent(UserEvent 
20   * event) does nothing in case that the event is already in the repository. 
21   * Events are considered to be already in the repository if an event with the same 
22   * triplet of family, type and code is already in the repository.<p>
23   * If an application loses exclusive access to a repository, it shall lose access to all 
24   * events defined in that repository.
25   * Repositories are resolved when they are passed into the methods of EventManager.
26   * Adding or removing events from the repository after those method calls does not 
27   * affect the subscription to those events.
