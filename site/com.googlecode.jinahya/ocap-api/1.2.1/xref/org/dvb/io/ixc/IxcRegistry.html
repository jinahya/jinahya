8   /** 
9    * This is the bootstrap mechanism for obtaining references to
10   * remote objects residing in other Xlets executing on the same
11   * MHP terminal, using a URL-like syntax. The identification of 
12   * a remote
13   * object is given using a syntax indicating the organisation
14   * ID and application ID:
15   *
16   * <br>  /organisation_id/application_id/name
17   * <br>  organisation_id = the organisation ID of the Xlet,
18   *                     as signalled in the application_identifier
19   *                     record, defined in the MHP specification.
20   * <br>  application_id = the application ID of the Xlet, as
21   *                     signalled in the application_identifier
22   *                     record, defined in the MHP specification.
23   * <br>  name           = the name under which the remote object was
24   *                        exported.
25   *
26   * <p>
27   * The organisation ID and the application ID shall each be encoded as a
28   * hexadecimal string, as would be accepted by
29   * java.lang.Integer.parseInt(String s, 16).
30   * <p>
31   * When RMI is used to communicate over a network, stubs generated by a
32   * tool like rmic are often required. This is not necessary for inter-xlet
33   * communication initiated with IxcRegistry. If such stubs are present,
34   * they shall be ignored.
35   * <p>
36   * Similarly, network RMI objects often extend the class 
37   * server.RemoteObject, in order to get appropriate implementations
38   * for Object.hashCode(), Object.equals(), and Object.toString(). Overriding
39   * Object's implementation of these methods in this way is not necessary for 
40   * inter-xlet communication initiated with IxcRegistry, although it is not
41   * harmful. Note that the class server.RemoteObject is not required
42   * in all MHP profiles.
43   **/
