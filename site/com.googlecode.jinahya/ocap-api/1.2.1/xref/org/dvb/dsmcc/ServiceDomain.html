6    /**
7     * A <code>ServiceDomain</code> represents a group of DSMCC objects. The objects 
8     * are sent either using the object carousel for a broadcast network or with 
9     * the DSM-CC User-to-User protocol for an interactive network.<p>
10    * 
11    * To access the objects of a <code>ServiceDomain</code>, it has to be attached to the 
12    * file system name space of the MHP terminal.
13    *
14    * To access the content of an object, the application has four ways:
15    * <ul>
16    * <li>It can instantiate the class that is used to read the object
17    *      (java.io.FileInputStream or java.io.RandomAccessFile for a File 
18    *      or DSMCCStream for a stream) from its pathname. 
19    *      The loading of the object is implicit but the application has 
20    *      no way to abort it.
21    *      NB: Obviously, for the Object Carousel, the write mode of
22    *          java.io.RandomAccessFile is not allowed.
23    * <li>It can instantiate a DSMCCObject and carry out a 
24    *       Synchronous loading. The loading can be aborted by the abort method
25    *       of the DSMCCObject class.
26    *		  When the object is loaded, the application will instantiate the 
27    *       class used to read the object.
28    * <li>It can instantiate a DSMCCObject and carry out an Asynchronous 
29    *       loading. So several loading can be started in parallel from the
30    *		  same thread.
31    * <li>It is also possible to create directly a java.io.File for a DSMCC
32    *     object.
33    * </ul>
34    * Instances of <code>ServiceDomain</code> exist in two states, attached and
35    * detached. Newly created instances are always in the detached state. They
36    * become attached when a call to the <code>attach</code> method succeeds.
37    * They become detached following a call to the <code>detach</code> method.<p>
38    * When service domains in the attached state temporarily lose their network connection (e.g.
39    * if the MHP terminal tunes away from the transport stream
40    * where they are carried), 
41    * the behaviour of DSMCC objects which are part of the
42    * service domain is specified in the main body of the present document. If such
43    * a network connection becomes available again then the service domain shall 
44    * resume normal behaviour.<p>
45    * A service domain which is temporarily lost its network connection may be
46    * forced into the detached state by the implementation if the loss of the
47    * network connection becomes irrecoverable. The precise details of when this 
48    * happens are implementation dependent. This is the only situation when
49    * shall be forced into the detached state. Once a ServiceDomain is detached, 
50    * it will never be automatically attached. 
51    */
