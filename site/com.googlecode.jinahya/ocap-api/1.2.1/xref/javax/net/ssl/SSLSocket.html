55  /** 
56   * This class extends <code>Socket</code>s and provides secure
57   * socket using protocols such as the "Secure
58   * Sockets Layer" (SSL) or IETF "Transport Layer Security" (TLS) protocols.
59   * <P>
60   * Such sockets are normal stream sockets, but they
61   * add a layer of security protections over the underlying network transport
62   * protocol, such as TCP.  Those protections include: <UL>
63   *
64   *	<LI> <em>Integrity Protection</em>.  SSL protects against
65   *	modification of messages by an active wiretapper.
66   *
67   *	<LI> <em>Authentication</em>.  In most modes, SSL provides
68   *	peer authentication.  Servers are usually authenticated,
69   *	and clients may be authenticated as requested by servers.
70   *
71   *	<LI> <em>Confidentiality (Privacy Protection)</em>.  In most
72   *	modes, SSL encrypts data being sent between client and server.
73   *	This protects the confidentiality of data, so that passive
74   *	wiretappers won't see sensitive data such as financial
75   *	information or personal information of many kinds.
76   *
77   *	</UL>
78   *
79   * <P>These kinds of protection are specified by a "cipher suite", which
80   * is a combination of cryptographic algorithms used by a given SSL connection.
81   * During the negotiation process, the two endpoints must agree on
82   * a ciphersuite that is available in both environments.
83   * If there is no such suite in common, no SSL connection can
84   * be established, and no data can be exchanged.
85   *
86   * <P> The cipher suite used is established by a negotiation process
87   * called "handshaking".  The goal of this
88   * process is to create or rejoin a "session", which may protect many
89   * connections over time.  After handshaking has completed, you can access
90   * session attributes by using the <em>getSession</em> method.
91   * The initial handshake on this connection can be initiated in
92   * one of three ways: <UL>
93   *
94   *      <LI> calling <code>startHandshake</code> which explicitly
95   *		begins handshakes, or
96   *      <LI> any attempt to read or write application data on
97   *		this socket causes an implicit handshake, or
98   *      <LI> a call to <code>getSession</code> tries to set up a session
99   *		if there is no currently valid session, and
100  *		an implicit handshake is done.
101  * </UL>
102  *
103  * <P>If handshaking fails for any reason, the <code>SSLSocket</code>
104  * is closed, and no futher communications can be done.
105  *
106  * <P>There are two groups of cipher suites which you will need to know
107  * about when managing cipher suites: <UL>
108  *
109  *	<LI> <em>Supported</em> cipher suites:  all the suites which are
110  *	supported by the SSL implementation.  This list is reported
111  *	using <em>getSupportedCipherSuites</em>.
112  *
113  *	<LI> <em>Enabled</em> cipher suites, which may be fewer
114  *	than the full set of supported suites.  This group is
115  *	set using the <em>setEnabledCipherSuites</em> method, and
116  *	queried using the <em>getEnabledCipherSuites</em> method.
117  *	Initially, a default set of cipher suites will be enabled on
118  *      a new socket that represents the minimum suggested configuration.
119  *
120  *	</UL>
121  *
122  * <P> Implementation defaults require that only cipher
123  * suites which authenticate servers and provide confidentiality
124  * be enabled by default.
125  * Only if both sides explicitly agree to unauthenticated and/or
126  * non-private (unencrypted) communications will such a ciphersuite be
127  * selected.
128  *
129  * <P>When <code>SSLSocket</code>s are first created, no handshaking
130  * is done so that applications may first set their communication
131  * preferences:  what cipher suites to use, whether the socket should be
132  * in client or server mode, etc.
133  * However, security is always provided by the time that application data
134  * is sent over the connection.
135  *
136  * <P> You may register to receive event notification of handshake
137  * completion.  This involves
138  * the use of two additional classes.  <em>HandshakeCompletedEvent</em>
139  * objects are passed to <em>HandshakeCompletedListener</em> instances,
140  * which are registered by users of this API.
141  *
142  * <code>SSLSocket</code>s are created by <code>SSLSocketFactory</code>s,
143  * or by <code>accept</code>ing a connection from a
144  * <code>SSLServerSocket</code>.
145  *
146  * <P>A SSL socket may choose to operate in the client or server mode.
147  * This will determine who begins the handshaking process, as well
148  * as which messages should be sent by each party.  However, each
149  * connection must have one client and one server, or handshaking
150  * will not progress properly.
151  *
152  * @see java.net.Socket
153  * @see SSLServerSocket
154  * @see SSLSocketFactory
155  *
156  * @since 1.4
157  * @version 1.22
158  * @author David Brownell
159  */
