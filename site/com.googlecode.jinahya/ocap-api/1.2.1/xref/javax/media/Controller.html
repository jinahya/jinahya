38  package javax.media;
39  
40  /** 
41   * <code>Controller</code>, which extends <code>Clock</code>,
42   * provides resource-allocation
43   * state information, event generation, and a mechanism for obtaining objects
44   * that provide additional control over a <code>Controller</code>.
45   *
46   * <h2>Controller life-cycle</h2>
47   *
48   *
49   * As a <code>Clock</code>, a <code>Controller</code> is always either
50   * <I>Started</I> or <I>Stopped</I>. 
51   * However, <code>Controller</code> subdivides
52   * <code>Clock's</code>&nbsp;<I>Stopped</I> state into five
53   * resource-allocation phases:
54   * <i>Unrealized</i>, <i>Realizing</i>, <i>Realized</i>,
55   * <i>Prefetching</i>, and <i>Prefetched</i>.
56   * <p>
57   *
58   * The motivation for these life-cycle states is to provide
59   * programmatic control over potentially time-consuming operations.
60   * For example, when a <code>Controller</code> is first constructed, it's in 
61   * the <i>Unrealized</i> state.
62   * While <I>Realizing</I>, the <code>Controller</code> 
63   * performs the communication necessary to locate all of the resources
64   * it needs to  function (such as communicating with a server,
65   * other controllers, or a file system).
66   * The <code>realize</code> method allows an application to initiate this
67   * potentially time-consuming process (<i>Realizing</i>) at an
68   * appropriate time.
69   * When a <code>Controller</code> is <i>Realizing</i> or <i>Prefetching</i>,
70   * it will eventually transition to another state, such as <i>Realized</i>,
71   * <i>Prefetched</i>, or even <i>Unrealized</i>.
72   * <p>
73   * Because a <code>Controller</code> is often in one state on its way to
74   * another, its destination or <i>target</i> state is an integral part
75   * of the <code>Controller</code> life-cycle.
76   * You can query a <code>Controller</code> to determine both its
77   * current state and its target state.
78   * <p>
79   *
80   * A <code>Controller</code> typically moves from the <i>Unrealized</i> state
81   * through <i>Realizing</i> to the <i>Realized</i> state,
82   * then through <i>Prefetching</i>
83   * to the <i>Prefetched</i> state, and finally on to the <i>Started</i> state.
84   * When a <code>Controller</code> finishes because
85   * the end of the  media stream is reached,
86   * its stop time is reached, 
87   * or the <code>stop</code> method is invoked,
88   * the <code>Controller</code> moves from the <i>Started</i> state
89   * back to <i>Prefetched</i>
90   * or possibly back to <i>Realized</i>, ready to 
91   * repeat the cycle.
92   * <p>
93   *
94   * To use a <code>Controller</code>, you set up parameters to
95   * manage its movement through these life-cycle states and then
96   * move it through the states using the <code>Controller</code>
97   * state transition methods. 
98   * To keep track of the <code>Controller's</code> current state,
99   * you monitor the state transition events that it posts when changing states.
100  *
101  * <h3>State Transition Methods</h3>
102  *
103  * A <code>Controller</code> has five methods that are used to induce
104  * life-cycle state changes:
105  * <code>realize</code>, <code>prefetch</code>, <code>deallocate</code>,
106  * <code>syncStart</code>, and <code>stop</code>.
107  *
108  * To transition a <code>Controller</code> to the <i>Realized</i>,
109  * <i>Prefetched</i>, or <i>Started</i> state,
110  * you use the corresponding method: <code>realize</code>, 
111  * <code>prefetch</code>, or <code>syncStart</code>. The
112  * <code>deallocate</code> and <code>stop</code> methods can change
113  * a requested state transition or trigger a state change.
114  * <p>
115  *
116  * The forward transition methods (<code>realize</code>,
117  * <code>prefetch</code>, and <code>syncStart</code>) are executed
118  * asynchronously and return immediately.
119  * When the requested operation is complete, the <code>Controller</code>
120  * posts a <code>ControllerEvent</code>
121  * that indicates that the target state has been reached,
122  * <code>stop</code> or <code>deallocate</code> has been invoked, or
123  * that an error occurred.
124  * <p>
125  *
126  * The <code>deallocate</code>, and <code>stop</code> methods can
127  * change the target state and induce a transition back to
128  * a previous state. For example, calling <code>deallocate</code> on
129  * a <code>Controller</code> in the 
130  * <i>Prefetching</i> state will move it back to
131  * <i>Realized</i>.  These methods are synchronous. 
132  *
133  * 
134  * <h3>State Transition Events</h3>
135  *
136  * A <code>Controller</code> often moves between states
137  * in an asynchronous manner.
138  * To facilitate the tracking of a <code>Controller's</code> state,
139  * every time its state or target state changes,
140  * the <code>Controller</code> is required to post a <code>TransitionEvent</code> that describes
141  * its previous state, current state, and new target state.
142  * By monitoring the <code>Controller</code> event stream, you can determine
143  * exactly what a <code>Controller</code> is doing at any point in time.
144  * <p>
145  *
146  * When one of the asynchronous forward state transition methods 
147  * completes, the <code>Controller</code> posts the appropriate <code>TransitionEvent</code> 
148  * or a <code>ControllerErrorEvent</code> indicating that
149  * the <code>Controller</code> is no longer usable.
150  * For more information about <code>ControllerEvents</code>, see the <a href="#CE"><I>Controller Events section</I></a>. 
151  * <p>
152  *
153  * To facilitate simple asynchronous method protocols,
154  * a <code>Controller</code> always posts a method completion event
155  * when one of the asynchronous forward state transition methods is invoked, even 
156  * if no state or target state change occurs.
157  * For example, if <code>realize</code> is called on a
158  * <i>Prefetching</i>&nbsp;<code>Controller</code>,
159  * a <code>RealizeCompleteEvent</code>
160  * is immediately posted, even though the <code>Controller</code> remains
161  * in the <i>Prefetching</i> state and the target state is still
162  * <i>Prefetched</i>.
163  * The method completion events always report the <code>Controller's</code> previous,
164  * current, and target state at the time the event was posted.
165  * <p>
166  *
167  *
168  * <h2>Controller States</h2>
169  *
170  * This section describes the semantics of each of the <code>Controller</code>
171  * states.
172  * 
173  *
174  * <h3>Unrealized State</h3>
175  * A newly instanced <code>Controller</code> starts in
176  * the <i>Unrealized</i> state.
177  * An <i>Unrealized</i>&nbsp;<code>Controller</code> knows very little about its
178  * internals and does not have enough information to acquire all of the resources
179  * it needs to function. 
180  * In particular, an <i>Unrealized</i>&nbsp;<code>Controller</code> does
181  * not know enough to properly construct
182  * a <code>Clock</code>. 
183  * Therefore, it is illegal to call the following methods
184  * on an <i>Unrealized</i>&nbsp;<code>Controller</code>:
185  * <ul>
186  * <li> <CODE>getTimeBase</CODE>
187  * <li> <CODE>setTimeBase</CODE>
188  * <li> <CODE>setMediaTime</CODE>
189  * <li> <CODE>setRate</CODE>
190  * <li> <CODE>setStopTime</CODE>
191  * <li> <CODE>getStartLatency</CODE>
192  * </ul>
193  *
194  * A <code>NotRealizedError</code> is thrown if any of these methods are called
195  * on an <I>Unrealized</I>&nbsp;<CODE>Controller</CODE>.
196  *
197  * <h3>Realizing and Realized States</h3>
198  *
199  * A <code>Controller</code> is <i>Realized</i> when it has obtained 
200  * all of the information necessary for it to acquire the resources it needs 
201  * to function.
202  * A <i>Realizing</i>&nbsp;<code>Controller</code> is in the process of
203  * identifying the resources that it needs to acquire.
204  * <i>Realizing</i> can be a resource and time-consuming process.
205  * A <i>Realizing</i>&nbsp;<code>Controller</code> might have to communicate with
206  * a server, read a file, or interact with a set of other objects.
207  * <p>
208  *
209  * Although a <i>Realized</i>&nbsp;<code>Controller</code> does not have to acquire
210  * any resources, a <i>Realized</i>&nbsp;<code>Controller</code> is likely to have
211  * acquired all of the resources it needs except those that imply exclusive use of a 
212  * scarce system resource, such as an audio device or MPEG decoding hardware.
213  * <p>
214  *
215  * Normally, a <code>Controller</code> moves from the <i>Unrealized</i> state
216  * through <i>Realizing</i> and on to the <i>Realized</i> state.
217  * After <code>realize</code> has been invoked on a <code>Controller</code>, the only way it can return 
218  * to the <i>Unrealized</i> state is if <code>deallocate</code> is
219  * invoked before <i>Realizing</i> completes.
220  * Once a <code>Controller</code> reaches the <i>Realized</i> state, it
221  * never returns to the <i>Unrealized</i> state; it remains in one of four
222  * states: <i>Realized</i>, <i>Prefetching</i>, <i>Prefetched</i>, or
223  * <i>Started</i>.
224  * <p>
225  *
226  * <h4>Realize method</h4>
227  * The <code>realize</code> method executes asynchronously and completion
228  * is signaled by a <code>RealizeCompleteEvent</code> or a
229  * <code>ControllerErrorEvent</code>.
230  * <p>
231  *
232  * <h3>Prefetching and Prefetched States</h3>
233  *
234  * Once <i>Realized</i>, a <code>Controller</code> might still need to 
235  * perform a number of time-consuming tasks before it is ready to be started. 
236  * For example, it might need to acquire scarce hardware resources, 
237  * fill buffers with media data, or perform other start-up processing. 
238  * While performing these tasks, the <code>Controller</code>
239  * is in the <i>Prefetching</i> state. 
240  * When finished, it moves into the <i>Prefetched</i> state.
241  * Over a <code>Controller's</code> lifetime, <i>Prefetching</i> might have to recur when certain methods are invoked.
242  * For example, calling <code>setMediaTime</code> might cause a <code>Player</code> to be
243  * <i>Prefetched</i> again before it is <i>Started</i>.
244  * <p>
245  *
246  * Once a <code>Controller</code> is <i>Prefetched</i>, it is capable of starting as quickly 
247  * as is possible for that <code>Controller</code>. 
248  * <i>Prefetching</i> reduces the startup latency of a <code>Controller</code> to the minimum
249  * possible value. (The startup latency is the value returned by <code>getStartLatency</code>.)
250  * <p>
251  *
252  * Typically, a <code>Controller</code> moves from the <i>Realized</i> state through
253  * <i>Prefetching</i> and on to the <i>Prefetched</i> state.
254  * Once <i>Prefetched</i>, a <code>Controller</code> 
255  * remains <i>Prefetched</i> unless <code>deallocate</code>, <code>syncStart</code> 
256  * or a method that changes its state and 
257  * increases its startup latency is invoked, such as <code>setMediaTime</code>.
258  * 
259  * <p>
260  * 
261  * A <i>Started</i>&nbsp;<code>Controller</code> returns to the <i>Prefetched</i>
262  * or <i>Realized</i> state when it stops.
263  * <p> 
264  *
265  * <h4>Prefetch Method</h4>
266  *
267  * The <code>prefetch</code> method is asynchronous and its completion is signaled
268  * by a <code>PrefetchCompleteEvent</code> or a <code>ControllerErrorEvent</code>.
269  * As a convenience, if <code>prefetch</code> is invoked before a
270  * <code>Controller</code> has reached the <i>Realized</i> state, 
271  * an implicit <code>realize</code> is invoked by changing the target state
272  * to <i>Prefetched</i>.
273  * Both a <code>RealizeCompleteEvent</code> and a <code>PrefetchCompleteEvent</code> are
274  * posted by the <code>Controller</code> as it
275  * transitions to the <I>Prefetched</I> state.
276  * <p>
277  *
278  * If a <code>Controller</code> is <i>Prefetching</i> and cannot obtain all of the
279  * resources it needs to start, it posts a <code>ResourceUnavailableEvent</code>
280  * instead of a <code>PrefetchCompleteEvent</code>. 
281  * This is a catastrophic error condition
282  * from which the <code>Controller</code> cannot recover.
283  *
284  * <h3>Started State</h3>
285  *
286  * Once <i>Prefetched</i>, a <code>Controller</code> can enter the
287  * <i>Started</i> state. A <I>Started</I>&nbsp;<CODE>Controller's</CODE>&nbsp;<CODE>Clock</CODE> is running and it is processing data.
288  * A <code>Controller</code> returns to the <i>Prefetched</i> or <i>Realized</i>
289  * state when it stops because it has reached its stop time, reached the end of the media,
290  * or because the <code>stop</code> method was invoked.
291  * <p>
292  *
293  * When the <code>Controller</code> moves from the <i>Prefetched</i>
294  * to the <i>Started</i> state, it posts a <code>StartEvent</code>. 
295  * When it moves from the <i>Started</i> state to a stopped state, it posts a <code>StopEvent</code>.
296  * <p>
297  * 
298  * A <code>Controller</code> is a <code>Clock</code>; therefore, <code>syncStart</code>,
299  * <code>setTimeBase</code>, <code>setMediaTime</code>, and <code>setRate</code>
300  * are illegal when the <code>Controller</code> is in the <i>Started</i> state.
301  * <p>
302  *
303  * <h4>syncStart</h4>
304  * The only way to start a <code>Controller</code> is to call
305  * <code>syncStart</code>.
306  * <p>
307  *
308  * It is illegal to call <code>syncStart</code> unless the <code>Controller</code> 
309  * is in the <i>Prefetched</i> state. If <code>syncStart</code> is called before the <code>Controller</code> is <I>Prefetched</I>,
310  *  a <code>NotPrefetchedError</code> is thrown.
311  * <code>Player</code> defines a <code>start</code> method that relaxes this
312  * requirement. 
313  * <p>
314  * 
315  * 
316  * <h3>Freeing the Resources Used by a Controller</h3>
317  *
318  * <code>Deallocate</code> is used to stop a  <code>Controller's</code> resource consumption. For example, 
319  * when <code>Applet.stop</code> is called, <code>deallocate</code> should be called to free the resources that the <code>Controller</code> was using.
320  * <code>Deallocate</code> stops any resource-consuming activity
321  * and releases any exclusive-use resources that the <code>Controller</code>
322  * has acquired.
323  * <code>Deallocate</code> executes synchronously;
324  * when <code>deallocate</code> returns, the resources have been released.
325  * <p>
326  *
327  * If the <code>Controller</code> is <i>Unrealized</i> or <i>Realizing</i>, calling <code>deallocate</code> returns it to the
328  * <i>Unrealized</i> state. 
329  * Otherwise, calling <code>deallocate</code> returns a <code>Controller</code> to  the <i>Realized</i> state.
330  * Regardless of the state that a <code>Controller</code> is in,
331  * <code>deallocate</code> must relinquish any exclusive-use
332  * system resources that it holds;
333  * the only way to guarantee that a <code>Controller</code> is not holding
334  * resources is to call the <code>deallocate</code> method.
335  * <p>
336  * 
337  * It is illegal to call <code>deallocate</code> on a  <i>Started</i>&nbsp;<code>Controller</code>. 
338  * You must stop the <code>Controller</code> before
339  * it can relinquish its resources.
340  * <p>
341  *
342  * When <code>deallocate</code> is called,  a <code>Controller</code> posts a special <code>StopEvent</code>,
343  * <code>DeallocateEvent</code>.
344  * 
345  * <a name="CE"><h2>Controller Events</h2></a>
346  *
347  * <code>Controller</code> events asynchronously deliver
348  * information about <code>Controller</code> state changes.
349  * There are four kinds of notifications:
350  * life-cycle transition, method acknowledgement, 
351  * state notification, and error notification.
352  * <p>
353  *
354  * To receive events, an object must implement the <code>ControllerListener</code>
355  * interface and use the <code>addControllerListener</code> method to register its interest in a <code>Controller's</code> events. 
356  * All <code>Controller</code> events are posted to each registered listener.
357  * <p>
358  *
359  * The <code>Controller</code> event mechanism is extensible and 
360  * some <code>Controllers</code> define events other than 
361  * the ones described here.
362  * For example, the <code>DurationUpdateEvents</code> that a <code>Player</code> posts
363  * are <code>ControllerEvents</code>.
364  *
365  * <dl>
366  * <dt> <code>TransitionEvent</code>
367  * <dd>
368  * <code>TransitionEvents</code> are posted when a <code>Controller's</code>
369  * current or target state changes. <code>TransitionEvent</code>
370  * is subclassed to provide a small set of events
371  * that are posted for particular kinds of transitions that
372  * merit special interest. The class name of the event indicates
373  * either the reason that the event was posted
374  * (such as <code>EndOfMediaEvent</code>),
375  * or the particular transition that the event represents (such as
376  * <code>PrefetchCompleteEvent</code>).
377  * <p>
378  *
379  * In addition to being posted for state transitions,
380  * the method acknowledgement events <code>RealizeCompleteEvent</code>,
381  * <code>PrefetchCompleteEvent</code>, <code>StartEvent</code>,
382  * <code>DeallocateEvent</code>,
383  * and <code>StopByRequestEvent</code> are always posted to signify method
384  * completion even if no transition has taken place.
385  * <p>
386  * 
387  * </dd>
388  *
389  * <dl>
390  * <dt> <code>RealizeCompleteEvent</code>
391  * <dd>
392  * Posted when a <code>Controller</code> moves from
393  * <i>Realizing</i> to the <i>Realized</i> state,
394  * or when the <code>realize</code> method is invoked and the
395  * <code>Controller</code> is already <i>Realized</i>.
396  * </dd>
397  *
398  * <dt> <code>PrefetchCompleteEvent</code>
399  * <dd>
400  * Posted when a <code>Controller</code> moves from <i>Prefetching</i> to
401  * the <i>Prefetched</i> state, or when the <code>prefetch</code> method is
402  * invoked and the <code>Controller</code> is already <i>Prefetched</i>.
403  * </dd>
404  *
405  * <dt> <code>StartEvent</code>
406  * <dd>
407  * Posted when a <code>Controller</code> moves from <i>Prefetched</i> to
408  * <i>Started</i>.
409  * <p>
410  * </dd>
411  *
412  * <dt> <code>StopEvent</code>
413  * <dd>
414  * Posted when a <code>Controller</code> moves backward. For example, when moving from  <i>Prefetched</i>
415  * to <i>Realized</i> or from <i>Started</i> to <i>Prefetched</i>.
416  * The <i>reason</i> that a stop event occurs is often important; this information is provided through several subclasses
417  * of <code>StopEvent</code>.
418  * </dd>
419  *
420  * <dl>
421  * <dt> <code>StopAtTimeEvent</code>
422  * <dd>
423  * Posted when a <code>Controller</code> changes state because it has reached
424  * its stop time.
425  * </dd>
426  *
427  * <dt> <code>StopByRequestEvent</code>
428  * <dd>
429  * Posted when a <code>Controller</code> changes state because <code>stop</code> is invoked.
430  * This event is also posted as an acknowledgement to <code>stop</code> requests.
431  * </dd>
432  *
433  * <dt> <code>DeallocateEvent</code>
434  * <dd>
435  * Posted when the <code>deallocate</code> method is invoked, indicating
436  * a possible state change and the loss of exclusive-use resources.
437  * The current state is either <i>Unrealized</i> or <i>Realized</i>.
438  * This event doesn't always indicate a state change. For example, it is posted
439  * even if <code>deallocate</code> is called on a <i>Realized</i>&nbsp;
440  * <code>Controller</code>.
441  * </dd>
442  *
443  * <dt> <code>EndOfMediaEvent</code>
444  * <dd>
445  * Posted when a <code>Controller</code> has reached the end of the media.
446  * </dd>
447  *
448  * <dt> <CODE>ControllerClosedEvent</CODE>
449  * <dd> When a <CODE>Controller</CODE> closes it is no longer usable,
450  * and it will post a <CODE>ControllerClosedEvent</CODE>.
451  * Once this has happened method calls on the <CODE>Controller</CODE>
452  * have undefined behavior.
453  * A <CODE>Controller</CODE> will close for one of two reasons. Either
454  * the <CODE>close</CODE> method was invoked on the <CODE>Controller</CODE>, or
455  * an error has occurred. If a <CODE>Controller</CODE> is closed because the
456  * <CODE>close</CODE> method was invoked, it posts a <CODE>ControllerClosedEvent</CODE>.
457  * If an error occurs it posts one of the <CODE>ControllerErrorEvents</CODE>.
458  * </dd>
459  * </dl>
460  * </dl>
461  *
462  * <dt> <code>ControllerErrorEvent</code>
463  * <dd>
464  * This is the super class of all of the error events that can be posted
465  * by a <code>Controller</code>. While this event is rarely posted, you should watch
466  * for it when processing
467  * other error events--this is how you can detect implementation-specific
468  * error events.
469  * <p>
470  *
471  * When a <code>ControllerErrorEvent</code> is posted, it indicates a catastrophic
472  * error from which the <code>Controller</code> cannot recover.  There is no
473  * recovery mechanism for a <code>Controller</code> once one of these events has
474  * been posted.
475  * <p>
476  *
477  * <dl>
478  * <dt> <code>ResourceUnavailableEvent</code>
479  * <dd>
480  * This error event is posted during <i>Prefetching</i> or <i>Realizing</i>
481  * to indicate that the operation has failed because a required resource was
482  * unavailable.
483  * </dd>
484  * 
485  * <dt> <code>DataLostErrorEvent</code>
486  * <dd>
487  * This error event is posted when a <code>Controller</code> has lost data.
488  * </dd>
489  * <dt> <code>InternalErrorEvent</code>
490  * <dd>
491  * This error event is posted when something goes
492  * wrong with the <code>Controller</code> for an implementation-specific reason. 
493  * This usually indicates that there is a problem with the implementation.
494  * </dd>
495  * </dl>
496  *
497  * <dt> Status Change Events
498  * <dd>
499  * A small number of status changes occur
500  * in a <code>Controller</code> where notification of the change is useful,
501  * particularly for updating user interface components.
502  * Notification of these changes is provided through three
503  * <code>ControllerEvents</code>:
504  * </dd>
505  *
506  * <dl>
507  * <dt> <code>RateChangeEvent</code>
508  * <dd>
509  * Posted when the rate of a <code>Controller</code> changes.
510  * 
511  * <dt> <code>StopTimeChangeEvent</code>
512  * <dd>
513  * Posted when the stop time of a <code>Controller</code> changes.
514  *
