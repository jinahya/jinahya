38  package javax.media;
39  
40  /** 
41   * The <code>Clock</code> interface is implemented by objects that support
42   * the Java Media time model. 
43   * For example, this interface might be implemented by an object that 
44   * decodes and renders MPEG movies.
45   *
46   * <p>
47   *
48   * <h2>Clock and TimeBase</h2>
49   *
50   * <p>
51   * A <code>Clock</code> contains a <code>TimeBase</code> that provides a source of time, 
52   * much like a crystal oscillator. The only information that a <code>TimeBase</code> provides is 
53   * its current time; it does not provide any methods for influencing how time is kept.  
54   * 
55   * A <code>Clock</code> defines a transformation on the time that its <code>TimeBase</code> keeps, typically marking
56   * time for a particular media stream. The time that a <code>Clock</code> keeps is referred to as the <i>media time</i>.
57   *
58   * <p>
59   * <h3>Clock Transform</h3>
60   * 
61   * The transformation that a <code>Clock</code> defines on a <code>TimeBase</code>
62   * is defined by three parameters: rate, <i>media start-time</i> (mst), and 
63   * <i>time-base start-time</i> (tbst). 
64   * Given a <i>time-base time</i> (tbt),
65   * the <i>media time</i> (mt) can be calculated
66   * using the following transformation:
67   * <p>
68   * 
69   * <CENTER><CODE>mt =  mst + (tbt - tbst)*rate </CODE></CENTER>
70   * <p>
71   * 
72   * The rate is simply a scale factor that is applied 
73   * to the <code>TimeBase</code>.
74   * For example, a rate of 2.0 indicates that the <code>Clock</code>
75   * will run at twice the rate
76   * of its <code>TimeBase</code>.  Similarly, a negative rate indicates that
77   * the <code>Clock</code> runs in the opposite direction of its <code>TimeBase</code>.
78   * <p>
79   * 
80   * The <i>time-base start-time</i> and the
81   * <i>media  start-time</i> define a common point in time
82   * at which the <code>Clock</code> and the <code>TimeBase</code> are synchronized.
83   * <p>
84   * 
85   * <h3>Default Time Base</h3>
86   * 
87   * A <code>Clock</code> has a default <code>TimeBase</code>.
88   * For many objects that support the <code>Clock</code> interface, the default
89   * <code>TimeBase</code> is the system <code>TimeBase</code>. 
90   * The system <code>TimeBase</code> can be obtained from <code>Manager</code> through
91   * the  <code>getSystemTimeBase</code> method. 
92   * <p>
93   * 
94   * Some <code>Clocks</code> have a <code>TimeBase</code> other than 
95   * the system <code>TimeBase</code>. For example, an audio renderer that implements the <code>Clock</code>
96   * interface might have a <code>TimeBase</code> that represents a
97   * hardware clock.
98   * <p>
99   * 
100  * 
101  * <h2>Using a Clock</h2>
102  * 
103  * You can get the <code>TimeBase</code> associated with a <code>Clock</code> by calling the <code>getTimeBase</code> method. To change the 
104  * <CODE>TimeBase</CODE> that a <CODE>Clock</CODE> uses, you call the <code>setTimeBase</code> method.
105  * These get and set methods can be used together to synchronize different <code>Clocks</code> to the
106  * same <code>TimeBase</code>.
107  * <p>
108  * For example, an application might want to force a video renderer to sync to the <code>TimeBase</code> of an audio renderer. To do this,
109  * the application would call <code>getTimeBase</code> on the audio renderer and then use the value returned to call <code>setTimeBase</code> on the video renderer.
110  * This would ensure that the two rendering objects use the same source of time. 
111  *
112  * You can reset a <CODE>Clock</CODE> to use its default <CODE>TimeBase</CODE> by
113  * calling <CODE>setTimeBase(null)</CODE>.
114  * <p>
115  *
116  * Some <code>Clocks</code> are incapable of using another <code>TimeBase</code>.
117  * If this is the case, an <code>IncompatibleTimeBaseException</code> is thrown when <code>setTimeBase</code> 
118  * is called.
119  * <p>
120  * 
121  * <code>Clock</code> also provides methods for getting and setting a <code>Clock's</code>&nbsp;<i>media time</i> and rate: 
122  * <ul>
123  * <li><code>getMediaTime</code> and <code>setMediaTime</code>
124  * <li><code>getRate</code> and <code>setRate</code>
125  * </ul>
126  * <a name="start">
127  * <h3>Starting a Clock</h3></a>
128  *
129  * Until a <code>Clock's</code>&nbsp;<code>TimeBase</code> transformation takes effect, the <code>Clock</code> is 
130  * in the <i>Stopped</i> state. Once all three transformation parameters (<i>media start-time</i>, 
131  * <i>time-base start-time</i>, and rate)  have been provided 
132  * to the <code>Clock</code>, it enters the <i>Started</i> state.
133  * <p>
134  *
135  * To start a <code>Clock</code>, <code>syncStart</code> is called with 
136  * the <i>time-base start-time</i> as an argument.
137  * The new <i>media start-time</i> is taken as the current
138  * <i>media time</i>, and the current rate defines the <code>Clock's</code> rate parameter.
139  * When <code>syncStart</code> is called, the <code>Clock</code> and its <code>TimeBase</code> are
140  * locked in sync and the <code>Clock</code> is considered to be in the <i>Started</i> state.
141  * <p>
142  *
143  * When a <code>Clock</code> is stopped and then restarted (using <code>syncStart</code>), 
144  * the <i>media start-time</i>
145  * for the restarted <code>Clock</code> is the current <i>media time</i>.
146  *
147  * The <code>syncStart</code> method is often used to synchronize
148  * two <code>Clocks</code> that share the same <code>TimeBase</code>. 
149  * When the <i>time-base start-time</i> and rate of each clock are set to the same values and 
150  * each <code>Clock</code> is set with the appropriate <i>media start-time</i>,
151  * the two <code>Clocks</code> will run in sync.
152  * <p>
153  *
154  * When <code>syncStart</code> is called with a
155  * new <i>time-base start-time</i>,
156  * the synchronization with the <i>media time</i> doesn't occur
157  * until the <code>TimeBase</code>
158  * reaches the <i>time-base start-time</i>.
159  * The <code>getMediaTime</code> method returns
160  * the untransformed <i>media time</i> until the <code>TimeBase</code>
161  * reaches the <i>time-base start-time</i>. 
162  * <p>
163  * The <CODE>getSyncTime</CODE> method behaves slightly differently.
164  * Once <CODE>syncStart</CODE> is invoked, <CODE>getSyncTime</CODE>
165  * always reports the transformed <i>time-base time</i>, 
166  * whether or not the <i>time-base start-time</i> has been reached.
167  * You can use <code>getSyncTime</code> to determine how much time remains
168  * before the <i>time-base start-time</i> is reached. 
169  * When the <i>time-base start-time</i> is reached,
170  * both
171  * <CODE>getMediaTime</CODE> and <CODE>getSyncTime</CODE>
172  * return the same value.
173  * <p>
174  *
175  * Objects that implement
176  * the <code>Clock</code> interface can provide more convenient start
177  * methods than <CODE>syncStart</CODE>.
178  * For example, <code>Player</code> defines <code>start</code>, 
179  * which should be used instead of <code>syncStart</code> 
180  * to start a <code>Player</code>.
181  * 
182  * <h3>Stopping a Clock</h3>
183  * 
184  * A <i>Stopped</i>&nbsp;<code>Clock</code> is no longer synchronized to
185  * its <code>TimeBase</code>. When a <code>Clock</code> is <i>Stopped</i>,
186  * its <i>media time</i>  no longer moves in rate-adjusted synchronization with
187  * the <i>time-base time</i> provided by its <code>TimeBase</code>.
188  * <p>
189  * 
190  * There are two ways to explicitly stop a <code>Clock</code>: you can invoke
191  * <code>stop</code> or set a <i>media stop-time</i>.
192  * When <code>stop</code> is invoked, synchronization with the
193  * <code>TimeBase</code> immediately stops. 
194  * When a <i>media stop-time</i> is set,
195  * synchronization stops when the <i>media stop-time</i> passes.
196  * <p>
197  *
198  * A <CODE>Clock's</CODE> rate affects how its <i>media stop-time</i>
199  * is interpreted.
200  * If its rate is positive, the <code>Clock</code>
201  * stops when the <i>media time</i> becomes
202  * greater than or equal to the stop time. 
203  * If its rate is negative, the <code>Clock</code> stops
204  * when the <i>media time</i> becomes
205  * less than or equal to the stop time.
206  * <p>
207  *
208  * If the stop-time is set to a value that the <code>Clock</code>
209  * has already passed, the <code>Clock</code> immediately stops.
210  * <p>
211  * 
212  * Once a stop-time is set, it remains in effect until it is changed
213  * or cleared.
214  * To clear a stop-time, call <code>setStopTime</code>
215  * with <code>Clock.RESET</code>.
216  * A <CODE>Clock's</CODE> stop-time is cleared automatically when it stops.
217  * <p>
218  *
219  * If no stop-time is ever set or if the stop-time is cleared,
220  * the only way to stop the <code>Clock</code> is
221  * to call the <code>stop</code> method.
222  * <p>
223  *
224  * <h2>Clock State</h2>
225  * 
226  * Conceptually, a <code>Clock</code> is
227  * always in one of two states: <i>Started</i> or <i>Stopped</i>.
228  * A <code>Clock</code> enters the <i>Started</i> state after 
229  * <code>syncStart</code> has been called and the <CODE>Clock</CODE>
230  * is mapped to its <code>TimeBase</code>.
231  * A <code>Clock</code> returns to the <i>Stopped</i> state immediately
232  * when the <code>stop</code> method is called or the
233  * <i>media time</i> passes the stop time.
234  * <p>
235  *
236  * Certain methods can only be invoked when the <code>Clock</code> is in a
237  * particular state.
238  * If the <code>Clock</code> is in the wrong state when one of these methods
239  * is called, an error or exception is thrown.
240  * 
241  * <h3>Methods Restricted to <i>Started</i> Clocks</h3>
242  *
243  * The <code>mapToTimeBase</code> method can only be called on a 
244  * <code>Clock</code> in the <i>Started</i> state. 
245  * If it is invoked on a <i>Stopped</i>&nbsp;<code>Clock</code>,
246  * a <code>ClockStoppedException</code> is thrown.
247  * This is because the <code>Clock</code> is not synchronized to
248  * a <code>TimeBase</code> when it is <i>Stopped</i>.
249  *
250  * <h3>Methods Restricted to <i>Stopped</i> Clocks</h3>
251  *
252  * The following methods can only be called on a  
253  * <code>Clock</code> in the <i>Stopped</i> state.
254  * If invoked on a <i>Started</i>&nbsp;
255  * <code>Clock</code>, these methods throw a <code>ClockStartedError</code>.
256  * <ul>
257  * <li> <code>syncStart</code>
258  * <li> <code>setTimeBase</code>
259  * <li> <code>setMediaTime</code>
260  * <li> <code>setRate</code>
261  * </ul>
262  * 
263  * Resetting the rate, the <i>media time</i>, the time base, or the 
264  * <i>time-base start-time</i> implies a complete remapping 
265  * between the <code>Clock</code> and its
266  * <code>TimeBase</code> and is not allowed on
267  * a <i>Started</i>&nbsp;<code>Clock</code>. 
268  *
269  * <h3>Methods with Additional Restrictions</h3>
270  * 
271  * A race condition occurs if a new <i>media stop-time</i> is set
272  * when a <code>Clock</code> is already approaching a previously
273  * set <i>media stop-time</i>.
274  * In this situation, it impossible to guarantee when the <code>Clock</code>
275  * will stop. To prevent this race condition, <code>setStopTime</code> can
276  * only be set once on  a <i>Started</i>&nbsp;<code>Clock</code>.
277  * A <code>StopTimeSetError</code> is thrown if <code>setStopTime</code>
278  * is called and the <i>media stop-time</i> has already been set.
279  * <p>
280  *
281  * There are no restrictions on calling <code>setStopTime</code> on a 
282  * <i>Stopped</i>&nbsp;<code>Clock</code>; the stop time can always be
283  * reset if  the <code>Clock</code> is <i>Stopped</i>.
284  * 
285  * @see TimeBase
286  * @see Player
287  * @version 1.43, 07/10/09
288  */
