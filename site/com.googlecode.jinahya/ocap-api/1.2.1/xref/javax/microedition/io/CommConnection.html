41  /** 
42   * This interface defines a logical serial port connection.
43   * A "logical" serial port is defined as a logical connection through
44   * which bytes are transferring serially.  The logical serial port
45   * is defined within the underlying operating system and may not
46   * necessarily correspond to a physical RS-232 serial port.  For
47   * instance, IrDA IRCOMM ports can commonly be configured as a logical serial
48   * port within the operating system so that it can act as a "logical"
49   * serial port.
50   * <P>
51   * A comm port is accessed using a Generic Connection Framework string
52   * with an explicit port identifier and embedded configuration
53   * parameters, each separated with a semi-colon (;).
54   * </P>
55   * <P>
56   * Only one application may be connected to a particular serial port at a
57   * given time. 
58   * An <code>java.io.IOException</code> is thrown, if an attempt is
59   * made to open the serial port with <code>Connector.open()</code>
60   * and the connection is already open.
61   * </P>
62   * <P>
63   * A URI with the type and parameters is used to open the connection.
64   * The scheme (defined in RFC 2396) must be:<BR>
65   * <code>comm:&lt;port identifier&gt;[&lt;optional parameters&gt;]</code>
66   * </P>
67   * <P>
68   * The first parameter must be a port identifier, which is a
69   * logical device name.
70   * These identifiers are most likely device specific and should
71   * be used with care.
72   * </P>
73   * <P>
74   * The valid identifiers for a particular device and OS can be queried through
75   * the  method <code>System.getProperty()</code> using the key
76   * <em>"microedition.commports"</em>. A comma separated list of ports
77   * is returned which can be combined with a <code>comm:</code> prefix
78   * as the URL string to be used to open a serial port connection.
79   * (See port naming convention below.)
80   * </P>
81   * <P>
82   * Any additional parameters must be separated by a semi-colon (;) and
83   * spaces are not allowed in the string.  If a particular optional parameter
84   * is not applicable to a particular port, the parameter MAY be ignored.
85   * The port identifier MUST NOT contain a semi-colon (;).
86   * </P>
87   * <P>
88   * Legal parameters are defined by the definition of the parameters below.
89   * Illegal or unrecognized parameters cause an <code>IllegalArgumentException</code>.
90   * If the value of a parameter is supported by the device, 
91   * it must be honored.  If the value of a parameter is not supported
92   * a <code>java.io.IOException</code> is thrown.
93   * If a <code>baudrate</code> parameter is requested, it is treated
94   * in the same way that the <code>setBaudRate</code> method
95   * handles baudrates. e.g., if the baudrate requested is not
96   * supported the system MAY substitute a valid baudrate, which
97   * can be discovered using the <code>getBaudRate</code> method.
98   * </P>

106  * <TH>Description</TH>
107  * </TR>
108  * <TR>
109  * <TD><code>baudrate</code></TD>
110  * <TD><code>platform dependent</code></TD>
111  * <TD>The speed of the port.</TD>
112  * </TR>

116  * <TD>The number bits per character(<code>7</code> or <code>8</code>).</TD>
117  * </TR>
118  * <TR>

225  * <H2>
226  * Security
227  * </H2>
228  * <P>
229  * Access to serial ports is restricted to prevent unauthorized
230  * transmission or reception of data. The security model applied to
231  * the serial port connection is defined in the implementing profile.
232  * The security model may be applied on the invocation of the
233  * <code>Connector.open()</code>
234  * method with a valid serial port connection string.
235  * Should the application
236  * not be granted access to the serial port through the profile
237  * authorization scheme, a
238  * <code>java.lang.SecurityException</code> will be thrown from the
239  * <code>Connector.open()</code> method.
240  * The security model MAY also be applied during execution, specifically
241  * when the methods
242  * <code>openInputStream()</code>, <code>openDataInputStream()</code>,
243  * <code>openOutputStream()</code>, and
244  * <code>openDataOutputStream()</code> are invoked.
245  * </P>
246  * <H2>
247  * Examples
248  * </H2>
249  * <P>
250  * The following example shows how a <code>CommConnection</code>
251  * would be used to access a simple loopback program.
252  * </P>
253  * <PRE>
254  * CommConnection cc = (CommConnection)
255  *            Connector.open("comm:com0;baudrate=19200");
256  * int baudrate = cc.getBaudRate();
257  * InputStream is  = cc.openInputStream();
258  * OutputStream os = cc.openOutputStream();
259  * int ch = 0;
260  * while(ch != 'Z') {
261  *     os.write(ch);
262  *     ch = is.read();
263  *     ch++;
264  * }
265  * is.close();
266  * os.close();
267  * cc.close();
268  * </PRE>
269  * <P>
270  * The following example shows how a <code>CommConnection</code>
271  * would be used to discover available comm ports.
272  * </P>
273  * <PRE>
274  * String port1;
275  * String ports = System.getProperty("microedition.commports");
276  * int comma = ports.indexOf(',');
277  * if (comma > 0) {
278  *     // Parse the first port from the available ports list.
279  *     port1 = ports.substring(0, comma);
280  * } else {
281  *     // Only one serial port available.
282  *     port1 =ports;
283  * }
284  * </PRE>
285  * <H2>Recommended Port Naming Convention</H2>
286  * <P>
287  * Logical port names can be defined to match platform naming conventions
288  * using any combination of alphanumeric characters.  However, it is
289  * recommended that ports be named consistently among the implementations
290  * of this class according to a proposed convention. VM implementations
291  * should follow the following convention:<BR>
292  * Port names contain a text abbreviation indicating port capabilities followed
293  * by a sequential number for the port.  The following device name types should
294  * be used:
295  * </P>
296  * <UL>
297  * <LI>COM#, where COM is for RS-232 ports and # is a number assigned
298  * to the port</LI>
299  * <LI>IR#, where IR is for IrDA IRCOMM ports and # is a number assigned to
300  * the port </LI>
301  * </UL>
302  * <P>
303  * This naming scheme allows API users to generally determine the type of port
304  * that they would like to use.  For instance, if a application desires to
305  * "beam" a piece of data, the app could look for "IR#" ports for opening
306  * the connection.
307  * The alternative is a trial and error approach with all available ports.
308  * </P>
309  * @since MIDP 2.0
310  */
