52  /** 
53   * This class represents access to a file or directory.  A FilePermission consists
54   * of a pathname and a set of actions valid for that pathname.
55   * <P>
56   * Pathname is the pathname of the file or directory granted the specified
57   * actions. A pathname that ends in "/*" (where "/" is
58   * the file separator character, <code>File.separatorChar</code>) indicates
59   * all the files and directories contained in that directory. A pathname
60   * that ends with "/-" indicates (recursively) all files
61   * and subdirectories contained in that directory. A pathname consisting of
62   * the special token "&lt;&lt;ALL FILES&gt;&gt;" matches <b>any</b> file.
63   * <P>
64   * Note: A pathname consisting of a single "*" indicates all the files
65   * in the current directory, while a pathname consisting of a single "-" 
66   * indicates all the files in the current directory and
67   * (recursively) all files and subdirectories contained in the current 
68   * directory.
69   * <P>
70   * The actions to be granted are passed to the constructor in a string containing 
71   * a list of one or more comma-separated keywords. The possible keywords are
72   * "read", "write", "execute", and "delete". Their meaning is defined as follows:
73   * <P>
74   * <DL> 
75   *    <DT> read <DD> read permission
76   *    <DT> write <DD> write permission
77   *    <DT> execute 
78   *    <DD> execute permission. Allows <code>Runtime.exec</code> to
79   *         be called. Corresponds to <code>SecurityManager.checkExec</code>.
80   *    <DT> delete
81   *    <DD> delete permission. Allows <code>File.delete</code> to
82   *         be called. Corresponds to <code>SecurityManager.checkDelete</code>.
83   * </DL>
84   * <P>
85   * The actions string is converted to lowercase before processing.
86   * <P>
87   * Be careful when granting FilePermissions. Think about the implications 
88   * of granting read and especially write access to various files and 
89   * directories. The "&lt;&lt;ALL FILES>>" permission with write action is 
90   * especially dangerous. This grants permission to write to the entire 
91   * file system. One thing this effectively allows is replacement of the 
92   * system binary, including the JVM runtime environment.
93   * 
94   * <p>Please note: Code can always read a file from the same
95   * directory it's in (or a subdirectory of that directory); it does not
96   * need explicit permission to do so.
97   * 
98   * @see java.security.Permission
99   * @see java.security.Permissions
100  * @see java.security.PermissionCollection
101  *
102  * @version 1.67 00/02/02
