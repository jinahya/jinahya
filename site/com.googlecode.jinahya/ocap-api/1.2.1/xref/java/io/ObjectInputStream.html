47  /** 
48   * An ObjectInputStream deserializes primitive data and objects previously
49   * written using an ObjectOutputStream.
50   * 
51   * <p>ObjectOutputStream and ObjectInputStream can provide an application
52   * with persistent storage for graphs of objects when used with a
53   * FileOutputStream and FileInputStream respectively.
54   * ObjectInputStream is used to recover those objects previously
55   * serialized. Other uses include passing objects between hosts using
56   * a socket stream or for marshaling and unmarshaling arguments and
57   * parameters in a remote communication system.<p>
58   *
59   * ObjectInputStream ensures that the types of all objects in the
60   * graph created from the stream match the classes present in the
61   * Java Virtual Machine.  Classes are loaded as required using the
62   * standard mechanisms. <p>
63   *
64   * Only objects that support the java.io.Serializable or
65   * java.io.Externalizable interface can be read from streams.
66   *
67   * <p>The method <STRONG>readObject</STRONG> is used to read an object
68   * from the stream.  Java's safe casting should be used to get the
69   * desired type.  In Java, strings and arrays are objects and are
70   * treated as objects during serialization. When read they need to be
71   * cast to the expected type.<p>
72   *
73   * Primitive data types can be read from the stream using the appropriate
74   * method on DataInput. <p>
75   * 
76   * The default deserialization mechanism for objects restores the
77   * contents of each field to the value and type it had when it was written.
78   * Fields declared as transient or static are ignored by the
79   * deserialization process.  References to other objects cause those
80   * objects to be read from the stream as necessary.  Graphs of objects
81   * are restored correctly using a reference sharing mechanism.  New
82   * objects are always allocated when deserializing, which prevents
83   * existing objects from being overwritten.
84   *
85   * <p>Reading an object is analogous to running the constructors of a new
86   * object.  Memory is allocated for the object and initialized to zero (NULL).
87   * No-arg constructors are invoked for the non-serializable classes and then
88   * the fields of the serializable classes are restored from the stream starting
89   * with the serializable class closest to java.lang.object and finishing with
90   * the object's most specific class.
91   *
92   * <p>For example to read from a stream as written by the example in
93   * ObjectOutputStream: 
94   * <br>
95   * <pre>
96   *	FileInputStream fis = new FileInputStream("t.tmp");
97   *	ObjectInputStream ois = new ObjectInputStream(fis);
98   *
99   *	int i = ois.readInt();
100  *	String today = (String) ois.readObject();
101  *	Date date = (Date) ois.readObject();
102  *
103  *	ois.close();
104  * </pre>
105  *
106  * <p>Classes control how they are serialized by implementing either the
107  * java.io.Serializable or java.io.Externalizable interfaces.<P>
108  *
109  * Implementing the Serializable interface allows object serialization
110  * to save and restore the entire state of the object and it allows
111  * classes to evolve between the time the stream is written and the time it is
112  * read.  It automatically traverses references between objects,
113  * saving and restoring entire graphs.
114  *
115  * <p>Serializable classes that require special handling during the
116  * serialization and deserialization process should implement the following
117  * methods:<p>
118  *
119  * <PRE>
120  * private void writeObject(java.io.ObjectOutputStream stream)
121  *     throws IOException;
122  * private void readObject(java.io.ObjectInputStream stream)
123  *     throws IOException, ClassNotFoundException;
124  * private void readObjectNoData() 
125  *     throws ObjectStreamException; 
126  * </PRE><p>
127  *
128  * The readObject method is responsible for reading and restoring the
129  * state of the object for its particular class using data written to
130  * the stream by the corresponding writeObject method.  The method
131  * does not need to concern itself with the state belonging to its
132  * superclasses or subclasses.  State is restored by reading data from
133  * the ObjectInputStream for the individual fields and making
134  * assignments to the appropriate fields of the object.  Reading
135  * primitive data types is supported by DataInput. 
136  * 
137  * <p>Any attempt to read object data which exceeds the boundaries of the
138  * custom data written by the corresponding writeObject method will cause an
139  * OptionalDataException to be thrown with an eof field value of true.
140  * Non-object reads which exceed the end of the allotted data will reflect the
141  * end of data in the same way that they would indicate the end of the stream:
142  * bytewise reads will return -1 as the byte read or number of bytes read, and
143  * primitive reads will throw EOFExceptions.  If there is no corresponding
144  * writeObject method, then the end of default serialized data marks the end of
145  * the allotted data.
146  *
147  * <p>Primitive and object read calls issued from within a readExternal method
148  * behave in the same manner--if the stream is already positioned at the end of
149  * data written by the corresponding writeExternal method, object reads will
150  * throw OptionalDataExceptions with eof set to true, bytewise reads will
151  * return -1, and primitive reads will throw EOFExceptions.  Note that this
152  * behavior does not hold for streams written with the old
153  * <code>ObjectStreamConstants.PROTOCOL_VERSION_1</code> protocol, in which the
154  * end of data written by writeExternal methods is not demarcated, and hence
155  * cannot be detected.
156  *
157  * <p>The readObjectNoData method is responsible for initializing the state of
158  * the object for its particular class in the event that the serialization
159  * stream does not list the given class as a superclass of the object being
160  * deserialized.  This may occur in cases where the receiving party uses a
161  * different version of the deserialized instance's class than the sending
162  * party, and the receiver's version extends classes that are not extended by
163  * the sender's version.  This may also occur if the serialization stream has
164  * been tampered; hence, readObjectNoData is useful for initializing
165  * deserialized objects properly despite a "hostile" or incomplete source
166  * stream.
167  *
168  * <p>Serialization does not read or assign values to the fields of any
169  * object that does not implement the java.io.Serializable interface.
170  * Subclasses of Objects that are not serializable can be
171  * serializable. In this case the non-serializable class must have a
172  * no-arg constructor to allow its fields to be initialized.  In this
173  * case it is the responsibility of the subclass to save and restore
174  * the state of the non-serializable class. It is frequently the case that
175  * the fields of that class are accessible (public, package, or
176  * protected) or that there are get and set methods that can be used
177  * to restore the state.
178  *
179  * <p>Any exception that occurs while deserializing an object will be
180  * caught by the ObjectInputStream and abort the reading process.
181  *
182  * <p>Implementing the Externalizable interface allows the object to
183  * assume complete control over the contents and format of the object's
184  * serialized form.  The methods of the Externalizable interface,
185  * writeExternal and readExternal, are called to save and restore the
186  * objects state.  When implemented by a class they can write and read
187  * their own state using all of the methods of ObjectOutput and
188  * ObjectInput.  It is the responsibility of the objects to handle any
189  * versioning that occurs.
190  *
191  * @author	Mike Warres
192  * @author	Roger Riggs
193  * @version 1.135, 01/09/15
194  * @see java.io.DataInput
195  * @see java.io.ObjectOutputStream
196  * @see java.io.Serializable
197  * @see <a href="../../../guide/serialization/spec/input.doc.html"> Object Serialization Specification, Section 3, Object Input Classes</a>
198  * @since   JDK1.1
199  */
