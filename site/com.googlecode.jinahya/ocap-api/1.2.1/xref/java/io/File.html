49  /** 
50   * An abstract representation of file and directory pathnames.
51   *
52   * <p> User interfaces and operating systems use system-dependent <em>pathname
53   * strings</em> to name files and directories.  This class presents an
54   * abstract, system-independent view of hierarchical pathnames.  An
55   * <em>abstract pathname</em> has two components:
56   *
57   * <ol>
58   * <li> An optional system-dependent <em>prefix</em> string,
59   *      such as a disk-drive specifier, <code>"/"</code>&nbsp;for the UNIX root
60   *      directory, or <code>"\\"</code>&nbsp;for a Microsoft Windows UNC pathname, and
61   * <li> A sequence of zero or more string <em>names</em>.
62   * </ol>
63   *
64   * Each name in an abstract pathname except for the last denotes a directory;
65   * the last name may denote either a directory or a file.  The <em>empty</em>
66   * abstract pathname has no prefix and an empty name sequence.
67   *
68   * <p> The conversion of a pathname string to or from an abstract pathname is
69   * inherently system-dependent.  When an abstract pathname is converted into a
70   * pathname string, each name is separated from the next by a single copy of
71   * the default <em>separator character</em>.  The default name-separator
72   * character is defined by the system property <code>file.separator</code>, and
73   * is made available in the public static fields <code>{@link
74   * #separator}</code> and <code>{@link #separatorChar}</code> of this class.
75   * When a pathname string is converted into an abstract pathname, the names
76   * within it may be separated by the default name-separator character or by any
77   * other name-separator character that is supported by the underlying system.
78   *
79   * <p> A pathname, whether abstract or in string form, may be either
80   * <em>absolute</em> or <em>relative</em>.  An absolute pathname is complete in
81   * that no other information is required in order to locate the file that it
82   * denotes.  A relative pathname, in contrast, must be interpreted in terms of
83   * information taken from some other pathname.  By default the classes in the
84   * <code>java.io</code> package always resolve relative pathnames against the
85   * current user directory.  This directory is named by the system property
86   * <code>user.dir</code>, and is typically the directory in which the Java
87   * virtual machine was invoked.
88   *
89   * <p> The prefix concept is used to handle root directories on UNIX platforms,
90   * and drive specifiers, root directories and UNC pathnames on Microsoft Windows platforms,
91   * as follows:
92   *
93   * <ul>
94   *
95   * <li> For UNIX platforms, the prefix of an absolute pathname is always
96   * <code>"/"</code>.  Relative pathnames have no prefix.  The abstract pathname
97   * denoting the root directory has the prefix <code>"/"</code> and an empty
98   * name sequence.
99   *
100  * <li> For Microsoft Windows platforms, the prefix of a pathname that contains a drive
101  * specifier consists of the drive letter followed by <code>":"</code> and
102  * possibly followed by <code>"\"</code> if the pathname is absolute.  The
103  * prefix of a UNC pathname is <code>"\\"</code>; the hostname and the share
104  * name are the first two names in the name sequence.  A relative pathname that
105  * does not specify a drive has no prefix.
106  *
107  * </ul>
108  *
109  * <p> Instances of the <code>File</code> class are immutable; that is, once
110  * created, the abstract pathname represented by a <code>File</code> object
111  * will never change.
112  *
113  * @version 1.95 10/17/00
114  * @author  unascribed
115  * @since   JDK1.0
116  */
