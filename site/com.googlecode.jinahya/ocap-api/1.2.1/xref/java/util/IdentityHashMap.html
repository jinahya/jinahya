39  /** 
40   * This class implements the <tt>Map</tt> interface with a hash table, using
41   * reference-equality in place of object-equality when comparing keys (and
42   * values).  In other words, in an <tt>IdentityHashMap</tt>, two keys
43   * <tt>k1</tt> and <tt>k2</tt> are considered equal if and only if
44   * <tt>(k1==k2)</tt>.  (In normal <tt>Map</tt> implementations (like
45   * <tt>HashMap</tt>) two keys <tt>k1</tt> and <tt>k2</tt> are considered equal
46   * if and only if <tt>(k1==null ? k2==null : k1.equals(k2))</tt>.)
47   *
48   * <p><b>This class is <i>not</i> a general-purpose <tt>Map</tt>
49   * implementation!  While this class implements the <tt>Map</tt> interface, it
50   * intentionally violates <tt>Map's</tt> general contract, which mandates the
51   * use of the <tt>equals</tt> method when comparing objects.  This class is
52   * designed for use only in the rare cases wherein reference-equality
53   * semantics are required.</b>
54   *
55   * <p>A typical use of this class is <i>topology-preserving object graph
56   * transformations</i>, such as serialization or deep-copying.  To perform such
57   * a transformation, a program must maintain a "node table" that keeps track
58   * of all the object references that have already been processed.  The node
59   * table must not equate distinct objects even if they happen to be equal.
60   * Another typical use of this class is to maintain <i>proxy objects</i>.  For
61   * example, a debugging facility might wish to maintain a proxy object for
62   * each object in the program being debugged.
63   *
64   * <p>This class provides all of the optional map operations, and permits
65   * <tt>null</tt> values and the <tt>null</tt> key.  This class makes no
66   * guarantees as to the order of the map; in particular, it does not guarantee
67   * that the order will remain constant over time.
68   *
69   * <p>This class provides constant-time performance for the basic
70   * operations (<tt>get</tt> and <tt>put</tt>), assuming the system
71   * identity hash function ({@link System#identityHashCode(Object)})
72   * disperses elements properly among the buckets.
73   *
74   * <p>This class has one tuning parameter (which affects performance but not
75   * semantics): <i>expected maximum size</i>.  This parameter is the maximum
76   * number of key-value mappings that the map is expected to hold.  Internally,
77   * this parameter is used to determine the number of buckets initially
78   * comprising the hash table.  The precise relationship between the expected
79   * maximum size and the number of buckets is unspecified.
80   *
81   * <p>If the size of the map (the number of key-value mappings) sufficiently
82   * exceeds the expected maximum size, the number of buckets is increased
83   * Increasing the number of buckets ("rehashing") may be fairly expensive, so
84   * it pays to create identity hash maps with a sufficiently large expected
85   * maximum size.  On the other hand, iteration over collection views requires
86   * time proportional to the the number of buckets in the hash table, so it
87   * pays not to set the expected maximum size too high if you are especially
88   * concerned with iteration performance or memory usage.
89   *
90   * <p><b>Note that this implementation is not synchronized.</b> If multiple
91   * threads access this map concurrently, and at least one of the threads
92   * modifies the map structurally, it <i>must</i> be synchronized externally.
93   * (A structural modification is any operation that adds or deletes one or
94   * more mappings; merely changing the value associated with a key that an
95   * instance already contains is not a structural modification.)  This is
96   * typically accomplished by synchronizing on some object that naturally
97   * encapsulates the map.  If no such object exists, the map should be
98   * "wrapped" using the <tt>Collections.synchronizedMap</tt> method.  This is
99   * best done at creation time, to prevent accidental unsynchronized access to
100  * the map: <pre>
101  *     Map m = Collections.synchronizedMap(new HashMap(...));
102  * </pre>
103  *
104  * <p>The iterators returned by all of this class's "collection view methods"
105  * are <i>fail-fast</i>: if the map is structurally modified at any time after
106  * the iterator is created, in any way except through the iterator's own
107  * <tt>remove</tt> or <tt>add</tt> methods, the iterator will throw a
108  * <tt>ConcurrentModificationException</tt>.  Thus, in the face of concurrent
109  * modification, the iterator fails quickly and cleanly, rather than risking
110  * arbitrary, non-deterministic behavior at an undetermined time in the
111  * future.
112  *
113  * <p>Note that the fail-fast behavior of an iterator cannot be guaranteed
114  * as it is, generally speaking, impossible to make any hard guarantees in the
115  * presence of unsynchronized concurrent modification.  Fail-fast iterators
116  * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
117  * Therefore, it would be wrong to write a program that depended on this
118  * exception for its correctness: <i>fail-fast iterators should be used only
119  * to detect bugs.</i>
120  *
121  * <p>Implementation note: This is a simple <i>linear-probe</i> hash table,
122  * as described for example in texts by Sedgewick and Knuth.  The array
123  * alternates holding keys and values.  (This has better locality for large
124  * tables than does using separate arrays.)  For many JRE implementations
125  * and operation mixes, this class will yield better performance than
126  * {@link HashMap} (which uses <i>chaining</i> rather than linear-probing). 
127  *
128  * <p>This class is a member of the 
129  * <a href="{@docRoot}/../guide/collections/index.html">
130  * Java Collections Framework</a>.
131  *
132  * @see     System#identityHashCode(Object)
133  * @see     Object#hashCode()
134  * @see     Collection
135  * @see	    Map
136  * @see	    HashMap
137  * @see	    TreeMap
138  * @author  Doug Lea and Josh Bloch
139  * @since   1.4
140  */
