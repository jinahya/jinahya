41  /** 
42   * <p>Hash table and linked list implementation of the <tt>Map</tt> interface,
43   * with predictable iteration order.  This implementation differs from
44   * <tt>HashMap</tt> in that it maintains a doubly-linked list running through
45   * all of its entries.  This linked list defines the iteration ordering,
46   * which is normally the order in which keys were inserted into the map
47   * (<i>insertion-order</i>).  Note that insertion order is not affected
48   * if a key is <i>re-inserted</i> into the map.  (A key <tt>k</tt> is
49   * reinserted into a map <tt>m</tt> if <tt>m.put(k, v)</tt> is invoked when
50   * <tt>m.containsKey(k)</tt> would return <tt>true</tt> immediately prior to
51   * the invocation.)
52   *
53   * <p>This implementation spares its clients from the unspecified, generally
54   * chaotic ordering provided by {@link HashMap} (and {@link Hashtable}),
55   * without incurring the increased cost associated with {@link TreeMap}.  It
56   * can be used to produce a copy of a map that has the same order as the
57   * original, regardless of the original map's implementation:
58   * <pre>
59   *     void foo(Map m) {
60   *         Map copy = new LinkedHashMap(m);
61   *         ...
62   *     }
63   * </pre>
64   * This technique is particularly useful if a module takes a map on input,
65   * copies it, and later returns results whose order is determined by that of
66   * the copy.  (Clients generally appreciate having things returned in the same
67   * order they were presented.)
68   *
69   * <p>A special {@link #LinkedHashMap(int,float,boolean) constructor} is
70   * provided to create a linked hash map whose order of iteration is the order
71   * in which its entries were last accessed, from least-recently accessed to
72   * most-recently (<i>access-order</i>).  This kind of map is well-suited to
73   * building LRU caches.  Invoking the <tt>put</tt> or <tt>get</tt> method
74   * results in an access to the corresponding entry (assuming it exists after
75   * the invocation completes).  The <tt>putAll</tt> method generates one entry
76   * access for each mapping in the specified map, in the order that key-value
77   * mappings are provided by the specified map's entry set iterator.  <i>No
78   * other methods generate entry accesses.</i> In particular, operations on
79   * collection-views do <i>not</i> affect the order of iteration of the backing
80   * map.
81   *
82   * <p>The {@link #removeEldestEntry(Map.Entry)} method may be overridden to 
83   * impose a policy for removing stale mappings automatically when new mappings
84   * are added to the map.
85   *
86   * <p>This class provides all of the optional <tt>Map</tt> operations, and
87   * permits null elements.  Like <tt>HashMap</tt>, it provides constant-time
88   * performance for the basic operations (<tt>add</tt>, <tt>contains</tt> and
89   * <tt>remove</tt>), assuming the the hash function disperses elements
90   * properly among the buckets.  Performance is likely to be just slightly
91   * below that of <tt>HashMap</tt>, due to the added expense of maintaining the
92   * linked list, with one exception: Iteration over the collection-views
93   * of a <tt>LinkedHashMap</tt> requires time proportional to the <i>size</i>
94   * of the map, regardless of its capacity.  Iteration over a <tt>HashMap</tt>
95   * is likely to be more expensive, requiring time proportional to its
96   * <i>capacity</i>.
97   *
98   * <p>A linked hash map has two parameters that affect its performance:
99   * <i>initial capacity</i> and <i>load factor</i>.  They are defined precisely
100  * as for <tt>HashMap</tt>.  Note, however, that the penalty for choosing an
101  * excessively high value for initial capacity is less severe for this class
102  * than for <tt>HashMap</tt>, as iteration times for this class are unaffected
103  * by capacity.
104  * 
105  * <p><strong>Note that this implementation is not synchronized.</strong> If
106  * multiple threads access a linked hash map concurrently, and at least
107  * one of the threads modifies the map structurally, it <em>must</em> be
108  * synchronized externally.  This is typically accomplished by synchronizing
109  * on some object that naturally encapsulates the map.  If no such object
110  * exists, the map should be "wrapped" using the
111  * <tt>Collections.synchronizedMap</tt>method.  This is best done at creation
112  * time, to prevent accidental unsynchronized access:
113  * <pre>
114  *    Map m = Collections.synchronizedMap(new LinkedHashMap(...));
115  * </pre>
116  * A structural modification is any operation that adds or deletes one or more
117  * mappings or, in the case of access-ordered linked hash maps, affects
118  * iteration order.  In insertion-ordered linked hash maps, merely changing
119  * the value associated with a key that is already contained in the map is not
120  * a structural modification.  <strong>In access-ordered linked hash maps,
121  * merely querying the map with <tt>get</tt> is a structural
122  * modification.</strong>)
123  *
124  * <p>The iterators returned by the <tt>iterator</tt> methods of the
125  * collections returned by all of this class's collection view methods are
126  * <em>fail-fast</em>: if the map is structurally modified at any time after
127  * the iterator is created, in any way except through the iterator's own
128  * remove method, the iterator will throw a
129  * <tt>ConcurrentModificationException</tt>.  Thus, in the face of concurrent
130  * modification, the Iterator fails quickly and cleanly, rather than risking
131  * arbitrary, non-deterministic behavior at an undetermined time in the
132  * future.
133  *
134  * <p>Note that the fail-fast behavior of an iterator cannot be guaranteed
135  * as it is, generally speaking, impossible to make any hard guarantees in the
136  * presence of unsynchronized concurrent modification.  Fail-fast iterators
137  * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
138  * Therefore, it would be wrong to write a program that depended on this
139  * exception for its correctness:   <i>the fail-fast behavior of iterators
140  * should be used only to detect bugs.</i>
141  *
142  * <p>This class is a member of the 
143  * <a href="{@docRoot}/../guide/collections/index.html">
144  * Java Collections Framework</a>.
145  *
146  * @author  Josh Bloch
147  * @version 1.12, 03/12/05
148  * @see     Object#hashCode()
149  * @see     Collection
150  * @see     Map
151  * @see     HashMap
152  * @see     TreeMap
153  * @see     Hashtable
154  * @since   JDK1.4
155  */

259      * Returns <tt>true</tt> if this map should remove its eldest entry.

266      * <p>Sample use: this override will allow the map to grow up to 100

268      * added, maintaining a steady state of 100 entries.
269      * <pre>

271      *
272      *     protected boolean removeEldestEntry(Map.Entry eldest) {
273      *        return size() > MAX_ENTRIES;
274      *     }

278      * instead allowing the map to modify itself as directed by its
279      * return value.  It <i>is</i> permitted for this method to modify
280      * the map directly, but if it does so, it <i>must</i> return
281      * <tt>false</tt> (indicating that the map should not attempt any
282      * further modification).  The effects of returning <tt>true</tt>
283      * after modifying the map from within this method are unspecified.
284      *
285      * <p>This implementation merely returns <tt>false</tt> (so that this
286      * map acts like a normal map - the eldest element is never removed).
287      *
