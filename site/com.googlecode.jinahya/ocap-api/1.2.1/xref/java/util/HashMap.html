41  /** 
42   * Hash table based implementation of the <tt>Map</tt> interface.  This
43   * implementation provides all of the optional map operations, and permits
44   * <tt>null</tt> values and the <tt>null</tt> key.  (The <tt>HashMap</tt>
45   * class is roughly equivalent to <tt>Hashtable</tt>, except that it is
46   * unsynchronized and permits nulls.)  This class makes no guarantees as to
47   * the order of the map; in particular, it does not guarantee that the order
48   * will remain constant over time.
49   *
50   * <p>This implementation provides constant-time performance for the basic
51   * operations (<tt>get</tt> and <tt>put</tt>), assuming the hash function
52   * disperses the elements properly among the buckets.  Iteration over
53   * collection views requires time proportional to the "capacity" of the
54   * <tt>HashMap</tt> instance (the number of buckets) plus its size (the number
55   * of key-value mappings).  Thus, it's very important not to set the initial
56   * capacity too high (or the load factor too low) if iteration performance is
57   * important.
58   *
59   * <p>An instance of <tt>HashMap</tt> has two parameters that affect its
60   * performance: <i>initial capacity</i> and <i>load factor</i>.  The
61   * <i>capacity</i> is the number of buckets in the hash table, and the initial
62   * capacity is simply the capacity at the time the hash table is created.  The
63   * <i>load factor</i> is a measure of how full the hash table is allowed to
64   * get before its capacity is automatically increased.  When the number of
65   * entries in the hash table exceeds the product of the load factor and the
66   * current capacity, the capacity is roughly doubled by calling the
67   * <tt>rehash</tt> method.
68   *
69   * <p>As a general rule, the default load factor (.75) offers a good tradeoff
70   * between time and space costs.  Higher values decrease the space overhead
71   * but increase the lookup cost (reflected in most of the operations of the
72   * <tt>HashMap</tt> class, including <tt>get</tt> and <tt>put</tt>).  The
73   * expected number of entries in the map and its load factor should be taken
74   * into account when setting its initial capacity, so as to minimize the
75   * number of <tt>rehash</tt> operations.  If the initial capacity is greater
76   * than the maximum number of entries divided by the load factor, no
77   * <tt>rehash</tt> operations will ever occur.
78   *
79   * <p>If many mappings are to be stored in a <tt>HashMap</tt> instance,
80   * creating it with a sufficiently large capacity will allow the mappings to
81   * be stored more efficiently than letting it perform automatic rehashing as
82   * needed to grow the table.
83   *
84   * <p><b>Note that this implementation is not synchronized.</b> If multiple
85   * threads access this map concurrently, and at least one of the threads
86   * modifies the map structurally, it <i>must</i> be synchronized externally.
87   * (A structural modification is any operation that adds or deletes one or
88   * more mappings; merely changing the value associated with a key that an
89   * instance already contains is not a structural modification.)  This is
90   * typically accomplished by synchronizing on some object that naturally
91   * encapsulates the map.  If no such object exists, the map should be
92   * "wrapped" using the <tt>Collections.synchronizedMap</tt> method.  This is
93   * best done at creation time, to prevent accidental unsynchronized access to
94   * the map: <pre> Map m = Collections.synchronizedMap(new HashMap(...));
95   * </pre>
96   *
97   * <p>The iterators returned by all of this class's "collection view methods"
98   * are <i>fail-fast</i>: if the map is structurally modified at any time after
99   * the iterator is created, in any way except through the iterator's own
100  * <tt>remove</tt> or <tt>add</tt> methods, the iterator will throw a
101  * <tt>ConcurrentModificationException</tt>.  Thus, in the face of concurrent
102  * modification, the iterator fails quickly and cleanly, rather than risking
103  * arbitrary, non-deterministic behavior at an undetermined time in the
104  * future.
105  *
106  * <p>Note that the fail-fast behavior of an iterator cannot be guaranteed
107  * as it is, generally speaking, impossible to make any hard guarantees in the
108  * presence of unsynchronized concurrent modification.  Fail-fast iterators
109  * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
110  * Therefore, it would be wrong to write a program that depended on this
111  * exception for its correctness: <i>the fail-fast behavior of iterators
112  * should be used only to detect bugs.</i>
113  *
114  * <p>This class is a member of the 
115  * <a href="{@docRoot}/../guide/collections/index.html">
116  * Java Collections Framework</a>.
117  *
