39  /** 
40   * Red-Black tree based implementation of the <tt>SortedMap</tt> interface.
41   * This class guarantees that the map will be in ascending key order, sorted
42   * according to the <i>natural order</i> for the key's class (see
43   * <tt>Comparable</tt>), or by the comparator provided at creation time,
44   * depending on which constructor is used.<p>
45   *
46   * This implementation provides guaranteed log(n) time cost for the
47   * <tt>containsKey</tt>, <tt>get</tt>, <tt>put</tt> and <tt>remove</tt>
48   * operations.  Algorithms are adaptations of those in Cormen, Leiserson, and
49   * Rivest's <I>Introduction to Algorithms</I>.<p>
50   *
51   * Note that the ordering maintained by a sorted map (whether or not an
52   * explicit comparator is provided) must be <i>consistent with equals</i> if
53   * this sorted map is to correctly implement the <tt>Map</tt> interface.  (See
54   * <tt>Comparable</tt> or <tt>Comparator</tt> for a precise definition of
55   * <i>consistent with equals</i>.)  This is so because the <tt>Map</tt>
56   * interface is defined in terms of the equals operation, but a map performs
57   * all key comparisons using its <tt>compareTo</tt> (or <tt>compare</tt>)
58   * method, so two keys that are deemed equal by this method are, from the
59   * standpoint of the sorted map, equal.  The behavior of a sorted map
60   * <i>is</i> well-defined even if its ordering is inconsistent with equals; it
61   * just fails to obey the general contract of the <tt>Map</tt> interface.<p>
62   *
63   * <b>Note that this implementation is not synchronized.</b> If multiple
64   * threads access a map concurrently, and at least one of the threads modifies
65   * the map structurally, it <i>must</i> be synchronized externally.  (A
66   * structural modification is any operation that adds or deletes one or more
67   * mappings; merely changing the value associated with an existing key is not
68   * a structural modification.)  This is typically accomplished by
69   * synchronizing on some object that naturally encapsulates the map.  If no
70   * such object exists, the map should be "wrapped" using the
71   * <tt>Collections.synchronizedMap</tt> method.  This is best done at creation
72   * time, to prevent accidental unsynchronized access to the map: 
73   * <pre>
74   *     Map m = Collections.synchronizedMap(new TreeMap(...));
75   * </pre><p>
76   *
77   * The iterators returned by all of this class's "collection view methods" are
78   * <i>fail-fast</i>: if the map is structurally modified at any time after the
79   * iterator is created, in any way except through the iterator's own
80   * <tt>remove</tt> or <tt>add</tt> methods, the iterator throws a
81   * <tt>ConcurrentModificationException</tt>.  Thus, in the face of concurrent
82   * modification, the iterator fails quickly and cleanly, rather than risking
83   * arbitrary, non-deterministic behavior at an undetermined time in the
84   * future.
85   *
86   * <p>Note that the fail-fast behavior of an iterator cannot be guaranteed
87   * as it is, generally speaking, impossible to make any hard guarantees in the
88   * presence of unsynchronized concurrent modification.  Fail-fast iterators
89   * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
90   * Therefore, it would be wrong to write a program that depended on this
91   * exception for its correctness:   <i>the fail-fast behavior of iterators
92   * should be used only to detect bugs.</i><p>
93   *
94   * This class is a member of the 
95   * <a href="{@docRoot}/../guide/collections/index.html">
96   * Java Collections Framework</a>.
97   *
98   * @author  Josh Bloch and Doug Lea
99   * @version 1.43, 02/02/00
100  * @see Map
101  * @see HashMap
102  * @see Hashtable
103  * @see Comparable
104  * @see Comparator
105  * @see Collection
106  * @see Collections#synchronizedMap(Map)
107  * @since 1.2
108  */

398      * Returns a view of the portion of this map whose keys range from
399      * <tt>fromKey</tt>, inclusive, to <tt>toKey</tt>, exclusive.  (If
400      * <tt>fromKey</tt> and <tt>toKey</tt> are equal, the returned sorted map
401      * is empty.)  The returned sorted map is backed by this map, so changes
402      * in the returned sorted map are reflected in this map, and vice-versa.
403      * The returned sorted map supports all optional map operations.<p>
404      *
405      * The sorted map returned by this method will throw an
406      * <tt>IllegalArgumentException</tt> if the user attempts to insert a key
407      * less than <tt>fromKey</tt> or greater than or equal to

409      *

424      *             <pre>    SortedMap sub = m.subMap(low+"\0", high);</pre>
425      *
426      * @param fromKey low endpoint (inclusive) of the subMap.
427      * @param toKey high endpoint (exclusive) of the subMap.
