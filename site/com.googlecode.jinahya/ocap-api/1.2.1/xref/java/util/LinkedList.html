39  /** 
40   * Linked list implementation of the <tt>List</tt> interface.  Implements all
41   * optional list operations, and permits all elements (including
42   * <tt>null</tt>).  In addition to implementing the <tt>List</tt> interface,
43   * the <tt>LinkedList</tt> class provides uniformly named methods to
44   * <tt>get</tt>, <tt>remove</tt> and <tt>insert</tt> an element at the
45   * beginning and end of the list.  These operations allow linked lists to be
46   * used as a stack, queue, or double-ended queue (deque).<p>
47   *
48   * All of the stack/queue/deque operations could be easily recast in terms of
49   * the standard list operations.  They're included here primarily for
50   * convenience, though they may run slightly faster than the equivalent List
51   * operations.<p>
52   *
53   * All of the operations perform as could be expected for a doubly-linked
54   * list.  Operations that index into the list will traverse the list from
55   * the begining or the end, whichever is closer to the specified index.<p>
56   *
57   * <b>Note that this implementation is not synchronized.</b> If multiple
58   * threads access a list concurrently, and at least one of the threads
59   * modifies the list structurally, it <i>must</i> be synchronized
60   * externally.  (A structural modification is any operation that adds or
61   * deletes one or more elements; merely setting the value of an element is not
62   * a structural modification.)  This is typically accomplished by
63   * synchronizing on some object that naturally encapsulates the list.  If no
64   * such object exists, the list should be "wrapped" using the
65   * Collections.synchronizedList method.  This is best done at creation time,
66   * to prevent accidental unsynchronized access to the list: <pre>
67   *     List list = Collections.synchronizedList(new LinkedList(...));
68   * </pre><p>
69   *
70   * The iterators returned by the this class's <tt>iterator</tt> and
71   * <tt>listIterator</tt> methods are <i>fail-fast</i>: if the list is
72   * structurally modified at any time after the iterator is created, in any way
73   * except through the Iterator's own <tt>remove</tt> or <tt>add</tt> methods,
74   * the iterator will throw a <tt>ConcurrentModificationException</tt>.  Thus,
75   * in the face of concurrent modification, the iterator fails quickly and
76   * cleanly, rather than risking arbitrary, non-deterministic behavior at an
77   * undetermined time in the future.
78   *
79   * <p>Note that the fail-fast behavior of an iterator cannot be guaranteed
80   * as it is, generally speaking, impossible to make any hard guarantees in the
81   * presence of unsynchronized concurrent modification.  Fail-fast iterators
82   * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
83   * Therefore, it would be wrong to write a program that depended on this
84   * exception for its correctness:   <i>the fail-fast behavior of iterators
85   * should be used only to detect bugs.</i><p>
86   *
87   * This class is a member of the 
88   * <a href="{@docRoot}/../guide/collections/index.html">
89   * Java Collections Framework</a>.
90   *

92   * @version 1.37, 03/12/05 
93   * @see	    List
94   * @see	    ArrayList
