49  /** 
50   * The class <code>Date</code> represents a specific instant
51   * in time, with millisecond precision.
52   * <p>
53   * Prior to JDK&nbsp;1.1, the class <code>Date</code> had two additional
54   * functions.  It allowed the interpretation of dates as year, month, day, hour,
55   * minute, and second values.  It also allowed the formatting and parsing
56   * of date strings.  Unfortunately, the API for these functions was not
57   * amenable to internationalization.  As of JDK&nbsp;1.1, the
58   * <code>Calendar</code> class should be used to convert between dates and time
59   * fields and the <code>DateFormat</code> class should be used to format and
60   * parse date strings.
61   * The corresponding methods in <code>Date</code> are deprecated.
62   * <p>
63   * Although the <code>Date</code> class is intended to reflect 
64   * coordinated universal time (UTC), it may not do so exactly, 
65   * depending on the host environment of the Java Virtual Machine. 
66   * Nearly all modern operating systems assume that 1&nbsp;day&nbsp;=
67   * 24&nbsp;&times;&nbsp;60&nbsp;&times;&nbsp;60&nbsp;= 86400 seconds 
68   * in all cases. In UTC, however, about once every year or two there 
69   * is an extra second, called a "leap second." The leap 
70   * second is always added as the last second of the day, and always 
71   * on December 31 or June 30. For example, the last minute of the 
72   * year 1995 was 61 seconds long, thanks to an added leap second. 
73   * Most computer clocks are not accurate enough to be able to reflect 
74   * the leap-second distinction. 
75   * <p>
76   * Some computer standards are defined in terms of Greenwich mean 
77   * time (GMT), which is equivalent to universal time (UT).  GMT is 
78   * the "civil" name for the standard; UT is the 
79   * "scientific" name for the same standard. The 
80   * distinction between UTC and UT is that UTC is based on an atomic 
81   * clock and UT is based on astronomical observations, which for all 
82   * practical purposes is an invisibly fine hair to split. Because the 
83   * earth's rotation is not uniform (it slows down and speeds up 
84   * in complicated ways), UT does not always flow uniformly. Leap 
85   * seconds are introduced as needed into UTC so as to keep UTC within 
86   * 0.9 seconds of UT1, which is a version of UT with certain 
87   * corrections applied. There are other time and date systems as 
88   * well; for example, the time scale used by the satellite-based 
89   * global positioning system (GPS) is synchronized to UTC but is 
90   * <i>not</i> adjusted for leap seconds. An interesting source of 
91   * further information is the U.S. Naval Observatory, particularly 
92   * the Directorate of Time at:
93   * <blockquote><pre>
94   *     <a href=http://tycho.usno.navy.mil>http://tycho.usno.navy.mil</a>
95   * </pre></blockquote>
96   * <p>
97   * and their definitions of "Systems of Time" at:
98   * <blockquote><pre>
99   *     <a href=http://tycho.usno.navy.mil/systime.html>http://tycho.usno.navy.mil/systime.html</a>
100  * </pre></blockquote>
101  * <p>
102  * In all methods of class <code>Date</code> that accept or return 
103  * year, month, date, hours, minutes, and seconds values, the 
104  * following representations are used: 
105  * <ul>
106  * <li>A year <i>y</i> is represented by the integer 
107  *     <i>y</i>&nbsp;<code>-&nbsp;1900</code>. 
108  * <li>A month is represented by an integer from 0 to 11; 0 is January, 
109  *     1 is February, and so forth; thus 11 is December. 
110  * <li>A date (day of month) is represented by an integer from 1 to 31 
111  *     in the usual manner. 
112  * <li>An hour is represented by an integer from 0 to 23. Thus, the hour 
113  *     from midnight to 1 a.m. is hour 0, and the hour from noon to 1 
114  *     p.m. is hour 12. 
115  * <li>A minute is represented by an integer from 0 to 59 in the usual manner.
116  * <li>A second is represented by an integer from 0 to 61; the values 60 and 
117  *     61 occur only for leap seconds and even then only in Java 
118  *     implementations that actually track leap seconds correctly. Because 
119  *     of the manner in which leap seconds are currently introduced, it is 
120  *     extremely unlikely that two leap seconds will occur in the same 
121  *     minute, but this specification follows the date and time conventions 
122  *     for ISO C.
123  * </ul>
124  * <p>
125  * In all cases, arguments given to methods for these purposes need 
126  * not fall within the indicated ranges; for example, a date may be 
127  * specified as January 32 and is interpreted as meaning February 1.
128  *
129  * @author  James Gosling
