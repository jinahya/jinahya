39  /** 
40   * The root interface in the <i>collection hierarchy</i>.  A collection
41   * represents a group of objects, known as its <i>elements</i>.  Some
42   * collections allow duplicate elements and others do not.  Some are ordered
43   * and others unordered.  The SDK does not provide any <i>direct</i>
44   * implementations of this interface: it provides implementations of more
45   * specific subinterfaces like <tt>Set</tt> and <tt>List</tt>.  This interface
46   * is typically used to pass collections around and manipulate them where
47   * maximum generality is desired.
48   *
49   * <p><i>Bags</i> or <i>multisets</i> (unordered collections that may contain
50   * duplicate elements) should implement this interface directly.
51   *
52   * <p>All general-purpose <tt>Collection</tt> implementation classes (which
53   * typically implement <tt>Collection</tt> indirectly through one of its
54   * subinterfaces) should provide two "standard" constructors: a void (no
55   * arguments) constructor, which creates an empty collection, and a
56   * constructor with a single argument of type <tt>Collection</tt>, which
57   * creates a new collection with the same elements as its argument.  In
58   * effect, the latter constructor allows the user to copy any collection,
59   * producing an equivalent collection of the desired implementation type.
60   * There is no way to enforce this convention (as interfaces cannot contain
61   * constructors) but all of the general-purpose <tt>Collection</tt>
62   * implementations in the Java platform libraries comply.
63   *
64   * <p>The "destructive" methods contained in this interface, that is, the
65   * methods that modify the collection on which they operate, are specified to
66   * throw <tt>UnsupportedOperationException</tt> if this collection does not
67   * support the operation.  If this is the case, these methods may, but are not
68   * required to, throw an <tt>UnsupportedOperationException</tt> if the
69   * invocation would have no effect on the collection.  For example, invoking
70   * the {@link #addAll(Collection)} method on an unmodifiable collection may,
71   * but is not required to, throw the exception if the collection to be added
72   * is empty.
73   *
74   * <p>Some collection implementations have restrictions on the elements that
75   * they may contain.  For example, some implementations prohibit null elements,
76   * and some have restrictions on the types of their elements.  Attempting to
77   * add an ineligible element throws an unchecked exception, typically
78   * <tt>NullPointerException</tt> or <tt>ClassCastException</tt>.  Attempting
79   * to query the presence of an ineligible element may throw an exception,
80   * or it may simply return false; some implementations will exhibit the former
81   * behavior and some will exhibit the latter.  More generally, attempting an
82   * operation on an ineligible element whose completion would not result in
83   * the insertion of an ineligible element into the collection may throw an
84   * exception or it may succeed, at the option of the implementation.
85   * Such exceptions are marked as "optional" in the specification for this
86   * interface. 
87   *
88   * <p>This interface is a member of the 
89   * <a href="{@docRoot}/../guide/collections/index.html">
90   * Java Collections Framework</a>.
91   *
92   * @author  Josh Bloch
93   * @version 1.31, 02/02/00
94   * @see	    Set
95   * @see	    List
96   * @see	    Map
97   * @see	    SortedSet
98   * @see	    SortedMap
99   * @see	    HashSet
100  * @see	    TreeSet
101  * @see	    ArrayList
102  * @see	    LinkedList
103  * @see	    Vector
104  * @see     Collections
105  * @see	    Arrays
106  * @see	    AbstractCollection
107  * @since 1.2
108  */

380     /** 
381      * Compares the specified object with this collection for equality. <p>
382      *
383      * While the <tt>Collection</tt> interface adds no stipulations to the
384      * general contract for the <tt>Object.equals</tt>, programmers who
385      * implement the <tt>Collection</tt> interface "directly" (in other words,
386      * create a class that is a <tt>Collection</tt> but is not a <tt>Set</tt>
387      * or a <tt>List</tt>) must exercise care if they choose to override the
388      * <tt>Object.equals</tt>.  It is not necessary to do so, and the simplest
389      * course of action is to rely on <tt>Object</tt>'s implementation, but
390      * the implementer may wish to implement a "value comparison" in place of
391      * the default "reference comparison."  (The <tt>List</tt> and
392      * <tt>Set</tt> interfaces mandate such value comparisons.)<p>
393      *
394      * The general contract for the <tt>Object.equals</tt> method states that
395      * equals must be symmetric (in other words, <tt>a.equals(b)</tt> if and
396      * only if <tt>b.equals(a)</tt>).  The contracts for <tt>List.equals</tt>
397      * and <tt>Set.equals</tt> state that lists are only equal to other lists,
398      * and sets to other sets.  Thus, a custom <tt>equals</tt> method for a
399      * collection class that implements neither the <tt>List</tt> nor
400      * <tt>Set</tt> interface must return <tt>false</tt> when this collection
401      * is compared to any list or set.  (By the same logic, it is not possible
402      * to write a class that correctly implements both the <tt>Set</tt> and
403      * <tt>List</tt> interfaces.)
404      *
405      * @param o Object to be compared for equality with this collection.
406      * @return <tt>true</tt> if the specified object is equal to this
407      * collection
408      * 
409      * @see Object#equals(Object)
410      * @see Set#equals(Object)
411      * @see List#equals(Object)
412      */
