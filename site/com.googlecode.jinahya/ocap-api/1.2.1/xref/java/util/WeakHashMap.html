42  /** 
43   * A hashtable-based <tt>Map</tt> implementation with <em>weak keys</em>.
44   * An entry in a <tt>WeakHashMap</tt> will automatically be removed when
45   * its key is no longer in ordinary use.  More precisely, the presence of a
46   * mapping for a given key will not prevent the key from being discarded by the
47   * garbage collector, that is, made finalizable, finalized, and then reclaimed.
48   * When a key has been discarded its entry is effectively removed from the map,
49   * so this class behaves somewhat differently than other <tt>Map</tt>
50   * implementations.
51   *
52   * <p> Both null values and the null key are supported. This class has
53   * performance characteristics similar to those of the <tt>HashMap</tt>
54   * class, and has the same efficiency parameters of <em>initial capacity</em>
55   * and <em>load factor</em>.
56   *
57   * <p> Like most collection classes, this class is not synchronized.  A
58   * synchronized <tt>WeakHashMap</tt> may be constructed using the
59   * <tt>Collections.synchronizedMap</tt> method.
60   *
61   * <p> This class is intended primarily for use with key objects whose
62   * <tt>equals</tt> methods test for object identity using the
63   * <tt>==</tt> operator.  Once such a key is discarded it can never be
64   * recreated, so it is impossible to do a lookup of that key in a
65   * <tt>WeakHashMap</tt> at some later time and be surprised that its entry
66   * has been removed.  This class will work perfectly well with key objects
67   * whose <tt>equals</tt> methods are not based upon object identity, such
68   * as <tt>String</tt> instances.  With such recreatable key objects,
69   * however, the automatic removal of <tt>WeakHashMap</tt> entries whose
70   * keys have been discarded may prove to be confusing.
71   *
72   * <p> The behavior of the <tt>WeakHashMap</tt> class depends in part upon
73   * the actions of the garbage collector, so several familiar (though not
74   * required) <tt>Map</tt> invariants do not hold for this class.  Because
75   * the garbage collector may discard keys at any time, a
76   * <tt>WeakHashMap</tt> may behave as though an unknown thread is silently
77   * removing entries.  In particular, even if you synchronize on a
78   * <tt>WeakHashMap</tt> instance and invoke none of its mutator methods, it
79   * is possible for the <tt>size</tt> method to return smaller values over
80   * time, for the <tt>isEmpty</tt> method to return <tt>false</tt> and
81   * then <tt>true</tt>, for the <tt>containsKey</tt> method to return
82   * <tt>true</tt> and later <tt>false</tt> for a given key, for the
83   * <tt>get</tt> method to return a value for a given key but later return
84   * <tt>null</tt>, for the <tt>put</tt> method to return
85   * <tt>null</tt> and the <tt>remove</tt> method to return
86   * <tt>false</tt> for a key that previously appeared to be in the map, and
87   * for successive examinations of the key set, the value set, and the entry set
88   * to yield successively smaller numbers of elements.
89   *
90   * <p> Each key object in a <tt>WeakHashMap</tt> is stored indirectly as
91   * the referent of a weak reference.  Therefore a key will automatically be
92   * removed only after the weak references to it, both inside and outside of the
93   * map, have been cleared by the garbage collector.
94   *
95   * <p> <strong>Implementation note:</strong> The value objects in a
96   * <tt>WeakHashMap</tt> are held by ordinary strong references.  Thus care
97   * should be taken to ensure that value objects do not strongly refer to their
98   * own keys, either directly or indirectly, since that will prevent the keys
99   * from being discarded.  Note that a value object may refer indirectly to its
100  * key via the <tt>WeakHashMap</tt> itself; that is, a value object may
101  * strongly refer to some other key object whose associated value object, in
102  * turn, strongly refers to the key of the first value object.  One way
103  * to deal with this is to wrap values themselves within 
104  * <tt>WeakReferences</tt> before
105  * inserting, as in: <tt>m.put(key, new WeakReference(value))</tt>,
106  * and then unwrapping upon each <tt>get</tt>.
107  *
108  * <p>The iterators returned by all of this class's "collection view methods"
109  * are <i>fail-fast</i>: if the map is structurally modified at any time after
110  * the iterator is created, in any way except through the iterator's own
111  * <tt>remove</tt> or <tt>add</tt> methods, the iterator will throw a
112  * <tt>ConcurrentModificationException</tt>.  Thus, in the face of concurrent
113  * modification, the iterator fails quickly and cleanly, rather than risking
114  * arbitrary, non-deterministic behavior at an undetermined time in the
115  * future.
116  *
117  * <p>Note that the fail-fast behavior of an iterator cannot be guaranteed
118  * as it is, generally speaking, impossible to make any hard guarantees in the
119  * presence of unsynchronized concurrent modification.  Fail-fast iterators
120  * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
121  * Therefore, it would be wrong to write a program that depended on this
122  * exception for its correctness:  <i>the fail-fast behavior of iterators
123  * should be used only to detect bugs.</i>
124  *
125  * <p>This class is a member of the 
126  * <a href="{@docRoot}/../guide/collections/index.html">
127  * Java Collections Framework</a>.
128  *
129  * @version	1.17, 03/12/05
130  * @author      Doug Lea
131  * @author      Josh Bloch
132  * @author	Mark Reinhold
133  * @since	1.2
134  * @see		java.util.HashMap
135  * @see		java.lang.ref.WeakReference
136  */
