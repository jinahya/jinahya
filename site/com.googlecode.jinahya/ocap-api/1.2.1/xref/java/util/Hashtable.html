41  /** 
42   * This class implements a hashtable, which maps keys to values. Any 
43   * non-<code>null</code> object can be used as a key or as a value. <p>
44   *
45   * To successfully store and retrieve objects from a hashtable, the 
46   * objects used as keys must implement the <code>hashCode</code> 
47   * method and the <code>equals</code> method. <p>
48   *
49   * An instance of <code>Hashtable</code> has two parameters that affect its
50   * performance: <i>initial capacity</i> and <i>load factor</i>.  The
51   * <i>capacity</i> is the number of <i>buckets</i> in the hash table, and the
52   * <i>initial capacity</i> is simply the capacity at the time the hash table
53   * is created.  Note that the hash table is <i>open</i>: in the case of a "hash
54   * collision", a single bucket stores multiple entries, which must be searched
55   * sequentially.  The <i>load factor</i> is a measure of how full the hash
56   * table is allowed to get before its capacity is automatically increased.
57   * When the number of entries in the hashtable exceeds the product of the load
58   * factor and the current capacity, the capacity is increased by calling the
59   * <code>rehash</code> method.<p>
60   *
61   * Generally, the default load factor (.75) offers a good tradeoff between
62   * time and space costs.  Higher values decrease the space overhead but
63   * increase the time cost to look up an entry (which is reflected in most
64   * <tt>Hashtable</tt> operations, including <tt>get</tt> and <tt>put</tt>).<p>
65   *
66   * The initial capacity controls a tradeoff between wasted space and the
67   * need for <code>rehash</code> operations, which are time-consuming.
68   * No <code>rehash</code> operations will <i>ever</i> occur if the initial
69   * capacity is greater than the maximum number of entries the
70   * <tt>Hashtable</tt> will contain divided by its load factor.  However,
71   * setting the initial capacity too high can waste space.<p>
72   *
73   * If many entries are to be made into a <code>Hashtable</code>, 
74   * creating it with a sufficiently large capacity may allow the 
75   * entries to be inserted more efficiently than letting it perform 
76   * automatic rehashing as needed to grow the table. <p>
77   *
78   * This example creates a hashtable of numbers. It uses the names of 
79   * the numbers as keys:
80   * <p><blockquote><pre>
81   *     Hashtable numbers = new Hashtable();
82   *     numbers.put("one", new Integer(1));
83   *     numbers.put("two", new Integer(2));
84   *     numbers.put("three", new Integer(3));
85   * </pre></blockquote>
86   * <p>
87   * To retrieve a number, use the following code: 
88   * <p><blockquote><pre>
89   *     Integer n = (Integer)numbers.get("two");
90   *     if (n != null) {
91   *         System.out.println("two = " + n);
92   *     }
93   * </pre></blockquote>
94   * <p>
95   * As of the Java 2 platform v1.2, this class has been retrofitted to
96   * implement Map, so that it becomes a part of Java's collection framework.
97   * Unlike the new collection implementations, Hashtable is synchronized.<p>
98   *
99   * The Iterators returned by the iterator and listIterator methods
100  * of the Collections returned by all of Hashtable's "collection view methods"
101  * are <em>fail-fast</em>: if the Hashtable is structurally modified
102  * at any time after the Iterator is created, in any way except through the
103  * Iterator's own remove or add methods, the Iterator will throw a
104  * ConcurrentModificationException.  Thus, in the face of concurrent
105  * modification, the Iterator fails quickly and cleanly, rather than risking
106  * arbitrary, non-deterministic behavior at an undetermined time in the future.
107  * The Enumerations returned by Hashtable's keys and values methods are
108  * <em>not</em> fail-fast.
109  *
110  * <p>Note that the fail-fast behavior of an iterator cannot be guaranteed
111  * as it is, generally speaking, impossible to make any hard guarantees in the
112  * presence of unsynchronized concurrent modification.  Fail-fast iterators
113  * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
114  * Therefore, it would be wrong to write a program that depended on this
115  * exception for its correctness: <i>the fail-fast behavior of iterators
116  * should be used only to detect bugs.</i><p>
117  *
118  * This class is a member of the 
119  * <a href="{@docRoot}/../guide/collections/index.html">
120  * Java Collections Framework</a>.
121  *
122  * @author  Arthur van Hoff
123  * @author  Josh Bloch
124  * @version 1.82, 02/02/00
125  * @see     Object#equals(java.lang.Object)
126  * @see     Object#hashCode()
127  * @see     Hashtable#rehash()
128  * @see     Collection
129  * @see	    Map
130  * @see	    HashMap
131  * @see	    TreeMap
132  * @since JDK1.0
133  */
