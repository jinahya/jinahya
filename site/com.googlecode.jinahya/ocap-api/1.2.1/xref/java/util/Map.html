39  /** 
40   * An object that maps keys to values.  A map cannot contain duplicate keys;
41   * each key can map to at most one value.
42   *
43   * <p>This interface takes the place of the <tt>Dictionary</tt> class, which
44   * was a totally abstract class rather than an interface.
45   *
46   * <p>The <tt>Map</tt> interface provides three <i>collection views</i>, which
47   * allow a map's contents to be viewed as a set of keys, collection of values,
48   * or set of key-value mappings.  The <i>order</i> of a map is defined as
49   * the order in which the iterators on the map's collection views return their
50   * elements.  Some map implementations, like the <tt>TreeMap</tt> class, make
51   * specific guarantees as to their order; others, like the <tt>HashMap</tt>
52   * class, do not.
53   *
54   * <p>Note: great care must be exercised if mutable objects are used as map
55   * keys.  The behavior of a map is not specified if the value of an object is
56   * changed in a manner that affects equals comparisons while the object is a
57   * key in the map.  A special case of this prohibition is that it is not
58   * permissible for a map to contain itself as a key.  While it is permissible
59   * for a map to contain itself as a value, extreme caution is advised: the
60   * equals and hashCode methods are no longer well defined on a such a map.
61   *
62   * <p>All general-purpose map implementation classes should provide two
63   * "standard" constructors: a void (no arguments) constructor which creates an
64   * empty map, and a constructor with a single argument of type <tt>Map</tt>,
65   * which creates a new map with the same key-value mappings as its argument.
66   * In effect, the latter constructor allows the user to copy any map,
67   * producing an equivalent map of the desired class.  There is no way to
68   * enforce this recommendation (as interfaces cannot contain constructors) but
69   * all of the general-purpose map implementations in the SDK comply.
70   *
71   * <p>The "destructive" methods contained in this interface, that is, the
72   * methods that modify the map on which they operate, are specified to throw
73   * <tt>UnsupportedOperationException</tt> if this map does not support the
74   * operation.  If this is the case, these methods may, but are not required
75   * to, throw an <tt>UnsupportedOperationException</tt> if the invocation would
76   * have no effect on the map.  For example, invoking the {@link #putAll(Map)}
77   * method on an unmodifiable map may, but is not required to, throw the
78   * exception if the map whose mappings are to be "superimposed" is empty.
79   *
80   * <p>Some map implementations have restrictions on the keys and values they
81   * may contain.  For example, some implementations prohibit null keys and
82   * values, and some have restrictions on the types of their keys.  Attempting
83   * to insert an ineligible key or value throws an unchecked exception,
84   * typically <tt>NullPointerException</tt> or <tt>ClassCastException</tt>.
85   * Attempting to query the presence of an ineligible key or value may throw an
86   * exception, or it may simply return false; some implementations will exhibit
87   * the former behavior and some will exhibit the latter.  More generally,
88   * attempting an operation on an ineligible key or value whose completion
89   * would not result in the insertion of an ineligible element into the map may
90   * throw an exception or it may succeed, at the option of the implementation.
91   * Such exceptions are marked as "optional" in the specification for this
92   * interface.
93   *
94   * <p>This interface is a member of the 
95   * <a href="{@docRoot}/../guide/collections/index.html">
96   * Java Collections Framework</a>.
97   *
98   * @author  Josh Bloch
99   * @version 1.32, 02/02/00
100  * @see HashMap
101  * @see TreeMap
102  * @see Hashtable
103  * @see SortedMap
104  * @see Collection
105  * @see Set
106  * @since 1.2
107  */
