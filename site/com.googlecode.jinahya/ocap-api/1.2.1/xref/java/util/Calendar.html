45  /** 
46   * <code>Calendar</code> is an abstract base class for converting between
47   * a <code>Date</code> object and a set of integer fields such as
48   * <code>YEAR</code>, <code>MONTH</code>, <code>DAY</code>, <code>HOUR</code>,
49   * and so on. (A <code>Date</code> object represents a specific instant in
50   * time with millisecond precision. See
51   * {@link Date}
52   * for information about the <code>Date</code> class.)
53   *
54   * <p>
55   * Subclasses of <code>Calendar</code> interpret a <code>Date</code>
56   * according to the rules of a specific calendar system. The platform
57   * provides one concrete subclass of <code>Calendar</code>:
58   * <code>GregorianCalendar</code>. Future subclasses could represent
59   * the various types of lunar calendars in use in many parts of the world.
60   *
61   * <p>
62   * Like other locale-sensitive classes, <code>Calendar</code> provides a
63   * class method, <code>getInstance</code>, for getting a generally useful
64   * object of this type. <code>Calendar</code>'s <code>getInstance</code> method
65   * returns a <code>Calendar</code> object whose
66   * time fields have been initialized with the current date and time:
67   * <blockquote>
68   * <pre>
69   * Calendar rightNow = Calendar.getInstance();
70   * </pre>
71   * </blockquote>
72   *
73   * <p>A <code>Calendar</code> object can produce all the time field values
74   * needed to implement the date-time formatting for a particular language and
75   * calendar style (for example, Japanese-Gregorian, Japanese-Traditional).
76   * <code>Calendar</code> defines the range of values returned by certain fields,
77   * as well as their meaning.  For example, the first month of the year has value
78   * <code>MONTH</code> == <code>JANUARY</code> for all calendars.  Other values
79   * are defined by the concrete subclass, such as <code>ERA</code> and
80   * <code>YEAR</code>.  See individual field documentation and subclass
81   * documentation for details.
82   *
83   * <p>When a <code>Calendar</code> is <em>lenient</em>, it accepts a wider range
84   * of field values than it produces.  For example, a lenient
85   * <code>GregorianCalendar</code> interprets <code>MONTH</code> ==
86   * <code>JANUARY</code>, <code>DAY_OF_MONTH</code> == 32 as February 1.  A
87   * non-lenient <code>GregorianCalendar</code> throws an exception when given
88   * out-of-range field settings.  When calendars recompute field values for
89   * return by <code>get()</code>, they normalize them.  For example, a
90   * <code>GregorianCalendar</code> always produces <code>DAY_OF_MONTH</code>
91   * values between 1 and the length of the month.
92   *
93   * <p><code>Calendar</code> defines a locale-specific seven day week using two
94   * parameters: the first day of the week and the minimal days in first week
95   * (from 1 to 7).  These numbers are taken from the locale resource data when a
96   * <code>Calendar</code> is constructed.  They may also be specified explicitly
97   * through the API.
98   *
99   * <p>When setting or getting the <code>WEEK_OF_MONTH</code> or
100  * <code>WEEK_OF_YEAR</code> fields, <code>Calendar</code> must determine the
101  * first week of the month or year as a reference point.  The first week of a
102  * month or year is defined as the earliest seven day period beginning on
103  * <code>getFirstDayOfWeek()</code> and containing at least
104  * <code>getMinimalDaysInFirstWeek()</code> days of that month or year.  Weeks
105  * numbered ..., -1, 0 precede the first week; weeks numbered 2, 3,... follow
106  * it.  Note that the normalized numbering returned by <code>get()</code> may be
107  * different.  For example, a specific <code>Calendar</code> subclass may
108  * designate the week before week 1 of a year as week <em>n</em> of the previous
109  * year.
110  *
111  * <p> When computing a <code>Date</code> from time fields, two special
112  * circumstances may arise: there may be insufficient information to compute the
113  * <code>Date</code> (such as only year and month but no day in the month), or
114  * there may be inconsistent information (such as "Tuesday, July 15, 1996" --
115  * July 15, 1996 is actually a Monday).
116  *
117  * <p>
118  * <strong>Insufficient information.</strong> The calendar will use default
119  * information to specify the missing fields. This may vary by calendar; for
120  * the Gregorian calendar, the default for a field is the same as that of the
121  * start of the epoch: i.e., YEAR = 1970, MONTH = JANUARY, DATE = 1, etc.
122  *
123  * <p>
124  * <strong>Inconsistent information.</strong> If fields conflict, the calendar
125  * will give preference to fields set more recently. For example, when
126  * determining the day, the calendar will look for one of the following
127  * combinations of fields.  The most recent combination, as determined by the
128  * most recently set single field, will be used.
129  *
130  * <blockquote>
131  * <pre>
132  * MONTH + DAY_OF_MONTH
133  * MONTH + WEEK_OF_MONTH + DAY_OF_WEEK
134  * MONTH + DAY_OF_WEEK_IN_MONTH + DAY_OF_WEEK
135  * DAY_OF_YEAR
136  * DAY_OF_WEEK + WEEK_OF_YEAR
137  * </pre>
138  * </blockquote>
139  *
140  * For the time of day:
141  *
142  * <blockquote>
143  * <pre>
144  * HOUR_OF_DAY
145  * AM_PM + HOUR
146  * </pre>
147  * </blockquote>
148  *
149  * <p>
150  * <strong>Note:</strong> for some non-Gregorian calendars, different
151  * fields may be necessary for complete disambiguation. For example, a full
152  * specification of the historical Arabic astronomical calendar requires year,
153  * month, day-of-month <em>and</em> day-of-week in some cases.
154  *
155  * <p>
156  * <strong>Note:</strong> There are certain possible ambiguities in
157  * interpretation of certain singular times, which are resolved in the
158  * following ways:
159  * <ol>
160  *     <li> 23:59 is the last minute of the day and 00:00 is the first minute of the
161  *          next day. Thus, 23:59 on Dec 31, 1999 &lt; 00:00 on Jan 1, 2000 &lt; 00:01 on
162  *          Jan 1, 2000.
163  *
164  *     <li> Although historically not precise, midnight also belongs to "am",
165  *          and noon belongs to "pm", so on the same day,
166  *          12:00 am (midnight) &lt; 12:01 am, and 12:00 pm (noon) &lt; 12:01 pm
167  * </ol>
168  *
169  * <p>
170  * The date or time format strings are not part of the definition of a
171  * calendar, as those must be modifiable or overridable by the user at
172  * runtime. Use {@link DateFormat}
173  * to format dates.
174  *
175  * <p><strong>Field manipulation methods</strong></p>
176  * 
177  * <p><code>Calendar</code> fields can be changed using three methods:
178  * <code>set()</code>, <code>add()</code>, and <code>roll()</code>.</p>
179  * 
180  * <p><strong><code>set(f, value)</code></strong> changes field
181  * <code>f</code> to <code>value</code>.  In addition, it sets an
182  * internal member variable to indicate that field <code>f</code> has
183  * been changed. Although field <code>f</code> is changed immediately,
184  * the calendar's milliseconds is not recomputed until the next call to
185  * <code>get()</code>, <code>getTime()</code>, or
186  * <code>getTimeInMillis()</code> is made. Thus, multiple calls to
187  * <code>set()</code> do not trigger multiple, unnecessary
188  * computations. As a result of changing a field using
189  * <code>set()</code>, other fields may also change, depending on the
190  * field, the field value, and the calendar system. In addition,
191  * <code>get(f)</code> will not necessarily return <code>value</code>
192  * after the fields have been recomputed. The specifics are determined by
193  * the concrete calendar class.</p>
194  * 
195  * <p><em>Example</em>: Consider a <code>GregorianCalendar</code>
196  * originally set to August 31, 1999. Calling <code>set(Calendar.MONTH,
197  * Calendar.SEPTEMBER)</code> sets the calendar to September 31,
198  * 1999. This is a temporary internal representation that resolves to
199  * October 1, 1999 if <code>getTime()</code>is then called. However, a
200  * call to <code>set(Calendar.DAY_OF_MONTH, 30)</code> before the call to
201  * <code>getTime()</code> sets the calendar to September 30, 1999, since
202  * no recomputation occurs after <code>set()</code> itself.</p>
203  * 
204  * <p><strong><code>add(f, delta)</code></strong> adds <code>delta</code>
205  * to field <code>f</code>.  This is equivalent to calling <code>set(f,
206  * get(f) + delta)</code> with two adjustments:</p>
207  * 
208  * <blockquote>
209  *   <p><strong>Add rule 1</strong>. The value of field <code>f</code>
210  *   after the call minus the value of field <code>f</code> before the
211  *   call is <code>delta</code>, modulo any overflow that has occurred in
212  *   field <code>f</code>. Overflow occurs when a field value exceeds its
213  *   range and, as a result, the next larger field is incremented or
214  *   decremented and the field value is adjusted back into its range.</p>
215  * 
216  *   <p><strong>Add rule 2</strong>. If a smaller field is expected to be
217  *   invariant, but &nbsp; it is impossible for it to be equal to its
218  *   prior value because of changes in its minimum or maximum after field
219  *   <code>f</code> is changed, then its value is adjusted to be as close
220  *   as possible to its expected value. A smaller field represents a
221  *   smaller unit of time. <code>HOUR</code> is a smaller field than
222  *   <code>DAY_OF_MONTH</code>. No adjustment is made to smaller fields
223  *   that are not expected to be invariant. The calendar system
224  *   determines what fields are expected to be invariant.</p>
225  * </blockquote>
226  * 
227  * <p>In addition, unlike <code>set()</code>, <code>add()</code> forces
228  * an immediate recomputation of the calendar's milliseconds and all
229  * fields.</p>
230  * 
231  * <p><em>Example</em>: Consider a <code>GregorianCalendar</code>
232  * originally set to August 31, 1999. Calling <code>add(Calendar.MONTH,
233  * 13)</code> sets the calendar to September 30, 2000. <strong>Add rule
234  * 1</strong> sets the <code>MONTH</code> field to September, since
235  * adding 13 months to August gives September of the next year. Since
236  * <code>DAY_OF_MONTH</code> cannot be 31 in September in a
237  * <code>GregorianCalendar</code>, <strong>add rule 2</strong> sets the
238  * <code>DAY_OF_MONTH</code> to 30, the closest possible value. Although
239  * it is a smaller field, <code>DAY_OF_WEEK</code> is not adjusted by
240  * rule 2, since it is expected to change when the month changes in a
241  * <code>GregorianCalendar</code>.</p>
242  * 
243  * <p><strong><code>roll(f, delta)</code></strong> adds
244  * <code>delta</code> to field <code>f</code> without changing larger
245  * fields. This is equivalent to calling <code>add(f, delta)</code> with
246  * the following adjustment:</p>
247  * 
248  * <blockquote>
249  *   <p><strong>Roll rule</strong>. Larger fields are unchanged after the
250  *   call. A larger field represents a larger unit of
251  *   time. <code>DAY_OF_MONTH</code> is a larger field than
252  *   <code>HOUR</code>.</p>
253  * </blockquote>
254  * 
255  * <p><em>Example</em>: See {@link java.util.GregorianCalendar#roll(int, int)}.
256  * 
257  * <p><strong>Usage model</strong>. To motivate the behavior of
258  * <code>add()</code> and <code>roll()</code>, consider a user interface
259  * component with increment and decrement buttons for the month, day, and
260  * year, and an underlying <code>GregorianCalendar</code>. If the
261  * interface reads January 31, 1999 and the user presses the month
262  * increment button, what should it read? If the underlying
263  * implementation uses <code>set()</code>, it might read March 3, 1999. A
264  * better result would be February 28, 1999. Furthermore, if the user
265  * presses the month increment button again, it should read March 31,
266  * 1999, not March 28, 1999. By saving the original date and using either
267  * <code>add()</code> or <code>roll()</code>, depending on whether larger
268  * fields should be affected, the user interface can behave as most users
269  * will intuitively expect.</p>
270  *
