37  package java.util;

39  /** 
40   * <p>Hash table and linked list implementation of the <tt>Set</tt> interface,
41   * with predictable iteration order.  This implementation differs from
42   * <tt>HashSet</tt> in that it maintains a doubly-linked list running through
43   * all of its entries.  This linked list defines the iteration ordering,
44   * which is the order in which elements were inserted into the set
45   * (<i>insertion-order</i>).  Note that insertion order is <i>not</i> affected
46   * if an element is <i>re-inserted</i> into the set.  (An element <tt>e</tt>
47   * is reinserted into a set <tt>s</tt> if <tt>s.add(e)</tt> is invoked when
48   * <tt>s.contains(e)</tt> would return <tt>true</tt> immediately prior to
49   * the invocation.)
50   *
51   * <p>This implementation spares its clients from the unspecified, generally
52   * chaotic ordering provided by {@link HashSet}, without incurring the
53   * increased cost associated with {@link TreeSet}.  It can be used to
54   * produce a copy of a set that has the same order as the original, regardless
55   * of the original set's implementation:
56   * <pre>
57   *     void foo(Set m) {
58   *         Set copy = new LinkedHashSet(m);
59   *         ...
60   *     }
61   * </pre>
62   * This technique is particularly useful if a module takes a set on input,
63   * copies it, and later returns results whose order is determined by that of
64   * the copy.  (Clients generally appreciate having things returned in the same
65   * order they were presented.)
66   *
67   * <p>This class provides all of the optional <tt>Set</tt> operations, and
68   * permits null elements.  Like <tt>HashSet</tt>, it provides constant-time
69   * performance for the basic operations (<tt>add</tt>, <tt>contains</tt> and
70   * <tt>remove</tt>), assuming the the hash function disperses elements
71   * properly among the buckets.  Performance is likely to be just slightly
72   * below that of <tt>HashSet</tt>, due to the added expense of maintaining the
73   * linked list, with one exception: Iteration over a <tt>LinkedHashSet</tt>
74   * requires time proportional to the <i>size</i> of the set, regardless of
75   * its capacity.  Iteration over a <tt>HashSet</tt> is likely to be more
76   * expensive, requiring time proportional to its <i>capacity</i>.
77   * 
78   * <p>A linked hash set has two parameters that affect its performance:
79   * <i>initial capacity</i> and <i>load factor</i>.  They are defined precisely
80   * as for <tt>HashSet</tt>.  Note, however, that the penalty for choosing an
81   * excessively high value for initial capacity is less severe for this class
82   * than for <tt>HashSet</tt>, as iteration times for this class are unaffected
83   * by capacity.
84   * 
85   * <p><strong>Note that this implementation is not synchronized.</strong> If
86   * multiple threads access a linked hash set concurrently, and at least one of
87   * the threads modifies the set, it <em>must</em> be synchronized externally.
88   * This is typically accomplished by synchronizing on some object that
89   * naturally encapsulates the set.  If no such object exists, the set should
90   * be "wrapped" using the <tt>Collections.synchronizedSet</tt>method.  This is
91   * best done at creation time, to prevent accidental unsynchronized access:
92   * <pre>
93   *     Set s = Collections.synchronizedSet(new LinkedHashSet(...));
94   * </pre>
95   *
96   * <p>The iterators returned by the this class's <tt>iterator</tt> method are
97   * <em>fail-fast</em>: if the set is modified at any time after the iterator
98   * is created, in any way except through the iterator's own remove method, the
99   * iterator will throw a <tt>ConcurrentModificationException</tt>.  Thus, in
100  * the face of concurrent modification, the Iterator fails quickly and
101  * cleanly, rather than risking arbitrary, non-deterministic behavior at an
102  * undetermined time in the future.
103  *
104  * <p>Note that the fail-fast behavior of an iterator cannot be guaranteed
105  * as it is, generally speaking, impossible to make any hard guarantees in the
106  * presence of unsynchronized concurrent modification.  Fail-fast iterators
107  * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
108  * Therefore, it would be wrong to write a program that depended on this
109  * exception for its correctness:   <i>the fail-fast behavior of iterators
110  * should be used only to detect bugs.</i>
111  *
112  * <p>This class is a member of the 
113  * <a href="{@docRoot}/../guide/collections/index.html">
114  * Java Collections Framework</a>.
115  *
116  * @author  Josh Bloch
117  * @version 1.8 01/20/03
118  * @see     Object#hashCode()
119  * @see     Collection
120  * @see     Set
121  * @see     HashSet
122  * @see     TreeSet
123  * @see     Hashtable
124  * @since   1.4
125  */
126 public class LinkedHashSet extends HashSet
127     implements Set, Cloneable, java.io.Serializable
