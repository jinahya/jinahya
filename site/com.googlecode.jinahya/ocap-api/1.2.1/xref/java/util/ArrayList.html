39  /** 
40   * Resizable-array implementation of the <tt>List</tt> interface.  Implements
41   * all optional list operations, and permits all elements, including
42   * <tt>null</tt>.  In addition to implementing the <tt>List</tt> interface,
43   * this class provides methods to manipulate the size of the array that is
44   * used internally to store the list.  (This class is roughly equivalent to
45   * <tt>Vector</tt>, except that it is unsynchronized.)<p>
46   *
47   * The <tt>size</tt>, <tt>isEmpty</tt>, <tt>get</tt>, <tt>set</tt>,
48   * <tt>iterator</tt>, and <tt>listIterator</tt> operations run in constant
49   * time.  The <tt>add</tt> operation runs in <i>amortized constant time</i>,
50   * that is, adding n elements requires O(n) time.  All of the other operations
51   * run in linear time (roughly speaking).  The constant factor is low compared
52   * to that for the <tt>LinkedList</tt> implementation.<p>
53   *
54   * Each <tt>ArrayList</tt> instance has a <i>capacity</i>.  The capacity is
55   * the size of the array used to store the elements in the list.  It is always
56   * at least as large as the list size.  As elements are added to an ArrayList,
57   * its capacity grows automatically.  The details of the growth policy are not
58   * specified beyond the fact that adding an element has constant amortized
59   * time cost.<p> 
60   *
61   * An application can increase the capacity of an <tt>ArrayList</tt> instance
62   * before adding a large number of elements using the <tt>ensureCapacity</tt>
63   * operation.  This may reduce the amount of incremental reallocation.<p>
64   *
65   * <strong>Note that this implementation is not synchronized.</strong> If
66   * multiple threads access an <tt>ArrayList</tt> instance concurrently, and at
67   * least one of the threads modifies the list structurally, it <i>must</i> be
68   * synchronized externally.  (A structural modification is any operation that
69   * adds or deletes one or more elements, or explicitly resizes the backing
70   * array; merely setting the value of an element is not a structural
71   * modification.)  This is typically accomplished by synchronizing on some
72   * object that naturally encapsulates the list.  If no such object exists, the
73   * list should be "wrapped" using the <tt>Collections.synchronizedList</tt>
74   * method.  This is best done at creation time, to prevent accidental
75   * unsynchronized access to the list:
76   * <pre>
77   *	List list = Collections.synchronizedList(new ArrayList(...));
78   * </pre><p>
79   *
80   * The iterators returned by this class's <tt>iterator</tt> and
81   * <tt>listIterator</tt> methods are <i>fail-fast</i>: if list is structurally
82   * modified at any time after the iterator is created, in any way except
83   * through the iterator's own remove or add methods, the iterator will throw a
84   * ConcurrentModificationException.  Thus, in the face of concurrent
85   * modification, the iterator fails quickly and cleanly, rather than risking
86   * arbitrary, non-deterministic behavior at an undetermined time in the
87   * future.<p>
88   *
89   * Note that the fail-fast behavior of an iterator cannot be guaranteed
90   * as it is, generally speaking, impossible to make any hard guarantees in the
91   * presence of unsynchronized concurrent modification.  Fail-fast iterators
92   * throw <tt>ConcurrentModificationException</tt> on a best-effort basis. 
93   * Therefore, it would be wrong to write a program that depended on this
94   * exception for its correctness: <i>the fail-fast behavior of iterators
95   * should be used only to detect bugs.</i><p>
