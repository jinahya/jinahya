48  /** 
49   * Represents a Uniform Resource Identifier (URI) reference.
50   *
51   * <p> An instance of this class represents a URI reference as defined by <a
52   * href="http://www.ietf.org/rfc/rfc2396.txt""><i>RFC&nbsp;2396: Uniform
53   * Resource Identifiers (URI): Generic Syntax</i></a>, amended by <a
54   * href="http://www.ietf.org/rfc/rfc2732.txt"><i>RFC&nbsp;2732: Format for
55   * Literal IPv6 Addresses in URLs</i></a> and with the minor deviations noted
56   * below.  This class provides constructors for creating URI instances from
57   * their components or by parsing their string forms, methods for accessing the
58   * various components of an instance, and methods for normalizing, resolving,
59   * and relativizing URI instances.  Instances of this class are immutable.
60   *
61   *
62   * <h4> URI syntax and components </h4>
63   *
64   * At the highest level a URI reference (hereinafter simply "URI") in string
65   * form has the syntax
66   *
67   * <blockquote>
68   * [<i>scheme</i><tt><b>:</b></tt><i></i>]<i>scheme-specific-part</i>[<tt><b>#</b></tt><i>fragment</i>]
69   * </blockquote>
70   *
71   * where square brackets [...] delineate optional components and the characters
72   * <tt><b>:</b></tt> and <tt><b>#</b></tt> stand for themselves.
73   *
74   * <p> An <i>absolute</i> URI specifies a scheme; a URI that is not absolute is
75   * said to be <i>relative</i>.  URIs are also classified according to whether
76   * they are <i>opaque</i> or <i>hierarchical</i>.
77   *
78   * <p> An <i>opaque</i> URI is an absolute URI whose scheme-specific part does
79   * not begin with a slash character (<tt>'/'</tt>).  Opaque URIs are not
80   * subject to further parsing.  Some examples of opaque URIs are:
81   *
82   * <blockquote><table cellpadding=0 cellspacing=0 summary="layout">
83   * <tr><td><tt>mailto:java-net@java.sun.com</tt><td></tr>
84   * <tr><td><tt>news:comp.lang.java</tt><td></tr>
85   * <tr><td><tt>urn:isbn:096139210x</tt></td></tr>
86   * </table></blockquote>
87   *
88   * <p> A <i>hierarchical</i> URI is either an absolute URI whose
89   * scheme-specific part begins with a slash character, or a relative URI, that
90   * is, a URI that does not specify a scheme.  Some examples of hierarchical
91   * URIs are:
92   *
93   * <blockquote>
94   * <tt>http://java.sun.com/j2se/1.3/</tt><br>
95   * <tt>docs/guide/collections/designfaq.html#28</tt></br>
96   * <tt>../../../demo/jfc/SwingSet2/src/SwingSet2.java</tt></br>
97   * <tt>file:///~/calendar</tt>
98   * </blockquote>
99   *
100  * <p> A hierarchical URI is subject to further parsing according to the syntax
101  *
102  * <blockquote>
103  * [<i>scheme</i><tt><b>:</b></tt>][<tt><b>//</b></tt><i>authority</i>][<i>path</i>][<tt><b>?</b></tt><i>query</i>][<tt><b>#</b></tt><i>fragment</i>]
104  * </blockquote>
105  *
106  * where the characters <tt><b>:</b></tt>, <tt><b>/</b></tt>,
107  * <tt><b>?</b></tt>, and <tt><b>#</b></tt> stand for themselves.  The
108  * scheme-specific part of a hierarchical URI consists of the characters
109  * between the scheme and fragment components.
110  *
111  * <p> The authority component of a hierarchical URI is, if specified, either
112  * <i>server-based</i> or <i>registry-based</i>.  A server-based authority
113  * parses according to the familiar syntax
114  *
115  * <blockquote>
116  * [<i>user-info</i><tt><b>@</b></tt>]<i>host</i>[<tt><b>:</b></tt><i>port</i>]
117  * </blockquote>
118  *
119  * where the characters <tt><b>@</b></tt> and <tt><b>:</b></tt> stand for
120  * themselves.  Nearly all URI schemes currently in use are server-based.  An
121  * authority component that does not parse in this way is considered to be
122  * registry-based.
123  *
124  * <p> The path component of a hierarchical URI is itself said to be absolute
125  * if it begins with a slash character (<tt>'/'</tt>); otherwise it is
126  * relative.  The path of a hierarchical URI that is either absolute or
127  * specifies an authority is always absolute.
128  *
129  * <p> All told, then, a URI instance has the following nine components:
130  *
131  * <blockquote><table summary="Describes the components of a URI:scheme,scheme-specific-part,authority,user-info,host,port,path,query,fragment">
132  * <tr><th><i>Component</i></th><th><i>Type</i></th></tr>
133  * <tr><td>scheme</td><td><tt>String</tt></td></tr>
134  * <tr><td>scheme-specific-part&nbsp;&nbsp;&nbsp;&nbsp;</td><td><tt>String</tt></td></tr>
135  * <tr><td>authority</td><td><tt>String</tt></td></tr>
136  * <tr><td>user-info</td><td><tt>String</tt></td></tr>
137  * <tr><td>host</td><td><tt>String</tt></td></tr>
138  * <tr><td>port</td><td><tt>int</tt></td></tr>
139  * <tr><td>path</td><td><tt>String</tt></td></tr>
140  * <tr><td>query</td><td><tt>String</tt></td></tr>
141  * <tr><td>fragment</td><td><tt>String</tt></td></tr>
142  * </table></blockquote>
143  *
144  * In a given instance any particular component is either <i>undefined</i> or
145  * <i>defined</i> with a distinct value.  Undefined string components are
146  * represented by <tt>null</tt>, while undefined integer components are
147  * represented by <tt>-1</tt>.  A string component may be defined to have the
148  * empty string as its value; this is not equivalent to that component being
149  * undefined.
150  *
151  * <p> Whether a particular component is or is not defined in an instance
152  * depends upon the type of the URI being represented.  An absolute URI has a
153  * scheme component.  An opaque URI has a scheme, a scheme-specific part, and
154  * possibly a fragment, but has no other components.  A hierarchical URI always
155  * has a path (though it may be empty) and a scheme-specific-part (which at
156  * least contains the path), and may have any of the other components.  If the
157  * authority component is present and is server-based then the host component
158  * will be defined and the user-information and port components may be defined.
159  *
160  *
161  * <h4> Operations on URI instances </h4>
162  *
163  * The key operations supported by this class are those of
164  * <i>normalization</i>, <i>resolution</i>, and <i>relativization</i>.
165  *
166  * <p> <i>Normalization</i> is the process of removing unnecessary <tt>"."</tt>
167  * and <tt>".."</tt> segments from the path component of a hierarchical URI.
168  * Each <tt>"."</tt> segment is simply removed.  A <tt>".."</tt> segment is
169  * removed only if it is preceded by a non-<tt>".."</tt> segment.
170  * Normalization has no effect upon opaque URIs.
171  *
172  * <p> <i>Resolution</i> is the process of resolving one URI against another,
173  * <i>base</i> URI.  The resulting URI is constructed from components of both
174  * URIs in the manner specified by RFC&nbsp;2396, taking components from the
175  * base URI for those not specified in the original.  For hierarchical URIs,
176  * the path of the original is resolved against the path of the base and then
177  * normalized.  The result, for example, of resolving
178  *
179  * <blockquote>
180  * <tt>docs/guide/collections/designfaq.html#28&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt>(1)
181  * </blockquote>
182  *
183  * against the base URI <tt>http://java.sun.com/j2se/1.3/</tt> is the result
184  * URI
185  *
186  * <blockquote>
187  * <tt>http://java.sun.com/j2se/1.3/docs/guide/collections/designfaq.html#28</tt>
188  * </blockquote>
189  *
190  * Resolving the relative URI
191  *
192  * <blockquote>
193  * <tt>../../../demo/jfc/SwingSet2/src/SwingSet2.java&nbsp;&nbsp;&nbsp;&nbsp;</tt>(2)
194  * </blockquote>
195  *
196  * against this result yields, in turn,
197  *
198  * <blockquote>
199  * <tt>http://java.sun.com/j2se/1.3/demo/jfc/SwingSet2/src/SwingSet2.java</tt>
200  * </blockquote>
201  *
202  * Resolution of both absolute and relative URIs, and of both absolute and
203  * relative paths in the case of hierarchical URIs, is supported.  Resolving
204  * the URI <tt>file:///~calendar</tt> against any other URI simply yields the
205  * original URI, since it is absolute.  Resolving the relative URI (2) above
206  * against the relative base URI (1) yields the normalized, but still relative,
207  * URI
208  *
209  * <blockquote>
210  * <tt>demo/jfc/SwingSet2/src/SwingSet2.java</tt>
211  * </blockquote>
212  *
213  * <p> <i>Relativization</i>, finally, is the inverse of resolution: For any
214  * two normalized URIs <i>u</i> and&nbsp;<i>v</i>,
215  *
216  * <blockquote>
217  *   <i>u</i><tt>.relativize(</tt><i>u</i><tt>.resolve(</tt><i>v</i><tt>)).equals(</tt><i>v</i><tt>)</tt>&nbsp;&nbsp;and<br>
218  *   <i>u</i><tt>.resolve(</tt><i>u</i><tt>.relativize(</tt><i>v</i><tt>)).equals(</tt><i>v</i><tt>)</tt>&nbsp;&nbsp;.<br>
219  * </blockquote>
220  *
221  * This operation is often useful when constructing a document containing URIs
222  * that must be made relative to the base URI of the document wherever
223  * possible.  For example, relativizing the URI
224  *
225  * <blockquote>
226  * <tt>http://java.sun.com/j2se/1.3/docs/guide/index.html</tt>
227  * </blockquote>
228  *
229  * against the base URI
230  *
231  * <blockquote>
232  * <tt>http://java.sun.com/j2se/1.3</tt>
233  * </blockquote>
234  *
235  * yields the relative URI <tt>docs/guide/index.html</tt>.
236  *
237  *
238  * <h4> Character categories </h4>
239  *
240  * RFC&nbsp;2396 specifies precisely which characters are permitted in the
241  * various components of a URI reference.  The following categories, most of
242  * which are taken from that specification, are used below to describe these
243  * constraints:
244  *
245  * <blockquote><table cellspacing=2 summary="Describes categories alpha,digit,alphanum,unreserved,punct,reserved,escaped,and other">
246  *   <tr><th valign=top><i>alpha</i></th>
247  *       <td>The US-ASCII alphabetic characters,
248  * 	  <tt>'A'</tt>&nbsp;through&nbsp;<tt>'Z'</tt>
249  * 	  and <tt>'a'</tt>&nbsp;through&nbsp;<tt>'z'</tt></td></tr>
250  *   <tr><th valign=top><i>digit</i></th>
251  *       <td>The US-ASCII decimal digit characters,
252  *       <tt>'0'</tt>&nbsp;through&nbsp;<tt>'9'</tt></td></tr>
253  *   <tr><th valign=top><i>alphanum</i></th>
254  *       <td>All <i>alpha</i> and <i>digit</i> characters</td></tr>
255  *   <tr><th valign=top><i>unreserved</i>&nbsp;&nbsp;&nbsp;&nbsp;</th>
256  *       <td>All <i>alphanum</i> characters together with those in the string
257  * 	  <tt>"_-!.~'()*"</tt></td></tr>
258  *   <tr><th valign=top><i>punct</i></th>
259  *       <td>The characters in the string <tt>",;:$&+="</tt></td></tr>
260  *   <tr><th valign=top><i>reserved</i></th>
261  *       <td>All <i>punct</i> characters together with those in the string
262  * 	  <tt>"?/[]@"</tt></td></tr>
263  *   <tr><th valign=top><i>escaped</i></th>
264  *       <td>Escaped octets, that is, triplets consisting of the percent
265  *           character (<tt>'%'</tt>) followed by two hexadecimal digits
266  *           (<tt>'0'</tt>-<tt>'9'</tt>, <tt>'A'</tt>-<tt>'F'</tt>, and
267  *           <tt>'a'</tt>-<tt>'f'</tt>)</td></tr>
268  *   <tr><th valign=top><i>other</i></th>
269  *       <td>The Unicode characters that are not in the US-ASCII character set,
270  *           are not control characters (according to the {@link
271  *           java.lang.Character#isISOControl(char) Character.isISOControl}
272  * 	     method), and are not space characters (according to the {@link
273  * 	     java.lang.Character#isSpaceChar(char) Character.isSpaceChar}
274  * 	     method)&nbsp;&nbsp;(<b><i>Deviation from RFC 2396</b>, which is
275  * 	     limited to US-ASCII)</td></tr>
276  * </table></blockquote>
277  *
278  * <p><a name="legal-chars"> The set of all legal URI characters consists of
279  * the <i>unreserved</i>, <i>reserved</i>, <i>escaped</i>, and <i>other</i>
280  * characters.
281  *
282  *
283  * <h4> Escaped octets, quotation, encoding, and decoding </h4>
284  *
285  * RFC 2396 allows escaped octets to appear in the user-info, path, query, and
286  * fragment components.  Escaping serves two purposes in URIs:
287  *
288  * <ul>
289  *
290  *   <li><p> To <i>encode</i> non-US-ASCII characters when a URI is required to
291  *   conform strictly to RFC&nbsp;2396 by not containing any <i>other</i>
292  *   characters.  </p></li>
293  *
294  *   <li><p> To <i>quote</i> characters that are otherwise illegal in a
295  *   component.  The user-info, path, query, and fragment components differ
296  *   slightly in terms of which characters are considered legal and illegal.
297  *   </p></li>
298  *
299  * </ul>
300  *
301  * These purposes are served in this class by three related operations:
302  *
303  * <ul>
304  *
305  *   <li><p><a name="encode"> A character is <i>encoded</i> by replacing it
306  *   with the sequence of escaped octets that represent that character in the
307  *   UTF-8 character set.  The Euro currency symbol (<tt>'&#92;u20AC'</tt>),
308  *   for example, is encoded as <tt>"%E2%82%AC"</tt>.  <i>(<b>Deviation from
309  *   RFC&nbsp;2396</b>, which does not specify any particular character
310  *   set.)</i> </li></p>
311  *
312  *   <li><p><a name="quote"> An illegal character is <i>quoted</i> simply by
313  *   encoding it.  The space character, for example, is quoted by replacing it
314  *   with <tt>"%20"</tt>.  UTF-8 contains US-ASCII, hence for US-ASCII
315  *   characters this transformation has exactly the effect required by
316  *   RFC&nbsp;2396.
317  *
318  *   <li><p><a name="decode"> A sequence of escaped octets is <i>decoded</i> by
319  *   replacing it with the sequence of characters that it represents in the
320  *   UTF-8 character set.  UTF-8 contains US-ASCII, hence decoding has the
321  *   effect of de-quoting any quoted US-ASCII characters as well as that of
322  *   decoding any encoded non-US-ASCII characters.  If a <a
323  *   href="http://java.sun.com/j2se/1.4.2/docs/api/java/nio/charset/CharsetDecoder.html#ce">decoding error</a> occurs
324  *   when decoding the escaped octets then the erroneous octets are replaced by
325  *   <tt>'&#92;uFFFD'</tt>, the Unicode replacement character.  </p></li>
326  *
327  * </ul>
328  *
329  * These operations are exposed in the constructors and methods of this class
330  * as follows:
331  *
332  * <ul>
333  *
334  *   <li><p> The {@link #URI(java.lang.String) </code>single-argument
335  *   constructor<code>} requires any illegal characters in its argument to be
336  *   quoted and preserves any escaped octets and <i>other</i> characters that
337  *   are present.  </p></li>
338  *
339  *   <li><p> The {@link
340  *   #URI(java.lang.String,java.lang.String,java.lang.String,int,java.lang.String,java.lang.String,java.lang.String)
341  *   </code>multi-argument constructors<code>} quote illegal characters as
342  *   required by the components in which they appear.  The percent character
343  *   (<tt>'%'</tt>) is always quoted by these constructors.  Any <i>other</i>
344  *   characters are preserved.  </p></li>
345  *
346  *   <li><p> The {@link #getRawUserInfo() getRawUserInfo}, {@link #getRawPath()
347  *   getRawPath}, {@link #getRawQuery() getRawQuery}, {@link #getRawFragment()
348  *   getRawFragment}, {@link #getRawAuthority() getRawAuthority}, and {@link
349  *   #getRawSchemeSpecificPart() getRawSchemeSpecificPart} methods return the
350  *   values of their corresponding components in raw form, without interpreting
351  *   any escaped octets.  The strings returned by these methods may contain
352  *   both escaped octets and <i>other</i> characters, and will not contain any
353  *   illegal characters.  </p></li>
354  *
355  *   <li><p> The {@link #getUserInfo() getUserInfo}, {@link #getPath()
356  *   getPath}, {@link #getQuery() getQuery}, {@link #getFragment()
357  *   getFragment}, {@link #getAuthority() getAuthority}, and {@link
358  *   #getSchemeSpecificPart() getSchemeSpecificPart} methods decode any escaped
359  *   octets in their corresponding components.  The strings returned by these
360  *   methods may contain both <i>other</i> characters and illegal characters,
361  *   and will not contain any escaped octets.  </p></li>
362  *
363  *   <li><p> The {@link #toString() toString} method returns a URI string with
364  *   all necessary quotation but which may contain <i>other</i> characters.
365  *   </p></li>
366  *
367  *   <li><p> The {@link #toASCIIString() toASCIIString} method returns a fully
368  *   quoted and encoded URI string that does not contain any <i>other</i>
369  *   characters.  </p></li>
370  *
371  * </ul>

374  * <h4> Identities </h4>
375  *
376  * For any URI <i>u</i>, it is always the case that
377  *
378  * <blockquote>
379  * <tt>new URI(</tt><i>u</i><tt>.toString()).equals(</tt><i>u</i><tt>)</tt>&nbsp;.
380  * </blockquote>
381  *
382  * For any URI <i>u</i> that does not contain redundant syntax such as two
383  * slashes before an empty authority (as in <tt>file:///tmp/</tt>&nbsp;) or a
384  * colon following a host name but no port (as in
385  * <tt>http://java.sun.com:</tt>&nbsp;), and that does not encode characters
386  * except those that must be quoted, the following identities also hold:
387  *
388  * <blockquote>
389  * <tt>new URI(</tt><i>u</i><tt>.getScheme(),<br>
390  * &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt><i>u</i><tt>.getSchemeSpecificPart(),<br>
391  * &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt><i>u</i><tt>.getFragment())<br>
392  * .equals(</tt><i>u</i><tt>)</tt>
393  * </blockquote>
394  *
395  * in all cases,
396  *
397  * <blockquote>
398  * <tt>new URI(</tt><i>u</i><tt>.getScheme(),<br>
399  * &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt><i>u</i><tt>.getUserInfo(),&nbsp;</tt><i>u</i><tt>.getAuthority(),<br>
400  * &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt><i>u</i><tt>.getPath(),&nbsp;</tt><i>u</i><tt>.getQuery(),<br>
401  * &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt><i>u</i><tt>.getFragment())<br>
402  * .equals(</tt><i>u</i><tt>)</tt>
403  * </blockquote>
404  *
405  * if <i>u</i> is hierarchical, and
406  *
407  * <blockquote>
408  * <tt>new URI(</tt><i>u</i><tt>.getScheme(),<br>
409  * &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt><i>u</i><tt>.getUserInfo(),&nbsp;</tt><i>u</i><tt>.getHost(),&nbsp;</tt><i>u</i><tt>.getPort(),<br>
410  * &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt><i>u</i><tt>.getPath(),&nbsp;</tt><i>u</i><tt>.getQuery(),<br>
411  * &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</tt><i>u</i><tt>.getFragment())<br>
412  * .equals(</tt><i>u</i><tt>)</tt>
413  * </blockquote>
414  *
415  * if <i>u</i> is hierarchical and has either no authority or a server-based
416  * authority.

419  * <h4> URIs, URLs, and URNs </h4>
420  *
421  * A URI is a uniform resource <i>identifier</i> while a URL is a uniform
422  * resource <i>locator</i>.  Hence every URL is a URI, abstractly speaking, but
423  * not every URI is a URL.  This is because there is another subcategory of
424  * URIs, uniform resource <i>names</i> (URNs), which name resources but do not
425  * specify how to locate them.  The <tt>mailto</tt>, <tt>news</tt>, and
426  * <tt>isbn</tt> URIs shown above are examples of URNs.
427  *
428  * <p> The conceptual distinction between URIs and URLs is reflected in the
429  * differences between this class and the {@link URL} class.
430  *
431  * <p> An instance of this class represents a URI reference in the syntactic
432  * sense defined by RFC&nbsp;2396.  A URI may be either absolute or relative.
433  * A URI string is parsed according to the generic syntax without regard to the
434  * scheme, if any, that it specifies.  No lookup of the host, if any, is
435  * performed, and no scheme-dependent stream handler is constructed.  Equality,
436  * hashing, and comparison are defined strictly in terms of the character
437  * content of the instance.  In other words, a URI instance is little more than
438  * a structured string that supports the syntactic, scheme-independent
439  * operations of comparison, normalization, resolution, and relativization.
440  *
441  * <p> An instance of the {@link URL} class, by contrast, represents the
442  * syntactic components of a URL together with some of the information required
443  * to access the resource that it describes.  A URL must be absolute, that is,
444  * it must always specify a scheme.  A URL string is parsed according to its
445  * scheme.  A stream handler is always established for a URL, and in fact it is
446  * impossible to create a URL instance for a scheme for which no handler is
447  * available.  Equality and hashing depend upon both the scheme and the
448  * Internet address of the host, if any; comparison is not defined.  In other
449  * words, a URL is a structured string that supports the syntactic operation of
450  * resolution as well as the network I/O operations of looking up the host and
451  * opening a connection to the specified resource.

480      * Constructs a URI by parsing the given string.
481      *
482      * <p> This constructor parses the given string exactly as specified by the
483      * grammar in <a
484      * href="http://www.ietf.org/rfc/rfc2396.txt">RFC&nbsp;2396</a>,
485      * Appendix&nbsp;A, <b><i>except for the following deviations:</i></b> </p>
486      *
487      * <ul type=disc>
488      *
489      *   <li><p> An empty authority component is permitted as long as it is
490      *   followed by a non-empty path, a query component, or a fragment
491      *   component.  This allows the parsing of URIs such as
492      *   <tt>"file:///foo/bar"</tt>, which seems to be the intent of
493      *   RFC&nbsp;2396 although the grammar does not permit it.  If the
494      *   authority component is empty then the user-information, host, and port
495      *   components are undefined. </p></li>
496      *
497      *   <li><p> Empty relative paths are permitted; this seems to be the
498      *   intent of RFC&nbsp;2396 although the grammar does not permit it.  The
499      *   primary consequence of this deviation is that a standalone fragment
500      *   such as <tt>"#foo"</tt> parses as a relative URI with an empty path
501      *   and the given fragment, and can be usefully <a
502      *   href="#resolve-frag">resolved</a> against a base URI.
503      *
504      *   <li><p> IPv4 addresses in host components are parsed rigorously, as
505      *   specified by <a
506      *   href="http://www.ietf.org/rfc/rfc2732.txt">RFC&nbsp;2732</a>: Each
507      *   element of a dotted-quad address must contain no more than three
508      *   decimal digits.  Each element is further constrained to have a value
509      *   no greater than 255. </p></li>
510      *
511      *   <li> <p> Hostnames in host components that comprise only a single
512      *   domain label are permitted to start with an <i>alphanum</i> 
513      *   character. This seems to be the intent of <a
514      *   href="http://www.ietf.org/rfc/rfc2396.txt">RFC&nbsp;2396</a>
515      *   section&nbsp;3.2.2 although the grammar does not permit it. The
516      *   consequence of this deviation is that the authority component of a
517      *   hierarchical URI such as <tt>s://123</tt>, will parse as a server-based 
518      *   authority. </p></li>
519      *
520      *   <li><p> IPv6 addresses are permitted for the host component.  An IPv6
521      *   address must be enclosed in square brackets (<tt>'['</tt> and
522      *   <tt>']'</tt>) as specified by <a
523      *   href="http://www.ietf.org/rfc/rfc2732.txt">RFC&nbsp;2732</a>.  The
524      *   IPv6 address itself must parse according to <a
525      *   href="http://www.ietf.org/rfc/rfc2373.txt">RFC&nbsp;2373</a>.  IPv6
526      *   addresses are further constrained to describe no more than sixteen
527      *   bytes of address information, a constraint implicit in RFC&nbsp;2373
528      *   but not expressible in the grammar. </p></li>
529      *
530      *   <li><p> Characters in the <i>other</i> category are permitted wherever
531      *   RFC&nbsp;2396 permits <i>escaped</i> octets, that is, in the
532      *   user-information, path, query, and fragment components, as well as in
533      *   the authority component if the authority is registry-based.  This
534      *   allows URIs to contain Unicode characters beyond those in the US-ASCII
535      *   character set. </p></li>
536      *
537      * </ul>
538      *
539      * @param  str   The string to be parsed into a URI
540      *
541      * @throws  NullPointerException
542      *          If <tt>str</tt> is <tt>null</tt>
543      *
544      * @throws  URISyntaxException
545      *          If the given string violates RFC&nbsp;2396, as augmented
546      *          by the above deviations
547      */
548     public URI(String str) throws URISyntaxException { }

551      * Constructs a hierarchical URI from the given components.
552      *
553      * <p> If a scheme is given then the path, if also given, must either be
554      * empty or begin with a slash character (<tt>'/'</tt>).  Otherwise a
555      * component of the new URI may be left undefined by passing <tt>null</tt>
556      * for the corresponding parameter or, in the case of the <tt>port</tt>
557      * parameter, by passing <tt>-1</tt>.
558      *
559      * <p> This constructor first builds a URI string from the given components
560      * according to the rules specified in <a
561      * href="http://www.ietf.org/rfc/rfc2396.txt">RFC&nbsp;2396</a>,
562      * section&nbsp;5.2, step&nbsp;7: </p>
563      *
564      * <ol>
565      *
566      *   <li><p> Initially, the result string is empty. </p></li>
567      *
568      *   <li><p> If a scheme is given then it is appended to the result,
569      *   followed by a colon character (<tt>':'</tt>).  </p></li>
570      *
571      *   <li><p> If user information, a host, or a port are given then the
572      *   string <tt>"//"</tt> is appended.  </p></li>
573      *
574      *   <li><p> If user information is given then it is appended, followed by
575      *   a commercial-at character (<tt>'@'</tt>).  Any character not in the
576      *   <i>unreserved</i>, <i>punct</i>, <i>escaped</i>, or <i>other</i>
577      *   categories is <a href="#quote">quoted</a>.  </p></li>
578      *
579      *   <li><p> If a host is given then it is appended.  If the host is a
580      *   literal IPv6 address but is not enclosed in square brackets
581      *   (<tt>'['</tt> and <tt>']'</tt>) then the square brackets are added.
582      *   </p></li>
583      *
584      *   <li><p> If a port number is given then a colon character
585      *   (<tt>':'</tt>) is appended, followed by the port number in decimal.
586      *   </p></li>
587      *
588      *   <li><p> If a path is given then it is appended.  Any character not in
589      *   the <i>unreserved</i>, <i>punct</i>, <i>escaped</i>, or <i>other</i>
590      *   categories, and not equal to the slash character (<tt>'/'</tt>) or the
591      *   commercial-at character (<tt>'@'</tt>), is quoted.  </p></li>
592      *
593      *   <li><p> If a query is given then a question-mark character
594      *   (<tt>'?'</tt>) is appended, followed by the query.  Any character that
595      *   is not a <a href="#legal-chars">legal URI character</a> is quoted.
596      *   </p></li>
597      *
598      *   <li><p> Finally, if a fragment is given then a hash character
599      *   (<tt>'#'</tt>) is appended, followed by the fragment.  Any character
600      *   that is not a legal URI character is quoted.  </p></li>
601      *
602      * </ol>
603      *
604      * <p> The resulting URI string is then parsed as if by invoking the {@link
605      * #URI(String)} constructor and then invoking the {@link
606      * #parseServerAuthority()} method upon the result; this may cause a {@link
607      * URISyntaxException} to be thrown.  </p>
608      *
609      * @param   scheme    Scheme name
610      * @param   userInfo  User name and authorization information
611      * @param   host      Host name
612      * @param   port      Port number
613      * @param   path      Path
614      * @param   query     Query
615      * @param   fragment  Fragment
616      *
617      * @throws URISyntaxException
618      *         If both a scheme and a path are given but the path is relative,
619      *         if the URI string constructed from the given components violates
620      *         RFC&nbsp;2396, or if the authority component of the string is
621      *         present but cannot be parsed as a server-based authority
622      */
