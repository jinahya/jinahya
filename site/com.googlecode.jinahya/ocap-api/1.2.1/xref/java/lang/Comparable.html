39  /** 
40   * This interface imposes a total ordering on the objects of each class that
41   * implements it.  This ordering is referred to as the class's <i>natural
42   * ordering</i>, and the class's <tt>compareTo</tt> method is referred to as
43   * its <i>natural comparison method</i>.<p>
44   *
45   * Lists (and arrays) of objects that implement this interface can be sorted
46   * automatically by <tt>Collections.sort</tt> (and <tt>Arrays.sort</tt>).
47   * Objects that implement this interface can be used as keys in a sorted map
48   * or elements in a sorted set, without the need to specify a comparator.<p>
49   *
50   * The natural ordering for a class <tt>C</tt> is said to be <i>consistent
51   * with equals</i> if and only if <tt>(e1.compareTo((Object)e2) == 0)</tt> has
52   * the same boolean value as <tt>e1.equals((Object)e2)</tt> for every
53   * <tt>e1</tt> and <tt>e2</tt> of class <tt>C</tt>.  Note that <tt>null</tt>
54   * is not an instance of any class, and <tt>e.compareTo(null)</tt> should
55   * throw a <tt>NullPointerException</tt> even though <tt>e.equals(null)</tt>
56   * returns <tt>false</tt>.<p>
57   *
58   * It is strongly recommended (though not required) that natural orderings be
59   * consistent with equals.  This is so because sorted sets (and sorted maps)
60   * without explicit comparators behave "strangely" when they are used with
61   * elements (or keys) whose natural ordering is inconsistent with equals.  In
62   * particular, such a sorted set (or sorted map) violates the general contract
63   * for set (or map), which is defined in terms of the <tt>equals</tt>
64   * method.<p>
65   *
66   * For example, if one adds two keys <tt>a</tt> and <tt>b</tt> such that
67   * <tt>(!a.equals((Object)b) && a.compareTo((Object)b) == 0)</tt> to a sorted
68   * set that does not use an explicit comparator, the second <tt>add</tt>
69   * operation returns false (and the size of the sorted set does not increase)
70   * because <tt>a</tt> and <tt>b</tt> are equivalent from the sorted set's
71   * perspective.<p>
72   *
73   * Virtually all Java core classes that implement comparable have natural
74   * orderings that are consistent with equals.  One exception is
75   * <tt>java.math.BigDecimal</tt>, whose natural ordering equates
76   * <tt>BigDecimal</tt> objects with equal values and different precisions 
77   * (such as 4.0 and 4.00).
78   * NOTE: <B>java.math.BigDecimal</B> is found in J2ME CDC profiles such as 
79   * J2ME Foundation Profile.<p>
80   *
81   * For the mathematically inclined, the <i>relation</i> that defines
82   * the natural ordering on a given class C is:<pre>
83   *       {(x, y) such that x.compareTo((Object)y) &lt;= 0}.
84   * </pre> The <i>quotient</i> for this total order is: <pre>
85   *       {(x, y) such that x.compareTo((Object)y) == 0}.
86   * </pre>
87   *
88   * It follows immediately from the contract for <tt>compareTo</tt> that the
89   * quotient is an <i>equivalence relation</i> on <tt>C</tt>, and that the
90   * natural ordering is a <i>total order</i> on <tt>C</tt>.  When we say that a
91   * class's natural ordering is <i>consistent with equals</i>, we mean that the
92   * quotient for the natural ordering is the equivalence relation defined by
93   * the class's <tt>equals(Object)</tt> method:<pre>
94   *     {(x, y) such that x.equals((Object)y)}.
95   * </pre><p>
96   *
97   * This interface is a member of the 
98   * <a href="{@docRoot}/../guide/collections/index.html">
99   * Java Collections Framework</a>.
100  *
101  * @author  Josh Bloch
102  * @version 1.13, 02/02/00
103  * @see java.util.Comparator
104  * @see java.util.Collections#sort(java.util.List)
105  * @see java.util.Arrays#sort(Object[])
106  * @see java.util.SortedSet
107  * @see java.util.SortedMap
108  * @see java.util.TreeSet
109  * @see java.util.TreeMap
110  * @since 1.2
111  */

115     /** 
116      * Compares this object with the specified object for order.  Returns a
117      * negative integer, zero, or a positive integer as this object is less
118      * than, equal to, or greater than the specified object.<p>
119      *
120      * In the foregoing description, the notation
121      * <tt>sgn(</tt><i>expression</i><tt>)</tt> designates the mathematical
122      * <i>signum</i> function, which is defined to return one of <tt>-1</tt>,
123      * <tt>0</tt>, or <tt>1</tt> according to whether the value of <i>expression</i>
124      * is negative, zero or positive.
125      *
126      * The implementor must ensure <tt>sgn(x.compareTo(y)) ==
127      * -sgn(y.compareTo(x))</tt> for all <tt>x</tt> and <tt>y</tt>.  (This
128      * implies that <tt>x.compareTo(y)</tt> must throw an exception iff
129      * <tt>y.compareTo(x)</tt> throws an exception.)<p>
130      *
131      * The implementor must also ensure that the relation is transitive:
132      * <tt>(x.compareTo(y)&gt;0 &amp;&amp; y.compareTo(z)&gt;0)</tt> implies
133      * <tt>x.compareTo(z)&gt;0</tt>.<p>
134      *
135      * Finally, the implementer must ensure that <tt>x.compareTo(y)==0</tt>
136      * implies that <tt>sgn(x.compareTo(z)) == sgn(y.compareTo(z))</tt>, for
137      * all <tt>z</tt>.<p>
138      *
139      * It is strongly recommended, but <i>not</i> strictly required that
140      * <tt>(x.compareTo(y)==0) == (x.equals(y))</tt>.  Generally speaking, any
141      * class that implements the <tt>Comparable</tt> interface and violates
142      * this condition should clearly indicate this fact.  The recommended
143      * language is "Note: this class has a natural ordering that is
144      * inconsistent with equals."
145      * 
146      * @param   o the Object to be compared.
147      * @return  a negative integer, zero, or a positive integer as this object
148      *		is less than, equal to, or greater than the specified object.
149      * 
150      * @throws ClassCastException if the specified object's type prevents it
151      *         from being compared to this Object.
152      */
153     public int compareTo(java.lang.Object o);
154 }
