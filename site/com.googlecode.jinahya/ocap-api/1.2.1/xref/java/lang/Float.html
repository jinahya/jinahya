140      * Returns a string representation of the <code>float</code>
141      * argument. All characters mentioned below are ASCII characters.
142      * <ul>
143      * <li>If the argument is NaN, the result is the string
144      * &quot;<code>NaN</code>&quot;.
145      * <li>Otherwise, the result is a string that represents the sign and 
146      *     magnitude (absolute value) of the argument. If the sign is
147      *     negative, the first character of the result is
148      *     '<code>-</code>' (<code>'&#92;u002D'</code>); if the sign is
149      *     positive, no sign character appears in the result. As for
150      *     the magnitude <i>m</i>:
151      * <ul>
152      * <li>If <i>m</i> is infinity, it is represented by the characters 
153      *     <code>"Infinity"</code>; thus, positive infinity produces
154      *     the result <code>"Infinity"</code> and negative infinity
155      *     produces the result <code>"-Infinity"</code>.
156      * <li>If <i>m</i> is zero, it is represented by the characters 
157      *     <code>"0.0"</code>; thus, negative zero produces the result
158      *     <code>"-0.0"</code> and positive zero produces the result
159      *     <code>"0.0"</code>.
160      * <li> If <i>m</i> is greater than or equal to 10<sup>-3</sup> but 
161      *      less than 10<sup>7</sup>, then it is represented as the
162      *      integer part of <i>m</i>, in decimal form with no leading
163      *      zeroes, followed by '<code>.</code>'
164      *      (<code>'&#92;u002E'</code>), followed by one or more
165      *      decimal digits representing the fractional part of
166      *      <i>m</i>.
167      * <li> If <i>m</i> is less than 10<sup>-3</sup> or greater than or
168      *      equal to 10<sup>7</sup>, then it is represented in
169      *      so-called "computerized scientific notation." Let <i>n</i>
170      *      be the unique integer such that 10<sup><i>n</i> </sup>&lt;= 
171      *      <i>m</i> &lt; 10<sup><i>n</i>+1</sup>; then let <i>a</i> 
172      *      be the mathematically exact quotient of <i>m</i> and 
173      *      10<sup><i>n</i></sup> so that 1 &lt;= <i>a</i> &lt; 10.
174      *      The magnitude is then represented as the integer part of
175      *      <i>a</i>, as a single decimal digit, followed by
176      *      '<code>.</code>' (<code>'&#92;u002E'</code>), followed by
177      *      decimal digits representing the fractional part of
178      *      <i>a</i>, followed by the letter '<code>E</code>'
179      *      (<code>'&#92;u0045'</code>), followed by a representation
180      *      of <i>n</i> as a decimal integer, as produced by the
181      *      method <code>{@link
182      *      java.lang.Integer#toString(int)}</code>.
183      * </ul>
184      * </ul>
185      * How many digits must be printed for the fractional part of
186      * <i>m</i> or <i>a</i>? There must be at least one digit
187      * to represent the fractional part, and beyond that as many, but
188      * only as many, more digits as are needed to uniquely distinguish
189      * the argument value from adjacent values of type
190      * <code>float</code>. That is, suppose that <i>x</i> is the
191      * exact mathematical value represented by the decimal
192      * representation produced by this method for a finite nonzero
193      * argument <i>f</i>. Then <i>f</i> must be the <code>float</code>
194      * value nearest to <i>x</i>; or, if two <code>float</code> values are
195      * equally close to <i>x</i>, then <i>f</i> must be one of
196      * them and the least significant bit of the significand of
197      * <i>f</i> must be <code>0</code>.
198      * <p>

200      * value, use subclasses of {@link java.text.NumberFormat}.

203      * @return a string representation of the argument.
204      */

210      * Returns a <code>Float</code> object holding the
211      * <code>float</code> value represented by the argument string
212      * <code>s</code>.
213      * <p>
214      * If <code>s</code> is <code>null</code>, then a 
215      * <code>NullPointerException</code> is thrown. 
216      * <p>
217      * Leading and trailing whitespace characters in <code>s</code>
218      * are ignored. The rest of <code>s</code> should constitute a
219      * <i>FloatValue</i> as described by the lexical syntax rules:
220      * <blockquote><i>
221      * <dl>
222      * <dt>FloatValue:
223      * <dd><i>Sign<sub>opt</sub></i> <code>NaN</code>
224      * <dd><i>Sign<sub>opt</sub></i> <code>Infinity</code>
225      * <dd>Sign<sub>opt</sub> FloatingPointLiteral
226      * </dl>
227      * </i></blockquote>
228      * where <i>Sign</i> and <i>FloatingPointLiteral</i> are as
229      * defined in <a href="http://java.sun.com/docs/books/jls/second_edition/html/lexical.doc.html#230798">&sect;3.10.2</a> 
230      * of the <a href="http://java.sun.com/docs/books/jls/html/">Java 
231      * Language Specification</a>. If <code>s</code> does not have the
232      * form of a <i>FloatValue</i>, then a
233      * <code>NumberFormatException</code> is thrown. Otherwise,
234      * <code>s</code> is regarded as representing an exact decimal
235      * value in the usual "computerized scientific notation"; this
236      * exact decimal value is then conceptually converted to an
237      * "infinitely precise" binary value that is then rounded to type
238      * <code>float</code> by the usual round-to-nearest rule of IEEE
239      * 754 floating-point arithmetic, which includes preserving the
240      * sign of a zero value.  Finally, a <code>Float</code> object
241      * representing this <code>float</code> value is returned.
242      * <p>
243      * To interpret localized string representations of a
244      * floating-point value, use subclasses of {@link
245      * java.text.NumberFormat}.
246      *
247      * <p>Note that trailing format specifiers, specifiers that
248      * determine the type of a floating-point literal
249      * (<code>1.0f</code> is a <code>float</code> value;
250      * <code>1.0d</code> is a <code>double</code> value), do
251      * <em>not</em> influence the results of this method.  In other
252      * words, the numerical value of the input string is converted
253      * directly to the target floating-point type.  In general, the
254      * two-step sequence of conversions, string to <code>double</code>
255      * followed by <code>double</code> to <code>float</code>, is
256      * <em>not</em> equivalent to converting a string directly to
257      * <code>float</code>.  For example, if first converted to an
258      * intermediate <code>double</code> and then to
259      * <code>float</code>, the string<br>
260      * <code>"1.00000017881393421514957253748434595763683319091796875001d"</code><br>
261      * results in the <code>float</code> value
262      * <code>1.0000002f</code>; if the string is converted directly to
263      * <code>float</code>, <code>1.000000<b>1</b>f</code> results.
264      *
265      * @param      s   the string to be parsed.
266      * @return     a <code>Float</code> object holding the value
267      *             represented by the <code>String</code> argument.
268      * @exception  NumberFormatException  if the string does not contain a
269      *               parsable number.

544     /** 
545      * Returns the <code>float</code> value corresponding to a given
546      * bit represention.
547      * The argument is considered to be a representation of a
548      * floating-point value according to the IEEE 754 floating-point
549      * "single format" bit layout.
550      * <p>
551      * If the argument is <code>0x7f800000</code>, the result is positive
552      * infinity.
553      * <p>
554      * If the argument is <code>0xff800000</code>, the result is negative
555      * infinity.
556      * <p>
557      * If the argument is any value in the range
558      * <code>0x7f800001</code> through <code>0x7fffffff</code> or in
559      * the range <code>0xff800001</code> through
560      * <code>0xffffffff</code>, the result is a NaN.  No IEEE 754
561      * floating-point operation provided by Java can distinguish
562      * between two NaN values of the same type with different bit
563      * patterns.  Distinct values of NaN are only distinguishable by
564      * use of the <code>Float.floatToRawIntBits</code> method.
565      * <p>
566      * In all other cases, let <i>s</i>, <i>e</i>, and <i>m</i> be three 
567      * values that can be computed from the argument: 
568      * <blockquote><pre>
569      * int s = ((bits &gt;&gt; 31) == 0) ? 1 : -1;
570      * int e = ((bits &gt;&gt; 23) & 0xff);
571      * int m = (e == 0) ?
572      *                 (bits & 0x7fffff) &lt;&lt; 1 :
573      *                 (bits & 0x7fffff) | 0x800000;

575      * Then the floating-point result equals the value of the mathematical

582      * differences between the two kinds of NaN are generally not
583      * visible in Java.  Arithmetic operations on signaling NaNs turn

586      * signaling NaN also performs that conversion.  In particular,
587      * copying a signaling NaN to return it to the calling method may
588      * perform this conversion.  So <code>intBitsToFloat</code> may
589      * not be able to return a <code>float</code> with a signaling NaN

591      * <code>floatToRawIntBits(intBitsToFloat(start))</code> may
592      * <i>not</i> equal <code>start</code>.  Moreover, which
593      * particular bit patterns represent signaling NaNs is platform
594      * dependent; although all NaN bit patterns, quiet or signaling,
595      * must be in the NaN range identified above.
596      *
597      * @param   bits   an integer.
