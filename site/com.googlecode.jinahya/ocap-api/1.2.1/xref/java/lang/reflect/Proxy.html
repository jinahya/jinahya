42  /** 
43   * <code>Proxy</code> provides static methods for creating dynamic proxy
44   * classes and instances, and it is also the superclass of all
45   * dynamic proxy classes created by those methods.
46   *
47   * <p>To create a proxy for some interface <code>Foo</code>:
48   * <pre>
49   *     InvocationHandler handler = new MyInvocationHandler(...);
50   *     Class proxyClass = Proxy.getProxyClass(
51   *         Foo.class.getClassLoader(), new Class[] { Foo.class });
52   *     Foo f = (Foo) proxyClass.
53   *         getConstructor(new Class[] { InvocationHandler.class }).
54   *         newInstance(new Object[] { handler });
55   * </pre>
56   * or more simply:
57   * <pre>
58   *     Foo f = (Foo) Proxy.newProxyInstance(Foo.class.getClassLoader(),
59   *                                          new Class[] { Foo.class },
60   *                                          handler);
61   * </pre>
62   *
63   * <p>A <i>dynamic proxy class</i> (simply referred to as a <i>proxy
64   * class</i> below) is a class that implements a list of interfaces
65   * specified at runtime when the class is created, with behavior as
66   * described below.
67   *
68   * A <i>proxy interface</i> is such an interface that is implemented
69   * by a proxy class.
70   *
71   * A <i>proxy instance</i> is an instance of a proxy class.
72   *
73   * Each proxy instance has an associated <i>invocation handler</i>
74   * object, which implements the interface {@link InvocationHandler}.
75   * A method invocation on a proxy instance through one of its proxy
76   * interfaces will be dispatched to the {@link InvocationHandler#invoke
77   * invoke} method of the instance's invocation handler, passing the proxy
78   * instance, a <code>java.lang.reflect.Method</code> object identifying
79   * the method that was invoked, and an array of type <code>Object</code>
80   * containing the arguments.  The invocation handler processes the
81   * encoded method invocation as appropriate and the result that it
82   * returns will be returned as the result of the method invocation on
83   * the proxy instance.
84   *
85   * <p>A proxy class has the following properties:
86   *
87   * <ul>
88   * <li>Proxy classes are public, final, and not abstract.
89   *
90   * <li>The unqualified name of a proxy class is unspecified.  The space
91   * of class names that begin with the string <code>"$Proxy"</code>
92   * should be, however, reserved for proxy classes.
93   *
94   * <li>A proxy class extends <code>java.lang.reflect.Proxy</code>.
95   *
96   * <li>A proxy class implements exactly the interfaces specified at its
97   * creation, in the same order.
98   *
99   * <li>If a proxy class implements a non-public interface, then it will
100  * be defined in the same package as that interface.  Otherwise, the
101  * package of a proxy class is also unspecified.  Note that package
102  * sealing will not prevent a proxy class from being successfully defined
103  * in a particular package at runtime, and neither will classes already
104  * defined by the same class loader and the same package with particular
105  * signers.
106  *
107  * <li>Since a proxy class implements all of the interfaces specified at
108  * its creation, invoking <code>getInterfaces</code> on its
109  * <code>Class</code> object will return an array containing the same
110  * list of interfaces (in the order specified at its creation), invoking
111  * <code>getMethods</code> on its <code>Class</code> object will return
112  * an array of <code>Method</code> objects that include all of the
113  * methods in those interfaces, and invoking <code>getMethod</code> will
114  * find methods in the proxy interfaces as would be expected.
115  *
116  * <li>The {@link Proxy#isProxyClass Proxy.isProxyClass} method will
117  * return true if it is passed a proxy class-- a class returned by
118  * <code>Proxy.getProxyClass</code> or the class of an object returned by
119  * <code>Proxy.newProxyInstance</code>-- and false otherwise.
120  *
121  * <li>The <code>java.security.ProtectionDomain</code> of a proxy class
122  * is the same as that of system classes loaded by the bootstrap class
123  * loader, such as <code>java.lang.Object</code>, because the code for a
124  * proxy class is generated by trusted system code.  This protection
125  * domain will typically be granted
126  * <code>java.security.AllPermission</code>.
127  *
128  * <li>Each proxy class has one public constructor that takes one argument,
129  * an implementation of the interface {@link InvocationHandler}, to set
130  * the invocation handler for a proxy instance.  Rather than having to use
131  * the reflection API to access the public constructor, a proxy instance
132  * can be also be created by calling the {@link Proxy#newProxyInstance
133  * Proxy.newInstance} method, which combines the actions of calling
134  * {@link Proxy#getProxyClass Proxy.getProxyClass} with invoking the
135  * constructor with an invocation handler.
136  * </ul>
137  *
138  * <p>A proxy instance has the following properties:
139  *
140  * <ul>
141  * <li>Given a proxy instance <code>proxy</code> and one of the
142  * interfaces implemented by its proxy class <code>Foo</code>, the
143  * following expression will return true:
144  * <pre>
145  *     <code>proxy instanceof Foo</code>
146  * </pre>
147  * and the following cast operation will succeed (rather than throwing
148  * a <code>ClassCastException</code>):
149  * <pre>
150  *     <code>(Foo) proxy</code>
151  * </pre>
152  *
153  * <li>Each proxy instance has an associated invocation handler, the one
154  * that was passed to its constructor.  The static
155  * {@link Proxy#getInvocationHandler Proxy.getInvocationHandler} method
156  * will return the invocation handler associated with the proxy instance
157  * passed as its argument.
158  *
159  * <li>An interface method invocation on a proxy instance will be
160  * encoded and dispatched to the invocation handler's {@link
161  * InvocationHandler#invoke invoke} method as described in the
162  * documentation for that method.
163  *
164  * <li>An invocation of the <code>hashCode</code>,
165  * <code>equals</code>, or <code>toString</code> methods declared in
166  * <code>java.lang.Object</code> on a proxy instance will be encoded and
167  * dispatched to the invocation handler's <code>invoke</code> method in
168  * the same manner as interface method invocations are encoded and
169  * dispatched, as described above.  The declaring class of the
170  * <code>Method</code> object passed to <code>invoke</code> will be
171  * <code>java.lang.Object</code>.  Other public methods of a proxy
172  * instance inherited from <code>java.lang.Object</code> are not
173  * overridden by a proxy class, so invocations of those methods behave
174  * like they do for instances of <code>java.lang.Object</code>.
175  * </ul>
176  *
177  * <h3>Methods Duplicated in Multiple Proxy Interfaces</h3>
178  *
179  * <p>When two or more interfaces of a proxy class contain a method with
180  * the same name and parameter signature, the order of the proxy class's
181  * interfaces becomes significant.  When such a <i>duplicate method</i>
182  * is invoked on a proxy instance, the <code>Method</code> object passed
183  * to the invocation handler will not necessarily be the one whose
184  * declaring class is assignable from the reference type of the interface
185  * that the proxy's method was invoked through.  This limitation exists
186  * because the corresponding method implementation in the generated proxy
187  * class cannot determine which interface it was invoked through.
188  * Therefore, when a duplicate method is invoked on a proxy instance,
189  * the <code>Method</code> object for the method in the foremost interface
190  * that contains the method (either directly or inherited through a
191  * superinterface) in the proxy class's list of interfaces is passed to
192  * the invocation handler's <code>invoke</code> method, regardless of the
193  * reference type through which the method invocation occurred.
194  *
195  * <p>If a proxy interface contains a method with the same name and
196  * parameter signature as the <code>hashCode</code>, <code>equals</code>,
197  * or <code>toString</code> methods of <code>java.lang.Object</code>,
198  * when such a method is invoked on a proxy instance, the
199  * <code>Method</code> object passed to the invocation handler will have
200  * <code>java.lang.Object</code> as its declaring class.  In other words,
201  * the public, non-final methods of <code>java.lang.Object</code>
202  * logically precede all of the proxy interfaces for the determination of
203  * which <code>Method</code> object to pass to the invocation handler.
204  *
205  * <p>Note also that when a duplicate method is dispatched to an
206  * invocation handler, the <code>invoke</code> method may only throw
207  * checked exception types that are assignable to one of the exception
208  * types in the <code>throws</code> clause of the method in <i>all</i> of
209  * the proxy interfaces that it can be invoked through.  If the
210  * <code>invoke</code> method throws a checked exception that is not
211  * assignable to any of the exception types declared by the method in one
212  * of the the proxy interfaces that it can be invoked through, then an
213  * unchecked <code>UndeclaredThrowableException</code> will be thrown by
214  * the invocation on the proxy instance.  This restriction means that not
215  * all of the exception types returned by invoking
216  * <code>getExceptionTypes</code> on the <code>Method</code> object
217  * passed to the <code>invoke</code> method can necessarily be thrown
218  * successfully by the <code>invoke</code> method.
219  *
220  * @author	Peter Jones
221  * @version	1.8, 00/02/02
222  * @see		InvocationHandler
223  * @since	JDK1.3
224  */

247      * permutation of interfaces has already been defined by the class
248      * loader, then the existing proxy class will be returned; otherwise,
249      * a proxy class for those interfaces will be generated dynamically
250      * and defined by the class loader.
251      *
252      * <p>There are several restrictions on the parameters that may be
253      * passed to <code>Proxy.getProxyClass</code>:
254      *
255      * <ul>
256      * <li>All of the <code>Class</code> objects in the
257      * <code>interfaces</code> array must represent interfaces, not
258      * classes or primitive types.
259      *
260      * <li>No two elements in the <code>interfaces</code> array may
261      * refer to identical <code>Class</code> objects.
262      *
263      * <li>All of the interface types must be visible by name through the
264      * specified class loader.  In other words, for class loader
265      * <code>cl</code> and every interface <code>i</code>, the following
266      * expression must be true:
267      * <pre>
268      *     Class.forName(i.getName(), false, cl) == i
269      * </pre>
270      *
271      * <li>All non-public interfaces must be in the same package;
272      * otherwise, it would not be possible for the proxy class to
273      * implement all of the interfaces, regardless of what package it is
274      * defined in.
275      *
276      * <li>No two interfaces may each have a method with the same name
277      * and parameter signature but different return type.
278      *
279      * <li>The resulting proxy class must not exceed any limits imposed
280      * on classes by the virtual machine.  For example, the VM may limit
281      * the number of interfaces that a class may implement to 65535; in
282      * that case, the size of the <code>interfaces</code> array must not
283      * exceed 65535.
284      * </ul>
285      *
286      * <p>If any of these restrictions are violated,
287      * <code>Proxy.getProxyClass</code> will throw an
288      * <code>IllegalArgumentException</code>.  If the <code>interfaces</code>
289      * array argument or any of its elements are <code>null</code>, a
290      * <code>NullPointerException</code> will be thrown.
291      *
292      * <p>Note that the order of the specified proxy interfaces is
293      * significant: two requests for a proxy class with the same combination
294      * of interfaces but in a different order will result in two distinct
295      * proxy classes.
296      *
297      * @param	loader the class loader to define the proxy class
298      * @param	interfaces the list of interfaces for the proxy class
299      *		to implement
300      * @return	a proxy class that is defined in the specified class loader
301      *		and that implements the specified interfaces
302      * @throws	IllegalArgumentException if any of the restrictions on the
303      *		parameters that may be passed to <code>getProxyClass</code>
304      *		are violated
305      * @throws	NullPointerException if the <code>interfaces</code> array
306      *		argument or any of its elements are <code>null</code>
