74       * Terminates the currently running Java virtual machine by initiating its
75       * shutdown sequence.  This method never returns normally.  The argument
76       * serves as a status code; by convention, a nonzero status code indicates
77       * abnormal termination.
78       *
79       * <p> The virtual machine's shutdown sequence consists of two phases.  In
80       * the first phase all registered {@link #addShutdownHook shutdown hooks},
81       * if any, are started in some unspecified order and allowed to run
82       * concurrently until they finish.  In the second phase all uninvoked
83       * finalizers are run if finalization-on-exit
84       * has been enabled.  Once this is done the virtual machine {@link #halt
85       * halts}.
86       *
87       * <p> If this method is invoked after the virtual machine has begun its
88       * shutdown sequence then if shutdown hooks are being run this method will
89       * block indefinitely.  If shutdown hooks have already been run and on-exit
90       * finalization has been enabled then this method halts the virtual machine
91       * with the given status code if the status is nonzero; otherwise, it
92       * blocks indefinitely.
93       *
94       * <p> The <tt>{@link System#exit(int) System.exit}</tt> method is the
95       * conventional and convenient means of invoking this method. <p>
96       *
97       * If process model exists, Runtime.exit() must rely on the process 
98       * exiting to release resources.
99       * <p>
100      * If process model does not exist, and a security manager exists,
101      * the security manager by default must prohibit calls to Runtime.exit() 
102      * by throwing SecurityException.  Otherwise, the system will hang 
103      * indefinitely when called.
104      * <p>
105      * @param  status
106      *         Termination status.  By convention, a nonzero status code
107      *         indicates abnormal termination.
108      *
109      * @throws SecurityException
110      *         If a security manager is present and its <tt>{@link
111      *         SecurityManager#checkExit checkExit}</tt> method does not permit
112      *         exiting with the specified status
113      *
114      * @see java.lang.SecurityException
115      * @see java.lang.SecurityManager#checkExit(int)
116      * @see #addShutdownHook
117      * @see #removeShutdownHook
118      * @see #halt(int)
119      */

122     /** 
123      * Registers a new virtual-machine shutdown hook.
124      *
125      * <p> The Java virtual machine <i>shuts down</i> in response to two kinds
126      * of events:
127      *
128      *   <ul>
129      *
130      *   <p> <li> The program <i>exits</i> normally, when the last non-daemon
131      *   thread exits or when the <tt>{@link #exit exit}</tt> (equivalently,
132      *   <tt>{@link System#exit(int) System.exit}</tt>) method is invoked, or
133      *
134      *   <p> <li> The virtual machine is <i>terminated</i> in response to a
135      *   user interrupt, such as typing <tt>^C</tt>, or a system-wide event,
136      *   such as user logoff or system shutdown.
137      *
138      *   </ul>
139      *
140      * <p> A <i>shutdown hook</i> is simply an initialized but unstarted
141      * thread.  When the virtual machine begins its shutdown sequence it will
142      * start all registered shutdown hooks in some unspecified order and let
143      * them run concurrently.  When all the hooks have finished it will then
144      * run all uninvoked finalizers if finalization-on-exit has been enabled.
145      * Finally, the virtual machine will halt.  Note that daemon threads will
146      * continue to run during the shutdown sequence, as will non-daemon threads
147      * if shutdown was initiated by invoking the <tt>{@link #exit exit}</tt>
148      * method.
149      *
150      * <p> Once the shutdown sequence has begun it can be stopped only by
151      * invoking the <tt>{@link #halt halt}</tt> method, which forcibly
152      * terminates the virtual machine.
153      *
154      * <p> Once the shutdown sequence has begun it is impossible to register a
155      * new shutdown hook or de-register a previously-registered hook.
156      * Attempting either of these operations will cause an
157      * <tt>{@link IllegalStateException}</tt> to be thrown.
158      *
159      * <p> Shutdown hooks run at a delicate time in the life cycle of a virtual
160      * machine and should therefore be coded defensively.  They should, in
161      * particular, be written to be thread-safe and to avoid deadlocks insofar
162      * as possible.  They should also not rely blindly upon services that may
163      * have registered their own shutdown hooks and therefore may themselves in
164      * the process of shutting down.
165      *
166      * <p> Shutdown hooks should also finish their work quickly.  When a
167      * program invokes <tt>{@link #exit exit}</tt> the expectation is
168      * that the virtual machine will promptly shut down and exit.  When the
169      * virtual machine is terminated due to user logoff or system shutdown the
170      * underlying operating system may only allow a fixed amount of time in
171      * which to shut down and exit.  It is therefore inadvisable to attempt any
172      * user interaction or to perform a long-running computation in a shutdown
173      * hook.
174      *
175      * <p> Uncaught exceptions are handled in shutdown hooks just as in any
176      * other thread, by invoking the <tt>{@link ThreadGroup#uncaughtException
177      * uncaughtException}</tt> method of the thread's <tt>{@link
178      * ThreadGroup}</tt> object.  The default implementation of this method
179      * prints the exception's stack trace to <tt>{@link System#err}</tt> and
180      * terminates the thread; it does not cause the virtual machine to exit or
181      * halt.
182      *
183      * <p> In rare circumstances the virtual machine may <i>abort</i>, that is,
184      * stop running without shutting down cleanly.  This occurs when the
185      * virtual machine is terminated externally, for example with the
186      * <tt>SIGKILL</tt> signal on Unix or the <tt>TerminateProcess</tt> call on
187      * Microsoft Windows.  The virtual machine may also abort if a native method goes awry
188      * by, for example, corrupting internal data structures or attempting to
189      * access nonexistent memory.  If the virtual machine aborts then no
190      * guarantee can be made about whether or not any shutdown hooks will be
191      * run. <p>
192      *
193      * @param   hook
194      *          An initialized but unstarted <tt>{@link Thread}</tt> object
195      *
196      * @throws  IllegalArgumentException
197      *          If the specified hook has already been registered,
198      *          or if it can be determined that the hook is already running or
199      *          has already been run
200      *
201      * @throws  IllegalStateException
202      *          If the virtual machine is already in the process
203      *          of shutting down
204      *
205      * @throws  SecurityException
206      *          If a security manager is present and it denies
207      *          <tt>{@link RuntimePermission}("shutdownHooks")</tt>
208      *
209      * @see #removeShutdownHook
210      * @see #halt(int)
211      * @see #exit(int)
212      * @since 1.3
213      */
214     public void addShutdownHook(java.lang.Thread hook) { }
