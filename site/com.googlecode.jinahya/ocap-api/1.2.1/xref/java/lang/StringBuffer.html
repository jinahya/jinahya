39  /** 
40   * A string buffer implements a mutable sequence of characters. 
41   * A string buffer is like a {@link String}, but can be modified. At any 
42   * point in time it contains some particular sequence of characters, but 
43   * the length and content of the sequence can be changed through certain 
44   * method calls.
45   * <p>
46   * String buffers are safe for use by multiple threads. The methods 
47   * are synchronized where necessary so that all the operations on any 
48   * particular instance behave as if they occur in some serial order 
49   * that is consistent with the order of the method calls made by each of 
50   * the individual threads involved. 
51   * <p>
52   * String buffers are used by the compiler to implement the binary 
53   * string concatenation operator <code>+</code>. For example, the code:
54   * <p><blockquote><pre>
55   *     x = "a" + 4 + "c"
56   * </pre></blockquote><p>
57   * is compiled to the equivalent of: 
58   * <p><blockquote><pre>
59   *     x = new StringBuffer().append("a").append(4).append("c")
60   *                           .toString()
61   * </pre></blockquote>
62   * which creates a new string buffer (initially empty), appends the string
63   * representation of each operand to the string buffer in turn, and then
64   * converts the contents of the string buffer to a string. Overall, this avoids
65   * creating many temporary strings.
66   * <p>
67   * The principal operations on a <code>StringBuffer</code> are the 
68   * <code>append</code> and <code>insert</code> methods, which are 
69   * overloaded so as to accept data of any type. Each effectively 
70   * converts a given datum to a string and then appends or inserts the 
71   * characters of that string to the string buffer. The 
72   * <code>append</code> method always adds these characters at the end 
73   * of the buffer; the <code>insert</code> method adds the characters at 
74   * a specified point. 
75   * <p>
76   * For example, if <code>z</code> refers to a string buffer object 
77   * whose current contents are "<code>start</code>", then 
78   * the method call <code>z.append("le")</code> would cause the string 
79   * buffer to contain "<code>startle</code>", whereas 
80   * <code>z.insert(4, "le")</code> would alter the string buffer to 
81   * contain "<code>starlet</code>". 
82   * <p>
83   * In general, if sb refers to an instance of a <code>StringBuffer</code>, 
84   * then <code>sb.append(x)</code> has the same effect as 
85   * <code>sb.insert(sb.length(),&nbsp;x)</code>.
86   * <p>
87   * Every string buffer has a capacity. As long as the length of the 
88   * character sequence contained in the string buffer does not exceed 
89   * the capacity, it is not necessary to allocate a new internal 
90   * buffer array. If the internal buffer overflows, it is 
91   * automatically made larger. 
92   *
93   * @author	Arthur van Hoff
94   * @version 	1.78, 05/16/03
95   * @see     java.io.ByteArrayOutputStream
96   * @see     java.lang.String
97   * @since   JDK1.0
98   */
