39  /** 
40   * Abstract class for representing access to a system resource.
41   * All permissions have a name (whose interpretation depends on the subclass),
42   * as well as abstract functions for defining the semantics of the
43   * particular Permission subclass. 
44   * 
45   * <p>Most Permission objects also include an "actions" list that tells the actions 
46   * that are permitted for the object.  For example, 
47   * for a <code>java.io.FilePermission</code> object, the permission name is
48   * the pathname of a file (or directory), and the actions list
49   * (such as "read, write") specifies which actions are granted for the
50   * specified file (or for files in the specified directory).
51   * The actions list is optional for Permission objects, such as 
52   * <code>java.lang.RuntimePermission</code>,
53   * that don't need such a list; you either have the named permission (such
54   * as "system.exit") or you don't.
55   * 
56   * <p>An important method that must be implemented by each subclass is
57   * the <code>implies</code> method to compare Permissions. Basically,
58   * "permission p1 implies permission p2" means that
59   * if one is granted permission p1, one is naturally granted permission p2.
60   * Thus, this is not an equality test, but rather more of a
61   * subset test.
62   * 
63   * <P> Permission objects are similar to String objects in that they
64   * are immutable once they have been created. Subclasses should not
65   * provide methods that can change the state of a permission
66   * once it has been created.
67   *
68   * @see Permissions
69   * @see PermissionCollection
70   *
71   * @version 1.35 00/02/02
72   *
73   * @author Marianne Mueller
74   * @author Roland Schemers 
75   */

112      * This must be implemented by subclasses of Permission, as they are the
