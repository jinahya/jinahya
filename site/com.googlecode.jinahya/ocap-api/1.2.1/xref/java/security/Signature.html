46  /** 
47   * This Signature class is used to provide applications the functionality
48   * of a digital signature algorithm. Digital signatures are used for
49   * authentication and integrity assurance of digital data.
50   *
51   * <p> The signature algorithm can be, among others, the NIST standard
52   * DSA, using DSA and SHA-1. The DSA algorithm using the
53   * SHA-1 message digest algorithm can be specified as <tt>SHA1withDSA</tt>.
54   * In the case of RSA, there are multiple choices for the message digest
55   * algorithm, so the signing algorithm could be specified as, for example,
56   * <tt>MD2withRSA</tt>, <tt>MD5withRSA</tt>, or <tt>SHA1withRSA</tt>.
57   * The algorithm name must be specified, as there is no default.
58   *
59   * <p>Like other algorithm-based classes in Java Security, Signature 
60   * provides implementation-independent algorithms, whereby a caller 
61   * (application code) requests a particular signature algorithm
62   * and is handed back a properly initialized Signature object. It is
63   * also possible, if desired, to request a particular algorithm from a
64   * particular provider. See the <code>getInstance </code> methods.
65   *
66   * <p>Thus, there are two ways to request a Signature algorithm object: by
67   * specifying either just an algorithm name, or both an algorithm name
68   * and a package provider. <ul>
69   *
70   * <li>If just an algorithm name is specified, the system will
71   * determine if there is an implementation of the algorithm requested
72   * available in the environment, and if there is more than one, if
73   * there is a preferred one.<p>
74   * 
75   * <li>If both an algorithm name and a package provider are specified,
76   * the system will determine if there is an implementation of the
77   * algorithm in the package requested, and throw an exception if there
78   * is not.
79   *
80   * </ul>
81   *
82   * <p>A Signature object can be used to generate and verify digital
83   * signatures.
84   *
85   * <p>There are three phases to the use of a Signature object for
86   * either signing data or verifying a signature:<ol>
87   *
88   * <li>Initialization, with either 
89   *
90   *     <ul>
91   *
92   *     <li>a public key, which initializes the signature for
93   *     verification (see {@link #initVerify(PublicKey) initVerify}), or
94   *
95   *     <li>a private key (and optionally a Secure Random Number Generator),
96   *     which initializes the signature for signing
97   *     (see {@link #initSign(PrivateKey)}
98   *     and {@link #initSign(PrivateKey, SecureRandom)}).
99   *
100  *     </ul><p>
101  *
102  * <li>Updating<p>
103  *
104  * <p>Depending on the type of initialization, this will update the
105  * bytes to be signed or verified. See the 
106  * {@link #update(byte) update} methods.<p>
107  *
108  * <li>Signing or Verifying a signature on all updated bytes. See the 
109  * {@link #sign() sign} methods and the {@link #verify(byte[]) verify}
110  * method.
111  *
112  * </ol>
113  *
114  * <p>Note that this class is abstract and extends from
115  * <code>SignatureSpi</code> for historical reasons.
116  * Application developers should only take notice of the methods defined in
117  * this <code>Signature</code> class; all the methods in
118  * the superclass are intended for cryptographic service providers who wish to
119  * supply their own implementations of digital signature algorithms.
120  *
121  * @author Benjamin Renaud 
122  *
123  * @version 1.86, 05/03/00
124  */
