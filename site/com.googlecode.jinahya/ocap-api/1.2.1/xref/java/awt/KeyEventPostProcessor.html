36  package java.awt;
37  
38  import java.awt.event.KeyEvent;
39  
40  /** 
41   * A KeyEventPostProcessor cooperates with the current KeyboardFocusManager
42   * in the final resolution of all unconsumed KeyEvents. KeyEventPostProcessors
43   * registered with the current KeyboardFocusManager will receive KeyEvents
44   * after the KeyEvents have been dispatched to and handled by their targets.
45   * KeyEvents that would have been otherwise discarded because no Component in
46   * the application currently owns the focus will also be forwarded to
47   * registered KeyEventPostProcessors. This will allow applications to implement
48   * features that require global KeyEvent post-handling, such as menu shortcuts.
49   * <p>
50   * Note that the KeyboardFocusManager itself implements KeyEventPostProcessor.
51   * By default, the current KeyboardFocusManager will be the final
52   * KeyEventPostProcessor in the chain. The current KeyboardFocusManager cannot
53   * be completely deregistered as a KeyEventPostProcessor. However, if a
54   * KeyEventPostProcessor reports that no further post-processing of the
55   * KeyEvent should take place, the AWT will consider the event fully handled
56   * and will take no additional action with regard to the event. (While it is
57   * possible for client code to register the current KeyboardFocusManager as
58   * a KeyEventPostProcessor one or more times, this is usually unnecessary and
59   * not recommended.)
60   *
61   * @author David Mendenhall
62   * @version 1.4, 01/23/03
63   *
64   * @see KeyboardFocusManager#addKeyEventPostProcessor
65   * @see KeyboardFocusManager#removeKeyEventPostProcessor
66   * @since 1.4
67   */
68  public interface KeyEventPostProcessor
69  {
70  
71      /** 
72       * This method is called by the current KeyboardFocusManager, requesting
73       * that this KeyEventPostProcessor perform any necessary post-processing
74       * which should be part of the KeyEvent's final resolution. At the time
75       * this method is invoked, typically the KeyEvent has already been
76       * dispatched to and handled by its target. However, if no Component in
77       * the application currently owns the focus, then the KeyEvent has not
78       * been dispatched to any Component. Typically, KeyEvent post-processing
79       * will be used to implement features which require global KeyEvent
80       * post-handling, such as menu shortcuts. Note that if a
81       * KeyEventPostProcessor wishes to dispatch the KeyEvent, it must use
82       * <code>redispatchEvent</code> to prevent the AWT from recursively
83       * requesting that this KeyEventPostProcessor perform post-processing
84       * of the event again.
85       * <p>
86       * If an implementation of this method returns <code>false</code>, then the
87       * KeyEvent is passed to the next KeyEventPostProcessor in the chain,
88       * ending with the current KeyboardFocusManager. If an implementation
89       * returns <code>true</code>, the KeyEvent is assumed to have been fully
90       * handled (although this need not be the case), and the AWT will take no
91       * further action with regard to the KeyEvent. If an implementation
92       * consumes the KeyEvent but returns <code>false</code>, the consumed
93       * event will still be passed to the next KeyEventPostProcessor in the
94       * chain. It is important for developers to check whether the KeyEvent has
95       * been consumed before performing any post-processing of the KeyEvent. By
96       * default, the current KeyboardFocusManager will perform no post-
97       * processing in response to a consumed KeyEvent.
98       *
99       * @param e the KeyEvent to post-process
100      * @return <code>true</code> if the AWT should take no further action with
101      *         regard to the KeyEvent; <code>false</code> otherwise
102      * @see KeyboardFocusManager#redispatchEvent
103      */
104     public boolean postProcessKeyEvent(KeyEvent e);
105 }
