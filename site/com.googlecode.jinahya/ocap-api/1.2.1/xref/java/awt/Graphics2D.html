53  /** 
54   * This <code>Graphics2D</code> class extends the
55   * {@link Graphics} class to provide more sophisticated
56   * control over
57   * <em>graphics operations.</em>
58   * 
59   * <p>
60   * <h2>Coordinate Spaces</h2>
61   * All coordinates passed to a <code>Graphics2D</code> object are specified 
62   * in a device-independent coordinate system called User Space, which is
63   * used by applications.  The <code>Graphics2D</code> object 
64   * <!- PBP/PP 6187216 ->
65   * <em>behaves as if it</em> contains
66   *  <em>a transform</em>
67   * object as part of its <em>internal</em> rendering state
68   * that defines how to convert coordinates from user space to
69   * device-dependent coordinates in Device Space.
70   * 
71   * <p>
72   * Some of the operations performed by the rendering attribute objects
73   * occur in the device space, but all <code>Graphics2D</code> methods take 
74   * user space coordinates.
75   * <p>
76   * Every <code>Graphics2D</code> object is associated with a target that
77   * defines where rendering takes place. A
78   * {@link GraphicsConfiguration} object defines the characteristics
79   * of the rendering target, such as pixel format and resolution.
80   * The same rendering target is used throughout the life of a
81   * <code>Graphics2D</code> object. 
82   * <p>
83   * When creating a <code>Graphics2D</code> object,  the 
84   * <code>GraphicsConfiguration</code> 
85   * specifies the
86   *  
87   * <!- PBP/PP 6187216 ->
88   * transform for
89   * the target of the <code>Graphics2D</code> (a 
90   * {@link Component} or {@link Image}).  This 
91   *  
92   * <!- PBP/PP 6187216 ->
93   * transform maps the
94   * user space coordinate system to screen and printer device coordinates
95   * such that the origin maps to the upper left hand corner of the
96   * target region of the device with increasing X coordinates extending
97   * to the right and increasing Y coordinates extending downward.
98   * The scaling of the 
99   *  
100  * <!- PBP/PP 6187216 ->
101  * transform is set to identity for those devices 
102  * that are close to 72 dpi, such as screen devices.
103  * The scaling of the 
104  *  
105  * <!- PBP/PP 6187216 ->
106  * transform is set to approximately 72 user 
107  * space coordinates per square inch for high resolution devices, such as
108  * printers.  For image buffers, the 
109  *  
110  * <!- PBP/PP 6187216 ->
111  * transform is the
112  *  
113  * <!- PBP/PP 6187216 ->
114  * <em>identity</em> 
115  * transform.
116  *
117  * <h2>Rendering Process</h2>
118  * The Rendering Process can be broken down into four phases that are 
119  * controlled by the <code>Graphics2D</code> rendering attributes.
120  * The renderer can optimize many of these steps, either by caching the
121  * results for future calls, by collapsing multiple virtual steps into
122  * a single operation, or by recognizing various attributes as common
123  * simple cases that can be eliminated by modifying other parts of the
124  * operation.
125  * <p>
126  * The steps in the rendering process are:
127  * <ol>
128  * <li>
129  * Determine what to render.  
130  * <li>
131  * Constrain the rendering operation to the current <code>Clip</code>.
132  * The <code>Clip</code> is specified by a {@link Shape} in user
133  * space and is controlled by the program using the various clip
134  * manipulation methods of <code>Graphics</code>
135  * .  
136  * <!- PBP/PP 6187216 ->
137  * This <i>user clip</i>
138  * is transformed into device space by the 
139  * 
140  * <em>target transform</em> 
141  * <!- PBP/PP 6187216 ->
142  * and combined with the
143  * <i>device clip</i>, which is defined by the visibility of windows and
144  * device extents.  The combination of the user clip and device clip
145  * defines the <i>composite clip</i>, which determines the final clipping
146  * region.  The user clip is not modified by the rendering
147  * system to reflect the resulting composite clip. 
148  * <li>
149  * Determine what colors to render.
150  * <li>
151  * Apply the colors to the destination drawing surface using the current
152  * {@link Composite} attribute in the <code>Graphics2D</code> context.
153  * </ol>
154  * <br>
155  * The three types of rendering operations, along with details of each 
156  * of their particular rendering processes are:
157  * <ol>
158  * <li>
159  * <b><a name="rendershape"><code>Shape</code> operations</a></b>
160  * <ol>
161  * <li>
162  * If the operation is a <code>draw</code> operation, then
163  * 
164  * the current {@link Stroke} attribute in the
165  * <code>Graphics2D</code> context is used to construct a new
166  * <code>Shape</code> object that contains the outline of the specified
167  * <code>Shape</code>.
168  * <li>
169  * The <code>Shape</code> is transformed from user space to device space
170  * using the 
171  * 
172  * <em> target transform </em>  
173  * in the <code>Graphics2D</code> context.
174  * 
175  * <li>
176  * The current 
177  * 
178  * <em> {@link Color} </em>
179  * in the <code>Graphics2D</code> context
180  * is 
181  * 
182  * <em> used to determine </em>
183  * the colors to render in device space.
184  * </ol>
185  * <li>
186  * <b><a name=rendertext>Text operations</a></b>
187  * <ol>
188  * <li>
189  * The following steps are used to determine the set of glyphs required
190  * to render the indicated <code>String</code>:
191  * <ol>
192  * <li>
193  * If the argument is a <code>String</code>, then the current
194  * <code>Font</code> in the <code>Graphics2D</code> context is asked to 
195  * convert the Unicode characters in the <code>String</code> into a set of 
196  * glyphs for presentation with whatever basic layout and shaping
197  * algorithms the font implements.
198  * <li>
199  * If the argument is an
200  * {@link AttributedCharacterIterator},
201  * 
202  * its embedded font attributes
203  * 
204  * <em> are used to implement </em> 
205  * more sophisticated glyph layout algorithms
206  * 
207  * .
208  * 
209  * </ol>
210  * <li>
211  * The current <code>Font</code> is queried to obtain outlines for the 
212  * indicated glyphs.  These outlines are treated as shapes in user space 
213  * relative to the position of each glyph that was determined in step 1.
214  * <li>
215  * The character outlines are filled as indicated above
216  * under <a href="#rendershape"><code>Shape</code> operations</a>.
217  * <li>
218   * The current 
219  * 
220  * <em> {@link Color} </em>
221  * is 
222  * 
223  * <em> used to determine </em>
224  * the colors to render in device space.
225  * </ol>
226  * <li>
227  * <b><a name= renderingimage><code>Image</code> Operations</a></b>
228  * <ol>
229  * <li>
230  * The region of interest is defined by the bounding box of the source 
231  * <code>Image</code>.
232  * This bounding box is specified in Image Space, which is the 
233  * <code>Image</code> object's local coordinate system. 
234  * <li>
235  * 
236  * <em>In this Profile,</em>
237  * the bounding box is <em>always</em> treated as if it is already in user space.
238  * <li>
239  * The bounding box of the source <code>Image</code> is transformed from user
240  * space into device space using the 
241  * <em> target transform </em>. 
242  * 
243  * Note that the result of transforming the bounding box does not
244  * necessarily result in a rectangular region in device space.
245  * <li>
246  * The <code>Image</code> object determines what colors to render, 
247  * sampled according to the source to destination
248  * coordinate mapping specified by the
249  * <em> target transform </em>. 
250  * 
251  * </ol>
252  * </ol>
253  * 
254  * <h2>Default Rendering Attributes</h2>
255  * The default values for the <code>Graphics2D</code> rendering attributes are:
256  * <dl compact>
257  *
258  * <dt><i><code><em>Color</em></code></i>
259  * <!- PBP/PP 6187216 ->
260  * <dd>The color of the <code>Component</code>.
261  *
262  * <dt><i><code>Font</code></i>
263  * <dd>The <code>Font</code> of the <code>Component</code>.
264  *
265  * <dt><i><code>Stroke</code></i>
266  * <dd>A square pen with a linewidth of 1, no dashing, miter segment joins
267  * and square end caps
268  * <em>(unless restricted - see restrictions in {@link BasicStroke})</em>
269  * <!- PBP/PP 6187216 ->
270  * .
271  *
272  * 
273  *
274  * <dt><i><code>Composite</code></i>
275  * <dd>The {@link AlphaComposite#SRC_OVER} rule.
276  *
277  * <dt><i><code>Clip</code></i>
278  * <dd>No rendering <code>Clip</code>, the output is clipped to the
279  * <code>Component</code>.
280  *
281  * </dl>
282  *
283  * 
284  *
285  * <p>
286  * <a name="restrictions">
287  * <h4>Restrictions</h4>
288  * <em>
289  * <ul>
290  * <li> Only instances of <code>AlphaComposite</code> are permitted when
291  * setting the <code>Composite</code> of the graphics context.
292  * See:
293  * <ul>
294  * <li> {@link #setComposite}
295  * </ul>
296  * <li>Only instances of <code>BasicStroke</code> are permitted when
297  * setting the <code>Stroke</code> of the graphics context. See:
298  * <ul>
299  * <li> {@link #setStroke}
300  * <li> {@link BasicStroke}
301  * </ul>
302  * <li> A Graphics2D implementation may ignore the <code>BasicStroke</code>
303  * attributes <em>end cap</em>, <em>line join</em>, and <em>miter limit</em>.
304  * See:
305  * <ul>
306  * <li> {@link #setStroke}
307  * <li> {@link BasicStroke}
308  * <li> <a href="../../doc-files/properties.html">Profile-specific properties</a>.
309  * </ul>
310  * </ul>
311  * 
312  * </em>
313  * @version 	1.78, 01/23/03
314  * @author Jim Graham
315 
316  */
