44  /** 
45   * The <code>Graphics</code> class is the abstract base class for 
46   * all graphics contexts that allow an application to draw onto 
47   * components that are realized on various devices, as well as 
48   * onto off-screen images.
49   * <p>
50   * A <code>Graphics</code> object encapsulates state information needed
51   * for the basic rendering operations that Java supports.  This
52   * state information includes the following properties:
53   * <p>
54   * <ul>
55   * <li>The <code>Component</code> object on which to draw.
56   * <li>A translation origin for rendering and clipping coordinates.
57   * <li>The current clip.  
58   * <li>The current color.
59   * <li>The current font.
60   * <li>The current logical pixel operation function (XOR or Paint).
61   * <li>The current XOR alternation color 
62   *     (see {@link Graphics#setXORMode}).
63   * </ul>
64   * <p>
65   * Coordinates are infinitely thin and lie between the pixels of the
66   * output device.
67   * Operations that draw the outline of a figure operate by traversing
68   * an infinitely thin path between pixels with a pixel-sized pen that hangs
69   * down and to the right of the anchor point on the path.
70   * Operations that fill a figure operate by filling the interior
71   * of that infinitely thin path.
72   * Operations that render horizontal text render the ascending
73   * portion of character glyphs entirely above the baseline coordinate.
74   * <p>
75   * The graphics pen hangs down and to the right from the path it traverses. 
76   * This has the following implications:
77   * <p><ul>
78   * <li>If you draw a figure that covers a given rectangle, that 
79   * figure occupies one extra row of pixels on the right and bottom edges 
80   * as compared to filling a figure that is bounded by that same rectangle.
81   * <li>If you draw a horizontal line along the same <i>y</i> coordinate as
82   * the baseline of a line of text, that line is drawn entirely below
83   * the text, except for any descenders.
84   * </ul><p>
85   * All coordinates that appear as arguments to the methods of this
86   * <code>Graphics</code> object are considered relative to the 
87   * translation origin of this <code>Graphics</code> object prior to 
88   * the invocation of the method.
89   * <p>
90   * All rendering operations modify only pixels which lie within the
91   * area bounded by the current clip, which is specified by a {@link Shape} 
92   * in user space and is controlled by the program using the 
93   * <code>Graphics</code> object.  This <i>user clip</i> 
94   * is transformed into device space and combined with the 
95   * <i>device clip</i>, which is defined by the visibility of windows and
96   * device extents.  The combination of the user clip and device clip 
97   * defines the <i>composite clip</i>, which determines the final clipping
98   * region.  The user clip cannot be modified by the rendering 
99   * system to reflect the resulting composite clip. The user clip can only
100  * be changed through the <code>setClip</code> or <code>clipRect</code> 
101  * methods.
102  * All drawing or writing is done in the current color, 
103  * using the current paint mode, and in the current font. 
104  * 
105  * @version 	1.65, 01/23/03
106  * @author 	Sami Shaio
107  * @author 	Arthur van Hoff
108  * @see     java.awt.Component
109  * @see     java.awt.Graphics#clipRect(int, int, int, int)
110  * @see     java.awt.Graphics#setColor(java.awt.Color)
111  * @see     java.awt.Graphics#setPaintMode()
112  * @see     java.awt.Graphics#setXORMode(java.awt.Color)
113  * @see     java.awt.Graphics#setFont(java.awt.Font)
114  * @since       JDK1.0
115  */

141     /** 
142      * Creates a new <code>Graphics</code> object based on this 
143      * <code>Graphics</code> object, but with a new translation and clip area.
144      * The new <code>Graphics</code> object has its origin 
145      * translated to the specified point (<i>x</i>,&nbsp;<i>y</i>). 
146      * Its clip area is determined by the intersection of the original
147      * clip area with the specified rectangle.  The arguments are all
148      * interpreted in the coordinate system of the original 
149      * <code>Graphics</code> object. The new graphics context is 
150      * identical to the original, except in two respects: 
151      * <p>
152      * <ul>
153      * <li>
154      * The new graphics context is translated by (<i>x</i>,&nbsp;<i>y</i>).  
155      * That is to say, the point (<code>0</code>,&nbsp;<code>0</code>) in the 
156      * new graphics context is the same as (<i>x</i>,&nbsp;<i>y</i>) in 
157      * the original graphics context. 
158      * <li>
159      * The new graphics context has an additional clipping rectangle, in 
160      * addition to whatever (translated) clipping rectangle it inherited 
161      * from the original graphics context. The origin of the new clipping 
162      * rectangle is at (<code>0</code>,&nbsp;<code>0</code>), and its size  
163      * is specified by the <code>width</code> and <code>height</code>
164      * arguments.
165      * </ul>

170      * @param      height   the height of the clipping rectangle.
171      * @return     a new graphics context.
172      * @see        java.awt.Graphics#translate
