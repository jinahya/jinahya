45  /** 
46   * An event which indicates that a keystroke occurred in a component.
47   * <p>
48   * This low-level event is generated by a component object (such as a text 
49   * field) when a key is pressed, released, or typed.  
50   * The event is passed to every <code>KeyListener</code>
51   * or <code>KeyAdapter</code> object which registered to receive such
52   * events using the component's <code>addKeyListener</code> method.
53   * (<code>KeyAdapter</code> objects implement the 
54   * <code>KeyListener</code> interface.)  Each such listener object 
55   * gets this <code>KeyEvent</code> when the event occurs.
56   *
57   * <p>
58   * <em>"Key typed" events</em> are higher-level and generally do not depend on 
59   * the platform or keyboard layout.  They are generated when a Unicode character 
60   * is entered, and are the preferred way to find out about character input.
61   * In the simplest case, a key typed event is produced by a single key press 
62   * (e.g., 'a').  Often, however, characters are produced by series of key 
63   * presses (e.g., 'shift' + 'a'), and the mapping from key pressed events to 
64   * key typed events may be many-to-one or many-to-many.  Key releases are not 
65   * usually necessary to generate a key typed event, but there are some cases 
66   * where the key typed event is not generated until a key is released (e.g., 
67   * entering ASCII sequences via the Alt-Numpad method in Windows).  
68   * No key typed events are generated for keys that don't generate Unicode 
69   * characters (e.g., action keys, modifier keys, etc.).
70   * The getKeyChar method always returns a valid Unicode character or 
71   * CHAR_UNDEFINED.  
72   * For key pressed and key released events, the getKeyCode method returns 
73   * the event's keyCode.  For key typed events, the getKeyCode method 
74   * always returns VK_UNDEFINED.
75   *
76   * <p>
77   * <em>"Key pressed" and "key released" events</em> are lower-level and depend 
78   * on the platform and keyboard layout. They are generated whenever a key is 
79   * pressed or released, and are the only way to find out about keys that don't 
80   * generate character input (e.g., action keys, modifier keys, etc.). The key 
81   * being pressed or released is indicated by the getKeyCode method, which returns 
82   * a virtual key code.
83   *
84   * <p>
85   * <em>Virtual key codes</em> are used to report which keyboard key has
86   * been pressed, rather than a character generated by the combination
87   * of one or more keystrokes (such as "A", which comes from shift and "a").  
88   *
89   * <p>
90   * For example, pressing the Shift key will cause a KEY_PRESSED event 
91   * with a VK_SHIFT keyCode, while pressing the 'a' key will result in 
92   * a VK_A keyCode.  After the 'a' key is released, a KEY_RELEASED event 
93   * will be fired with VK_A. Separately, a KEY_TYPED event with a keyChar 
94   * value of 'A' is generated.
95   *
96   * <p>
97   * Notes:
98   * <ul>
99   * <li>Key combinations which do not result in Unicode characters, such as action 
100  * keys like F1 and the HELP key, do not generate KEY_TYPED events.
101  * <li>Not all keyboards or systems are capable of generating all
102  * virtual key codes.  No attempt is made in Java to generate these keys 
103  * artificially. 
104  * <li>Virtual key codes do not identify a physical key: they depend on the
105  * platform and keyboard layout. For example, the key that generates VK_Q 
106  * when using a U.S. keyboard layout will generate VK_A when using a French 
107  * keyboard layout.
108  * <li>Not all characters have a keycode associated with them.  For example, 
109  * there is no keycode for the question mark because there is no keyboard 
110  * for which it appears on the primary layer.  
111  * <li>In order to support the platform-independent handling of action keys,
112  * the Java platform uses a few additional virtual key constants for functions
113  * that would otherwise have to be recognized by interpreting virtual key codes
114  * and modifiers. For example, for Japanese Windows keyboards, VK_ALL_CANDIDATES
115  * is returned instead of VK_CONVERT with the ALT modifier.
116  * </ul>
117  *
118  * <p>
119  * WARNING: Aside from those keys that are defined by the Java language
120  * (VK_ENTER, VK_BACK_SPACE, and VK_TAB), do not rely on the values of the VK_
121  * constants.  Sun reserves the right to change these values as needed
122  * to accomodate a wider range of keyboards in the future.  
123  *
124 <!-- PBP/PP -->
125  * <p>
126  * <a name="restrictions">
127  * <h4>Restrictions</h4>
128  * <em>
129  * Implementations of this Profile exhibit certain restrictions with respect
130  * to key events, specifically:
131  * <ul>
132  * <li> An implementation need not provide full support for the generation of
133  * key events.  In such a case, it may partially support the generation of
134  * key events or not support them at all.
135  * <p>
136  * Regardless of this restriction, applications may still
137  * create KeyEvent objects and post them to the system event queue.
138  * </ul>
139  * Note: An implementation that provides full support for key events must
140  * support a <a href="doc-files/basic-keys.html">basic set of virtual key
141  * codes</a>.
142  * <p>
143  * For more information, see <a href="../../../doc-files/properties.html">Profile-specific properties</a>.
144  * </em>
