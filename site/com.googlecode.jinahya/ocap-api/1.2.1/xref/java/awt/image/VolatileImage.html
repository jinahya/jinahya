45  /**
46   * VolatileImage is an image which can lose its 
47   * contents at any time due to circumstances beyond the control of the 
48   * application (e.g., situations caused by the operating system or by
49   * other applications). Because of the potential for hardware acceleration,
50   * a VolatileImage object can have significant performance benefits on 
51   * some platforms.
52   * <p>
53   * The drawing surface of an image (the memory where the image contents 
54   * actually reside) can be lost or invalidated, causing the contents of that 
55   * memory to go away.  The drawing surface thus needs to be restored
56   * or recreated and the contents of that surface need to be
57   * re-rendered.  VolatileImage provides an interface for 
58   * allowing the user to detect these problems and fix them
59   * when they occur.
60   * <p>
61   * This image should not be subclassed directly but should be created
62   * by using the {@link java.awt.Component#createVolatileImage(int, int)
63   * Component.createVolatileImage} or 
64   * {@link java.awt.GraphicsConfiguration#createCompatibleVolatileImage(int, int)
65   * GraphicsConfiguration.createCompatibleVolatileImage(int, int)} methods.
66   * <P>
67   * An example of using a VolatileImage object follows:
68   * <pre>
69   * // image creation
70   * VolatileImage vImg = createVolatileImage(w, h);
71   *
72   * 
73   * // rendering to the image
74   * void renderOffscreen() {
75   *	do {
76   *	    if (vImg.validate(getGraphicsConfiguration()) ==
77   *		VolatileImage.IMAGE_INCOMPATIBLE)
78   *	    {
79   *		// old vImg doesn't work with new GraphicsConfig; re-create it
80   *		vImg = createVolatileImage(w, h);
81   *	    }
82   *	    Graphics2D g = vImg.createGraphics();
83   *	    //
84   *	    // miscellaneous rendering commands...
85   *	    //
86   *	    g.dispose();
87   *	} while (vImg.contentsLost());
88   * }
89   *
90   *
91   * // copying from the image (here, gScreen is the Graphics
92   * // object for the onscreen window)
93   * do {
94   *	int returnCode = vImg.validate(getGraphicsConfiguration());
95   *	if (returnCode == VolatileImage.IMAGE_RESTORED) {
96   *	    // Contents need to be restored
97   *	    renderOffscreen();	    // restore contents
98   *	} else if (returnCode == VolatileImage.IMAGE_INCOMPATIBLE) {
99   *	    // old vImg doesn't work with new GraphicsConfig; re-create it
100  *	    vImg = createVolatileImage(w, h);
101  *	    renderOffscreen();
102  *	}
103  *	gScreen.drawImage(vImg, 0, 0, this);
104  * } while (vImg.contentsLost());
105  * </pre>
106  * <P>
107  * Note that this class subclasses from the {@link Image} class, which
108  * includes methods that take an {@link ImageObserver} parameter for
109  * asynchronous notifications as information is received from
110  * a potential {@link ImageProducer}.  Since this <code>VolatileImage</code> 
111  * is not loaded from an asynchronous source, the various methods that take
112  * an <code>ImageObserver</code> parameter will behave as if the data has
113  * already been obtained from the <code>ImageProducer</code>.  
114  * Specifically, this means that the return values from such methods 
115  * will never indicate that the information is not yet available and 
116  * the <code>ImageObserver</code> used in such methods will never 
117  * need to be recorded for an asynchronous callback notification.
118  */

263     /**
264      * Attempts to restore the drawing surface of the image if the surface
265      * had been lost since the last <code>validate</code> call.  Also
266      * validates this image against the given GraphicsConfiguration
267      * parameter to see whether operations from this image to the
268      * GraphicsConfiguration are compatible.  An example of an
269      * incompatible combination might be a situation where a VolatileImage
270      * object was created on one graphics device and then was used
271      * to render to a different graphics device.  Since VolatileImage
272      * objects tend to be very device-specific, this operation might
273      * not work as intended, so the return code from this validate
274      * call would note that incompatibility.  A null or incorrect 
275      * value for gc may cause incorrect values to be returned from
276      * <code>validate</code> and may cause later problems with rendering.
277      *
278      * @param   gc   a <code>GraphicsConfiguration</code> object for this
279      *		image to be validated against.  A null gc implies that the
280      *		validate method should skip the compatibility test.
281      * @return	<code>IMAGE_OK</code> if the image did not need validation<BR>
282      *		<code>IMAGE_RESTORED</code> if the image needed restoration.  
283      *		Restoration implies that the contents of the image may have 
284      *		been affected and the image may need to be re-rendered.<BR>
285      *		<code>IMAGE_INCOMPATIBLE</code> if the image is incompatible
286      *		with the <code>GraphicsConfiguration</code> object passed
287      *		into the <code>validate</code> method.  Incompatibility
288      *		implies that the image may need to be recreated with a
289      *		new <code>Component</code> or 
290      *		<code>GraphicsConfiguration</code> in order to get an image
291      *		that can be used successfully with this 
292      *		<code>GraphicsConfiguration</code>.
293      *		An incompatible image is not checked for whether restoration
294      *		was necessary, so the state of the image is unchanged 
295      *		after a return value of <code>IMAGE_INCOMPATIBLE</code>
296      *		and this return value implies nothing about whether the
297      *		image needs to be restored.
298      * @see java.awt.GraphicsConfiguration
299      * @see java.awt.Component
300      * @see #IMAGE_OK
301      * @see #IMAGE_RESTORED
302      * @see #IMAGE_INCOMPATIBLE
303      */
304     public abstract int validate(GraphicsConfiguration gc);
