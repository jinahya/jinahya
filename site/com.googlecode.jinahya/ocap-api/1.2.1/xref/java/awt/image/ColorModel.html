49  /** 
50   * The <code>ColorModel</code> abstract class encapsulates the
51   * methods for translating a pixel value to color components 
52   * (for example, red, green, and blue) and an alpha component.
53   * In order to render an image to the screen, a printer, or another
54   * image, pixel values must be converted to color and alpha components.
55   * As arguments to or return values from methods of this class,
56   * pixels are represented as 32-bit ints or as arrays of primitive types.
57   * The number, order, and interpretation of color components for a 
58   * <code>ColorModel</code> is specified by its <code>ColorSpace</code>.  
59   * A <code>ColorModel</code> used with pixel data that does not include
60   * alpha information treats all pixels as opaque, which is an alpha
61   * value of 1.0.
62   * <p>
63  
64   * <p>
65   * Methods in the <code>ColorModel</code> class use two different
66   * representations of color and alpha components - a normalized form
67   * and an unnormalized form.  In the normalized form, each component is a 
68   * <code>float</code> value between
69  <!-- PBP/PP [6217618] -->
70   * <em>0.0 and 1.0.</em>
71  
72   * Normalized color component values are not premultiplied.
73   * All <code>ColorModels</code> must support the normalized form.
74   * <p>
75   * In the unnormalized
76   * form, each component is an unsigned integral value between 0 and
77   * 2<sup>n</sup> - 1, where n is the number of significant bits for a
78   * particular component.  If pixel values for a particular 
79   * <code>ColorModel</code> represent color samples premultiplied by
80   * the alpha sample, unnormalized color component values are
81   * also premultiplied.  The unnormalized form is used only with instances
82   * of <code>ColorModel</code> whose <code>ColorSpace</code> has minimum
83   * component values of 0.0 for all components and maximum values of
84   * 1.0 for all components.
85   * The unnormalized form for color and alpha components can be a convenient
86   * representation for <code>ColorModels</code> whose normalized component
87   * values all lie
88   * between 0.0 and 1.0.  In such cases the integral value 0 maps to 0.0 and
89   * the value 2<sup>n</sup> - 1 maps to 1.0.  In other cases, such as
90   * when the normalized component values can be either negative or positive,
91   * the unnormalized form is not convenient.  Such <code>ColorModel</code>
92   * objects throw an {@link IllegalArgumentException} when methods involving
93   * an unnormalized argument are called.  Subclasses of <code>ColorModel</code>
94   * must specify the conditions under which this occurs.
95   *
96   * @see IndexColorModel
97  
98   * @see DirectColorModel
99   * @see java.awt.Image
100  * @see BufferedImage
101 
102  * @see java.awt.color.ColorSpace
103 
104  * @see DataBuffer
105  * @version 10 Feb 1997
106  */

142      * Constructs a <code>ColorModel</code> that translates pixel values
143      * to color/alpha components.  Color components will be in the
144      * specified <code>ColorSpace</code>. <code>pixel_bits</code> is the
145      * number of bits in the pixel values.  The bits array
146      * specifies the number of significant bits per color and alpha component.
147      * Its length should be the number of components in the 
148      * <code>ColorSpace</code> if there is no alpha information in the
149      * pixel values, or one more than this number if there is alpha
150      * information.  <code>hasAlpha</code> indicates whether or not alpha
151      * information is present.  The <code>boolean</code> 
152      * <code>isAlphaPremultiplied</code> specifies how to interpret pixel
153      * values in which color and alpha information are represented as
154      * separate spatial bands.  If the <code>boolean</code>
155      * is <code>true</code>, color samples are assumed to have been
156      * multiplied by the alpha sample.  The <code>transparency</code> 
157      * specifies what alpha values can be represented by this color model.
158      * The transfer type is the type of primitive array used to represent
159      * pixel values.  Note that the bits array contains the number of
160      * significant bits per color/alpha component after the translation
161      * from pixel values.  For example, for an
162      * <code>IndexColorModel</code> with <code>pixel_bits</code> equal to
163      * 16, the bits array might have four elements with each element set
164      * to 8.
165      * @param pixel_bits the number of bits in the pixel values
166      * @param bits array that specifies the number of significant bits
167      *		per color and alpha component
168      * @param cspace the specified <code>ColorSpace</code>
169      * @param hasAlpha <code>true</code> if alpha information is present;
170      *		<code>false</code> otherwise
171      * @param isAlphaPremultiplied <code>true</code> if color samples are
172      *		assumed to be premultiplied by the alpha samples;
173      *		<code>false</code> otherwise
174      * @param transparency what alpha values can be represented by this
175      *		color model
176      * @param transferType the type of the array used to represent pixel
177      *		values
178      * @throws IllegalArgumentException if the length of
179      *		the bit array is less than the number of color or alpha
180      *		components in this <code>ColorModel</code>, or if the
181      *          transparency is not a valid value.
182      * @throws IllegalArgumentException if the sum of the number
183      * 		of bits in <code>bits</code> is less than 1 or if
184      *          any of the elements in <code>bits</code> is less than 0.
185      * @see java.awt.Transparency
186      */
