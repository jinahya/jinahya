56  /**
57   * The <code>TextAttribute</code> class defines attribute keys and
58   * attribute values used for text rendering. 
59   * <p>
60   * <code>TextAttribute</code> instances are used as attribute keys to
61   * identify attributes in 
62   * {@link java.text.AttributedCharacterIterator AttributedCharacterIterator}, 
63   * {@link java.awt.Font Font}, and other classes handling text 
64   * attributes. Other constants defined in this class are used 
65   * as attribute values.
66   * <p>
67   * For each text attribute, the documentation describes:
68   * <UL>
69   *   <LI>the type of their values,
70   *   <LI>the valid values if there are limitations
71   *   <LI>relevant constants
72   *   <LI>the default effect if the attribute is absent (or has a
73   *	<code>null</code> value).
74   *   <LI>a description of the effect.
75   *   <LI>the fallback behavior if the exact attribute requested is not 
76   *	available.
77   * </UL>
78   * <p>
79   * <H4>Types of Values</H4>
80   * <UL>
81   *   <LI>The values of attributes must always be immutable.
82   *   <LI>Where a list of limitations is given, any value outside of that
83   *   set is reserved for future use, and ignored at present.
84   *   <LI>If the value is <code>null</code> or not of the proper type
85   *	then it has the default effect. The effect of a particular value
86   *	can be interpolated, especially in the case of multiple master
87   *	fonts. This interpolation is done based on the nearest defined
88   * 	constants above and below the request:<BR>
89   *      <BLOCKQUOTE><TT>
90   *	interpolation = (request - below)/(above - below);
91   *	</TT></BLOCKQUOTE>
92   * </UL>
93   * <p>
94   * <H4>Interpolation</H4>
95   * <UL>
96   *   <LI>Fonts should interpolate values in certain circumstances. For example,
97   *   when the WEIGHT value is 2.13. If the nearest surrounding values
98   *   in the font are WEIGHT_BOLD = 2.0 and WEIGHT_HEAVY = 2.25 then font would
99   *   then interpret the WEIGHT request as being 52% of the way between what
100  *   it considers BOLD and what it considers HEAVY. If the nearest surrounding
101  *   values are WEIGHT_SEMIBOLD = 1.25 and WEIGHT_ULTRABOLD = 2.75 then the
102  *   WEIGHT request is interpreted as being 58.67% of the way between SEMIBOLD
103  *   and ULTRABOLD.
104  *   <LI>Where a font does not have enough capability to handle a given 
105  *   request, such as superscript, then it should simulate it to the best of 
106  *   its ability.  To determine if simulation is being performed, the client
107  *   should query the font to see what actual attributes were used.
108  * </UL>
109  * 
110  * @see java.text.AttributedCharacterIterator
111  * @see java.awt.Font 
112  */

448      *     SUMMARY="Key, Value, Default, and Description for 
449      *     TextAttribute TRANSFORM">

463      * general, large rotations do not produce very useful results. The
464      * transform modifies both the glyph and the advance. The translations
465      * in the transform are interpreted as a ratio of the point size. That
466      * is, with a point size of 12, a translation of 0.5 results in a
467      * movement of 6 points.
468      * <p>
469      * The advance point of the transformed glyph is the transform of the 
470      * advance point projected onto the baseline. If the advance ends up
471      * to the left (top) of the glyph origin, the two points are swapped.

477      * The original advance point is &lt;10.0, 0.0&gt;; after the rotation it 
478      * is &lt;6.0, -8.0&gt;; when adding the offset this becomes 
479      * &lt;8.0,-10.0&gt;, when projecting on the (horizontal) baseline this
480      * becomes the new advance point: &lt;8.0, 0.0&gt;. The advance width is
481      * the distance from the origin to the advance point: 8.0. The rotated
482      * glyph is rendered two points up and to the right of its origin and
483      * rotated.  This does not affect the baseline for subsequent
484      * glyphs.</P></TD></TR>
